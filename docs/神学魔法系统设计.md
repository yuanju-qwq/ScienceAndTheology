# 神学魔法系统设计

## 1. 概述

"神学魔法系统"是「科学与神学」项目中与「GT 工业系统」对等的第二套核心体系。定位为**弱依赖、强联动**：纯魔法玩家可独立通关，纯工业玩家也可独立通关；但当两者结合使用时，通过联动交汇点获得超额收益。

### 1.1 核心原则

```
        工业线                           魔法线
    ┌──────────┐                     ┌──────────┐
    │  GT 机器  │                     │  符文手搓  │
    │  AE2 合成 │                     │  祭坛仪式  │
    │  电力网络  │                     │  玩家法力  │
    └─────┬────┘                     └─────┬─────┘
          │                                │
          │     两者独立可通关               │
          │                                │
          └────────────┬───────────────────┘
                       │
                强联动交汇点：
                · 高级符文可用机器量产（更快更多）
                · 机器附魔需要仪式完成
                · 但都不阻塞任何一方的通关
```

- **弱依赖**：两套体系互不阻塞。任一体系缺失不影响另一方通关。
- **强联动**：同时使用两套体系时获得超额增益，但非必须。联动是"锦上添花"而非"刚需绑定"。
- **手动 vs 自动化**：符文可由工业机器自动化生产；仪式必须玩家在祭坛前手动执行（3-10 秒）。

### 1.2 灵感来源与差异

参考 Minecraft 模组「新生魔艺」(Ars Nouveau) 的符文组合定制化理念，但根据本项目 2D 俯视角、工厂自动化核心玩法的语境做出以下关键调整：

| 特性 | Ars Nouveau | 本方案 |
|------|------------|--------|
| 符文自定义法术 | 雕文组合 → 法术书发动 | 符文排列 → 祭坛固定仪式 |
| 施法方式 | 即时手搓 | 3-10 秒固定仪式动画 |
| 自动化 | 无 | 符文生产全自动 + 仪式纯手动 |
| 与工业关系 | 独立 | **弱依赖、强联动** |
| 进度驱动 | 经验/等级解锁 | 探索 + 工业双线并行 |

## 2. 法力系统

法力（Mana）是**纯玩家个人属性**，不设法力蓄电池方块，不接管道网络。

```cpp
// src/core/magic/mana_pool.hpp
struct ManaPool {
    int current_mana = 0;
    int max_mana = 100;       // 可通过装备/仪式永久提升上限
    float regen_rate = 0.1f;  // 每 tick 自然回复（2/秒）

    bool consume(int amount);  // 返回 false 表示法力不足
    void add(int amount);
    bool is_full() const;
    void tick();               // 每 tick 自然回复（上限截断）
};
```

### 2.1 法力来源

| 来源方式 | 描述 | 量级 |
|---------|------|------|
| 自然回复 | 常驻，约 2 mana/秒 | 慢但持续 |
| 小法力药水 | 饮用道具，瞬间回复 | +30 |
| 大法力药水 | 饮用道具，瞬间回复 | +80 |
| 法力水晶 | 一次性使用道具，完全回复 | 按上限 |
| 祭坛共鸣 | 站在祭坛核心旁，回复速度 ×5 | 加速自然回复 |
| 献祭仪式 | 消耗生命值/物品换取法力 | +50 ∼ +200 |

## 3. 符文系统

### 3.1 符文元素体系（8 种）

| 元素 | 标识 | 核心效果 | 工业联动方向 |
|------|------|---------|-------------|
| 火 (Fire) | 🔥 | 加速、破坏、能量 | 提升机器速度 |
| 水 (Water) | 💧 | 流动、恢复、冷却 | 降低机器损耗 |
| 土 (Earth) | 🪨 | 稳固、防御、增产 | 提升产出倍率 |
| 风 (Air) | 🌪 | 移动、范围、效率 | 降低能耗 |
| 光 (Light) | ✨ | 揭示、纯净、祝福 | 机器免维护 |
| 暗 (Dark) | 🌑 | 隐匿、衰败、诅咒 | 特殊诅咒效果 |
| 秩序 (Order) | ⚙ | 结构、精确、控制 | 提升配方成功率 |
| 混沌 (Chaos) | 🌀 | 随机、转化、力量 | 产出随机翻倍 |

### 3.2 符文等级（4 级）

| 等级 | 强度值 (potency) | 命名后缀 |
|------|------------------|---------|
| 普通 (Common) | 1 | `_common` |
| 精良 (Refined) | 2 | `_refined` |
| 卓越 (Superior) | 3 | `_superior` |
| 传奇 (Legendary) | 5 | `_legendary` |

共 8 元素 × 4 等级 = **32 种符文物品**。

### 3.3 数据结构

```cpp
// src/core/magic/rune_def.hpp
namespace science_and_theology::magic {

enum class RuneElement : uint8_t {
    FIRE = 0,
    WATER,
    EARTH,
    AIR,
    LIGHT,
    DARK,
    ORDER,
    CHAOS,
    COUNT  // == 8
};

enum class RuneTier : uint8_t {
    COMMON = 0,
    REFINED,
    SUPERIOR,
    LEGENDARY,
    COUNT  // == 4
};

struct RuneDef {
    RuneElement element;
    RuneTier tier;
    int potency;  // 1/2/3/5，对应等级
    ResourceKey item_key;
};

}  // namespace science_and_theology::magic
```

符文继承 `MaterialItem`，复用现有物品体系，可直接存入背包、储物箱和 AE2 ME 系统。

### 3.4 双轨生产路线

#### 手动路线（纯魔法玩家，零工业基础）

```
神秘水晶矿（地下层生成）→ [手采] → 粗水晶
粗水晶 + 空白符石 → [手工雕刻台] → 普通符文
普通符文 × N + 元素媒介 → [手工融合台] → 高级符文
```

- 手工台是放置类可交互方块，右键打开 GUI
- 产出 1:1，无加成

#### 工业路线（联动玩家，更高效率）

```
神秘水晶矿 → [采矿机] → 粗水晶 × 1.5
粗水晶 → [破碎机] → 水晶粉 × 1.5
水晶粉 → [离心机] → 元素精华 × 1.2
元素精华 + 符文基底 → [符文蚀刻机] → 普通符文（10× 速度）
普通符文 × N → [符文融合器] → 高级符文（连续处理，无等待）
```

- 产能加成：+50% 矿物产出（采矿机）、+20% 精华产出（离心机）
- 速度加成：机器处理速度是手搓的 10 倍
- 完全兼容现有 GT 电力、流体管道、AE2 自动合成体系
- 工业路线是**联动福利**，纯魔法玩家完全不受影响

## 4. 祭坛仪式系统

### 4.1 祭坛结构（2D 俯视视角）

```
基础祭坛（4 台座）：            高级祭坛（8 台座）：

    ┌───┐                      ┌───┐   ┌───┐
    │ 0 │                      │ 0 │   │ 1 │
    └─┬─┘                      └─┬─┘   └─┬─┘
      │                          │       │
┌─┬─┬─┼─┬─┬─┐                ┌─┬─┼─┬─┬─┬─┼─┬─┐
│3│ │3│ A │1│1│                │7│ │7│ A │1│1│
└─┼─┼─┼─┼─┼─┘                └─┼─┼─┼─┼─┼─┼─┘
      │                          │       │
    │ 2 │                      │ 6 │   │ 2 │
    └───┘                      ├───┤   ├───┤
                               │ 5 │   │ 3 │
A = 核心祭坛（玩家站此处）       └───┘   └───┘
0-3 = 符文台座（放置符文）         4
                               ├───┤
                               │ 4 │
                               └───┘
```

- **基础祭坛**：4 个台座。石材建造。
- **高级祭坛**：8 个台座。需要工业制造的"高级台座"升级。（联动交汇点）

### 4.2 仪式配方

```cpp
// src/core/magic/ritual_recipe.hpp
namespace science_and_theology::magic {

struct RitualPedestalSlot {
    RuneElement element;      // 要求元素
    RuneTier min_tier;        // 最低等级
    bool strict_element;      // 是否必须严格匹配元素（false=接受任意）
};

struct RitualEffect {
    enum class Type : uint8_t {
        NONE = 0,
        MACHINE_BLESSING,     // 机器祝福
        TOOL_ENCHANTMENT,     // 工具附魔
        TERRAIN_ALTERATION,   // 地形改造
        PLAYER_BUFF,          // 玩家增益
        WORLD_EVENT,          // 世界事件（开关遗迹门等）
        CURSE,                // 诅咒
        TELEPORTATION,        // 传送链接
        DIVINATION,           // 占卜
        MANA_EXPANSION,       // 法力扩容（永久提升上限）
        COUNT
    };
    
    Type type;
    nlohmann::json params;  // 效果参数，序列化为 JSON
    int duration_ticks;     // 持续时长（0 = 瞬时/永久）
};

struct RitualRecipe {
    ResourceKey id;
    std::string display_name;
    
    // 台座配置（4/8 个插槽）
    std::vector<RitualPedestalSlot> pedestals;
    
    int mana_cost;          // 法力消耗
    int duration_ticks;     // 仪式执行时长（60-200 tick = 3-10 秒 @ 20TPS）
    bool consume_runes;     // 仪式后符文是否消耗
    
    RitualEffect effect;
};

}  // namespace science_and_theology::magic
```

### 4.3 仪式执行流程（3-10 秒）

```
玩家走到核心祭坛 [A]
  └→ 右键台座 [0-3/0-7]，逐个放入符文
     └→ 台座显示符文图标，列表更新
        └→ 实时匹配：符文排列 → 查找配方
           ├→ 未匹配：无反应
           └→ 匹配成功：显示效果预览（悬浮文字）
               └→ 玩家按下确认键（E）
                  ├→ 检查法力：mana ≥ mana_cost？
                  │  ├→ 否：提示"法力不足"
                  │  └→ 是：扣除法力，开始仪式
                  ├→ 播放动画（3-10 秒）
                  │  └→ 粒子特效、光柱、符文旋转
                  ├→ 仪式完成
                  │  ├→ apply_effect() 效果生效
                  │  └→ 若 consume_runes = true，清空台座符文
                  └→ 结束
```

## 6. 强联动交汇点

以下交汇点提供**超额增益**，但采用时两者之一缺失也不影响通关。

| 交汇点 | 纯魔法路径 | 工业联动路径 | 联动收益 |
|--------|-----------|-------------|---------|
| 符文生产 | 手搓 1:1 产出 | 机器量产 1:1.5 产出 | +50% |
| 高级符文 | 手工融合台 | 符文融合器（10× 速度） | 10× 速度 |
| 祭坛升级 | 固定 4 台座 | 工业制造"高级台座" → 8 台座 | 更多配方 |
| 精华制取 | 不可得 | 离心机分离元素精华 | 解锁最优化符文路线 |
| 机器附魔 | 无影响 | 仪式给机器添加临时附魔 | 机器性能强化 |
| 法力扩容 | 仪式永久 +25 | 工业制造"法力容器" +50 | +50 |
| 祭坛建材 | 石材手工制作 | 工业批量生产 | 建造成本降低 |

## 7. 与现有系统的集成

### 7.1 MaterialRegistry

符文、法力药水、祭坛建材均通过 `MaterialRegistry` 注册为标准的 `MaterialItem`。物品定义（`ItemDef`）直接复用现有体系，存储和传输无需额外适配。

### 7.2 EventBus（事件桥接）

仪式效果通过现有的 `EventBus` 发布事件，避免魔法核心与工业核心直接耦合：

```cpp
// 仪式执行器发布事件（src/core/simulation/event_types.hpp 中新增）
MachineBlessEvent { machine_id, effect_type, multiplier, duration_ticks }
PlayerBuffEvent   { player_id, buff_type, multiplier, duration_ticks }
WorldEvent         { event_type, position, params }
```

工业系统（`MachineSystem`）订阅 `MachineBlessEvent`，自行处理附魔计算。

### 7.3 世界机制系统（MechanismManager）

仪式引发的世界事件（遗迹门开关、天气改变等）通过现有的 `MechanismManager` 标记系统管理。与 Sun Altar → Ruin Gate 的现有机制完全兼容。

### 7.4 玩家系统（PlayerController + Inventory）

- `ManaPool` 挂在 `GDPlayerEquipment` 旁边，随玩家数据序列化
- 法力药水/符文是标准的 `MaterialItem`，存入 `PlayerInventory`
- 法力条 UI 在 `PlayerController.gd` 的 HUD 组件中渲染

### 7.5 存档系统（ChunkSerializer）

- `ManaPool` 数据序列化到 `PlayerData` 中
- 祭坛台座上放置的符文作为世界状态序列化到对应 Chunk
- 永久仪式效果（传送链接、法力扩容）记录在 WorldData

## 10. 设计约束

1. **零 Godot 依赖**：`src/core/magic/` 下所有代码为纯 C++20，不引入 `godot-cpp` 头文件。
2. **最小化耦合**：魔法核心与工业核心只通过 `EventBus` 事件和 `MaterialRegistry` 间接交互，不互相 `#include`。
3. **存档前向兼容**：`ManaPool` 字段序列化新增，旧存档加载时设默认值（0 mana / 100 max）。
4. **可测试性**：核心逻辑可在独立 C++ 测试中验证（`tests/`），无需启动 Godot。
5. **扩展预留**：`RitualEffect::params` 使用 JSON 串行化，可后续追加参数字段而不改数据结构定义。
