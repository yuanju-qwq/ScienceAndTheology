# 神学魔法系统设计

## 1. 概述

"神学魔法系统"是「科学与神学」项目中与「GT 工业系统」对等的第二套核心体系。定位为**弱依赖、强联动**：纯魔法玩家可独立通关，纯工业玩家也可独立通关；但当两者结合使用时，通过联动交汇点获得超额收益。

### 1.1 核心原则

```
        工业线                           魔法线
    ┌──────────┐                     ┌──────────┐
    │  GT 机器  │                     │  符文手搓  │
    │  AE2 合成 │                     │  祭坛仪式  │
    │  电力网络  │                     │  玩家法力  │
    └─────┬────┘                     └─────┬─────┘
          │                                │
          │     两者独立可通关               │
          │                                │
          └────────────┬───────────────────┘
                       │
                强联动交汇点：
                · 高级符文可用机器量产（更快更多）
                · 机器附魔需要仪式完成
                · 但都不阻塞任何一方的通关
```

- **弱依赖**：两套体系互不阻塞。任一体系缺失不影响另一方通关。
- **强联动**：同时使用两套体系时获得超额增益，但非必须。联动是"锦上添花"而非"刚需绑定"。
- **双轨施法**：魔符魔法书用于随身即时施法（采矿/战斗/移动）；祭坛仪式用于定点长期效果（机器祝福/地形改造/传送）。
- **手动 vs 自动化**：符文/魔符可由工业机器自动化生产；施法与仪式必须玩家手动执行。

### 1.2 两套魔法子系统

魔法系统分为**祭坛仪式**（定点、战略级）和**魔符魔法书**（随身、战术级），共享底层符文资源：

```
               符文 (Rune) — 32 种，8 元素 × 4 等级
                         │
          ┌──────────────┴──────────────┐
          ▼                              ▼
  ┌───────────────┐             ┌───────────────┐
  │  祭坛仪式       │             │  魔符魔法书     │
  │  定点 · 被动    │             │  随身 · 主动    │
  │  符文→台座排列  │             │  符文→魔符→拼装 │
  │  3-10 秒仪式    │             │  即时/蓄力施法  │
  │  机器祝福/地形  │             │  采矿/战斗/照明  │
  └───────────────┘             └───────────────┘
```

### 1.3 灵感来源与差异

参考 Minecraft 模组「新生魔艺」(Ars Nouveau) 的符文组合 + 雕文施法理念，根据本项目 3D 方块世界、工厂自动化核心玩法进行调整：

| 特性 | Ars Nouveau | 本方案 |
|------|------------|--------|
| 雕文/符文自定义 | 雕文组合 → 法术书发动 | 符文→魔符→魔法书施法 + 祭坛仪式双轨 |
| 施法方式 | 即时手搓 | 魔法书即时施法 + 祭坛 3-10 秒仪式 |
| 自动化 | 无 | 符文/魔符生产全自动；施法/仪式纯手动 |
| 与工业关系 | 独立 | **弱依赖、强联动** |
| 进度驱动 | 经验/等级解锁 | 探索 + 工业双线并行 |
| 法术切换 | 手动重排雕文 | 4 套预设，数字键 1-4 切换 |

## 2. 法力系统

法力（Mana）是**纯玩家个人属性**，不设法力蓄电池方块，不接管道网络。

```cpp
// src/core/magic/mana_pool.hpp
struct ManaPool {
    int current_mana = 0;
    int max_mana = 100;       // 可通过装备/仪式永久提升上限
    float regen_rate = 0.1f;  // 每 tick 自然回复（2/秒）

    bool consume(int amount);  // 返回 false 表示法力不足
    void add(int amount);
    bool is_full() const;
    void tick();               // 每 tick 自然回复（上限截断）
};
```

### 2.1 法力来源

| 来源方式 | 描述 | 量级 |
|---------|------|------|
| 自然回复 | 常驻，约 2 mana/秒 | 慢但持续 |
| 小法力药水 | 饮用道具，瞬间回复 | +30 |
| 大法力药水 | 饮用道具，瞬间回复 | +80 |
| 法力水晶 | 一次性使用道具，完全回复 | 按上限 |
| 祭坛共鸣 | 站在祭坛核心旁，回复速度 ×5 | 加速自然回复 |
| 献祭仪式 | 消耗生命值/物品换取法力 | +50 ∼ +200 |

### 2.2 法力上限扩容

|------|--------|------|
| 法力容器（工业制造） | +50 | 联动交汇点 |

## 3. 符文系统

### 3.1 符文元素体系（8 种）

| 元素 | 标识 | 核心效果 | 工业联动方向 |
|------|------|---------|-------------|
| 火 (Fire) | 🔥 | 加速、破坏、能量 | 提升机器速度 |
| 水 (Water) | 💧 | 流动、恢复、冷却 | 降低机器损耗 |
| 土 (Earth) | 🪨 | 稳固、防御、增产 | 提升产出倍率 |
| 风 (Air) | 🌪 | 移动、范围、效率 | 降低能耗 |
| 光 (Light) | ✨ | 揭示、纯净、祝福 | 机器免维护 |
| 暗 (Dark) | 🌑 | 隐匿、衰败、诅咒 | 特殊诅咒效果 |
| 秩序 (Order) | ⚙ | 结构、精确、控制 | 提升配方成功率 |
| 混沌 (Chaos) | 🌀 | 随机、转化、力量 | 产出随机翻倍 |

### 3.2 符文等级（4 级）

| 等级 | 强度值 (potency) | 命名后缀 |
|------|------------------|---------|
| 普通 (Common) | 1 | `_common` |
| 精良 (Refined) | 2 | `_refined` |
| 卓越 (Superior) | 3 | `_superior` |
| 传奇 (Legendary) | 5 | `_legendary` |

共 8 元素 × 4 等级 = **32 种符文物品**。

### 3.3 数据结构

```cpp
// src/core/magic/rune_def.hpp
namespace science_and_theology::magic {

enum class RuneElement : uint8_t {
    FIRE = 0,
    WATER,
    EARTH,
    AIR,
    LIGHT,
    DARK,
    ORDER,
    CHAOS,
    COUNT  // == 8
};

enum class RuneTier : uint8_t {
    COMMON = 0,
    REFINED,
    SUPERIOR,
    LEGENDARY,
    COUNT  // == 4
};

struct RuneDef {
    RuneElement element;
    RuneTier tier;
    int potency;  // 1/2/3/5，对应等级
    ResourceKey item_key;
};

}  // namespace science_and_theology::magic
```

符文继承 `MaterialItem`，复用现有物品体系，可直接存入背包、储物箱和 AE2 ME 系统。

### 3.4 双轨生产路线

#### 手动路线（纯魔法玩家，零工业基础）

```
神秘水晶矿（地下层生成）→ [手采] → 粗水晶
粗水晶 + 空白符石 → [手工雕刻台] → 普通符文
普通符文 × N + 元素媒介 → [手工融合台] → 高级符文
```

- 手工台是放置类可交互方块，右键打开 GUI
- 产出 1:1，无加成

#### 工业路线（联动玩家，更高效率）

```
神秘水晶矿 → [采矿机] → 粗水晶 × 1.5
粗水晶 → [破碎机] → 水晶粉 × 1.5
水晶粉 → [离心机] → 元素精华 × 1.2
元素精华 + 符文基底 → [符文蚀刻机] → 普通符文（10× 速度）
普通符文 × N → [符文融合器] → 高级符文（连续处理，无等待）
```

- 产能加成：+50% 矿物产出（采矿机）、+20% 精华产出（离心机）
- 速度加成：机器处理速度是手搓的 10 倍
- 完全兼容现有 GT 电力、流体管道、AE2 自动合成体系
- 工业路线是**联动福利**，纯魔法玩家完全不受影响

## 4. 魔符与魔法书系统

魔符 + 魔法书是神学魔法系统的**随身主动施法**体系，与祭坛仪式（定点被动）形成互补。

### 4.1 魔符类型

魔符是由符文转化而成的法术组件，分为三种槽位类型：

#### 形态魔符（Form Glyph）— 决定法术交付方式

| 形态 | 描述 | 特性 |
|------|------|------|
| 弹射 (Projectile) | 向前发射弹丸 | 远程、直线飞行 |
| 自施 (Self) | 以自身为中心立即生效 | 即时触发、无弹道 |
| 范围 (Area) | 目标点圆形区域 | AoE，可选中地面/敌人 |
| 射线 (Beam) | 持续射出光束 | 持续施法、穿透多个目标 |
| 触碰 (Touch) | 近身接触目标 | 零距离、法力效率最高 |

#### 效果魔符（Effect Glyph）— 继承源符文元素，决定法术核心效果

| 元素 | 法术效果 | 描述 |
|------|---------|------|
| 火 (Fire) | 伤害 | 火球术、烈焰射线 |
| 水 (Water) | 恢复 | 治愈术、净化水流 |
| 土 (Earth) | 采掘 | 土石崩解（范围采矿魔法） |
| 风 (Air) | 位移 | 冲刺、浮空、加速 |
| 光 (Light) | 照明/净化 | 圣光术、驱散暗影 |
| 暗 (Dark) | 诅咒 | 暗影诅咒、生命汲取 |
| 秩序 (Order) | 屏障 | 秩序壁垒（临时护盾） |
| 混沌 (Chaos) | 随机 | 混沌波动（随机 0.5-2× 效果） |

效果魔符的强度 = 源符文等级的 potency 值（Common=1, Refined=2, Superior=3, Legendary=5）。

#### 增幅魔符（Augment Glyph）— 修饰法术参数

| 元素 | 增幅名称 | 效果 | 可叠加 |
|------|---------|------|--------|
| 火 | 烈怒 | +25% 伤害 / +1 采掘强度 | 是（线性叠加，边际递减） |
| 水 | 延流 | +40% 持续时间 / -20% 冷却 | 是 |
| 土 | 广域 | +1 格效果范围 | 是 |
| 风 | 疾速 | -30% 施法时间 | 是 |
| 光 | 精准 | 弹丸自动追踪最近目标 | 单次（二元效果） |
| 暗 | 穿透 | 无视 25% 防御/抗性 | 是 |
| 秩序 | 节能 | -30% 法力消耗 | 是 |
| 混沌 | 变数 | 效果随机 ×(0.5~2.0) 倍率 | 是（每层独立掷骰） |

### 4.2 符文 → 魔符转化链

```
符文 (Rune)
  │
  ├──[铭刻台]──→ 魔符 (Glyph)  …… 手工路线，1:1
  │     消耗：符文 ×1 + 空白卷轴 ×1
  │
  └──[符文转印机]──→ 魔符 (Glyph)  …… 工业路线，1.2× 产出
        消耗：符文 ×1 + 卷轴 ×1 + 电力
        产出 1.2×（概率额外多产一张）
```

- 魔符继承源符文的**元素**和**等级**
- 形态魔符需单独合成（基底魔符 + 特定材料），不使用符文转化
- 转化后魔符是独立的 `MaterialItem`，可存入背包 / AE2 系统

### 4.3 魔法书

#### 等级与槽位

| 等级 | 槽位总数 | 形态 | 效果 | 增幅 | 预设数 | 法力倍率 | 制作难度 |
|------|---------|------|------|------|--------|---------|---------|
| 基础魔法书 | 3 | 1 | 1 | 1 | 2 | 1.0× | 低 |
| 进阶魔法书 | 4 | 1 | 1 | 2 | 3 | 0.85× | 中 |
| 大师魔法书 | 5 | 1 | 1 | 3 | 4 | 0.7× | 高 |

- **法力倍率**：实际法力消耗 = 编译法术消耗 × 倍率。高级书更节能。
- **预设数**：可保存的独立魔符配置数量。

#### 插槽规则（消耗式）

```
插入魔符 → 魔符即时消耗（销毁），槽位激活
         ├→ 魔符一旦插入即不可取出
         └→ 覆盖已有魔符 → 新魔符消耗 + 旧魔符销毁
```

- 这是**资源决策**：每次换法术都有成本，鼓励玩家提前规划
- 与祭坛台座（符文可取回）形成对比：祭坛灵活，魔法书锁定

#### 法术预设（快捷切换）

```
大师魔法书
├── 预设 1: 弹射·火·火·秩序·风  →  火球术 (追踪+ 低耗)
├── 预设 2: 范围·土·土·土        →  3 格范围采矿
├── 预设 3: 自施·水·水·水        →  强效治愈
└── 预设 4: 弹射·暗·暗·暗·暗     →  诅咒弹射 (高穿透)

快捷键: 数字键 1→预设1, 2→预设2, 3→预设3, 4→预设4
切换冷却: 0.5 秒
```

- 每个预设独立配置 5 个槽位
- 切换预设不消耗魔符（魔符已在插入时消耗）
- 制作预设：打开魔法书 UI → 拖入魔符 → 点击"保存到预设 N"

### 4.4 数据结构

```cpp
// src/core/magic/glyph_def.hpp
enum class GlyphSlotType : uint8_t {
    FORM = 0,    // 形态槽
    EFFECT,      // 效果槽
    AUGMENT,     // 增幅槽
    COUNT
};

enum class SpellForm : uint8_t {
    PROJECTILE = 0,
    SELF,
    AREA,
    BEAM,
    TOUCH,
    COUNT
};

struct GlyphDef {
    ResourceKey item_key;     // 魔符物品 ID
    GlyphSlotType slot_type;
    RuneElement element;      // 效果/增幅魔符：继承自符文；形态魔符：FIRE
    RuneTier tier;
    int potency;              // 继承自符文等级
};

// src/core/magic/spell_book.hpp
struct SpellPreset {
    std::optional<GlyphDef> form;          // 形态槽（为空则无法施法）
    std::optional<GlyphDef> effect;        // 效果槽（为空则无法施法）
    std::vector<GlyphDef> augments;        // 1-3 增幅槽（可留空）

    bool can_cast() const { return form.has_value() && effect.has_value(); }
};

struct SpellBook {
    ResourceKey item_id;
    SpellBookTier tier;

    int active_preset = 0;
    std::array<SpellPreset, 4> presets;    // 预设 0-3

    SpellDef compile(int preset_index = -1) const;
    int get_cast_cost(const SpellDef&) const;
    int get_cooldown_ticks(const SpellDef&) const;
};

// 编译产物
struct SpellDef {
    SpellForm form;
    RuneElement element;
    int potency;               // 基础强度
    float damage_mult;         // 伤害倍率（增幅累加）
    int range;                 // 作用格数
    float cast_time_sec;       // 施法耗时
    int mana_cost;             // 法力消耗
    int cooldown_ticks;        // 冷却（tick @ 20TPS）
    bool homing;               // 弹丸追踪
    float pierce_ratio;        // 穿透比例
    float chaos_factor;        // 混沌随机倍率
};
```

### 4.5 施法流程

```
玩家主手装备魔法书
  └→ 按下右键（施法键）
     ├→ 检查法力: current_mana ≥ spell_cost
     │  ├→ 否: 浮动提示"法力不足"
     │  └→ 是: 继续
     ├→ 检查冷却: 距上次施法 > cooldown_ticks
     │  ├→ 否: 忽略输入
     │  └→ 是: 继续
     ├→ 扣除法力
     ├→ 进入冷却
     └→ 创建法术实例（根据 SpellForm 分发）
        ├→ 弹射: 在玩家位置生成 Projectile 节点，朝向鼠标方向飞行
        ├→ 自施: 直接应用效果到玩家自身
        ├→ 范围: 在鼠标目标坐标生成 AoE 效果区域
        ├→ 射线: 创建持续 Beam 节点，沿面向方向延伸
        └→ 触碰: 检测玩家相邻实体，对其施加效果
```

### 4.7 与祭坛仪式的对比

| 维度 | 魔符魔法书 | 祭坛仪式 |
|------|-----------|---------|
| 使用场景 | 随身、移动 | 定点、固定位置 |
| 触发方式 | 右键即时施法 | 符文排列→确认→等待 |
| 反应速度 | 即时 / 短冷却 | 3-10 秒仪式动画 |
| 效果类型 | 伤害/恢复/位移/采矿 | 祝福/地形/传送/占卜 |
| 资源成本 | 符文→魔符（转化） + 魔符插入（消耗） | 符文放置（可消耗或不消耗） |
| 自动化 | 符文/魔符生产可自动；施法纯手动 | 符文生产可自动；仪式纯手动 |
| 适用玩家 | 探索/战斗型魔法玩家 | 建造/工业型魔法玩家 |

## 5. 祭坛仪式系统

### 5.1 祭坛结构（3D 方块世界布局）

```
基础祭坛（4 台座）：            高级祭坛（8 台座）：

    ┌───┐                      ┌───┐   ┌───┐
    │ 0 │                      │ 0 │   │ 1 │
    └─┬─┘                      └─┬─┘   └─┬─┘
      │                          │       │
┌─┬─┬─┼─┬─┬─┐                ┌─┬─┼─┬─┬─┬─┼─┬─┐
│3│ │3│ A │1│1│                │7│ │7│ A │1│1│
```

### 5.2 仪式配方

```cpp
// src/core/magic/ritual_recipe.hpp
namespace science_and_theology::magic {

struct RitualPedestalSlot {
    RuneElement element;      // 要求元素
    RuneTier min_tier;        // 最低等级
    bool strict_element;      // 是否必须严格匹配元素（false=接受任意）
};

struct RitualEffect {
    enum class Type : uint8_t {
        NONE = 0,
        MACHINE_BLESSING,     // 机器祝福
        TOOL_ENCHANTMENT,     // 工具附魔
        TERRAIN_ALTERATION,   // 地形改造
        PLAYER_BUFF,          // 玩家增益
        WORLD_EVENT,          // 世界事件（开关遗迹门等）
        CURSE,                // 诅咒
        TELEPORTATION,        // 传送链接
        DIVINATION,           // 占卜
        MANA_EXPANSION,       // 法力扩容（永久提升上限）
        COUNT
    };
    
    Type type;
    nlohmann::json params;  // 效果参数，序列化为 JSON
    int duration_ticks;     // 持续时长（0 = 瞬时/永久）
};

struct RitualRecipe {
    ResourceKey id;
    std::string display_name;
    
    // 台座配置（4/8 个插槽）
    std::vector<RitualPedestalSlot> pedestals;
    
    int mana_cost;          // 法力消耗
    int duration_ticks;     // 仪式执行时长（60-200 tick = 3-10 秒 @ 20TPS）
    bool consume_runes;     // 仪式后符文是否消耗
    
    RitualEffect effect;
};

}  // namespace science_and_theology::magic
```

### 5.3 仪式执行流程（3-10 秒）

```
玩家走到核心祭坛 [A]
  └→ 右键台座 [0-3/0-7]，逐个放入符文
     └→ 台座显示符文图标，列表更新
        └→ 实时匹配：符文排列 → 查找配方
           ├→ 未匹配：无反应
           └→ 匹配成功：显示效果预览（悬浮文字）
               └→ 玩家按下确认键（E）
                  ├→ 检查法力：mana ≥ mana_cost？
                  │  ├→ 否：提示"法力不足"
                  │  └→ 是：扣除法力，开始仪式
                  ├→ 播放动画（3-10 秒）
                  │  └→ 粒子特效、光柱、符文旋转
                  ├→ 仪式完成
                  │  ├→ apply_effect() 效果生效
                  │  └→ 若 consume_runes = true，清空台座符文
                  └→ 结束
```

## 8. 与现有系统的集成

### 8.1 MaterialRegistry

符文、魔符、法力药水、魔法书、祭坛建材均通过 `MaterialRegistry` 注册为标准的 `MaterialItem`。物品定义（`ItemDef`）直接复用现有体系，存储和传输无需额外适配。

### 8.2 EventBus（事件桥接）

仪式效果和施法效果通过现有的 `EventBus` 发布事件，避免魔法核心与工业核心直接耦合：

```cpp
// 仪式+施法事件（src/core/simulation/event_types.hpp 中新增）
MachineBlessEvent  { machine_id, effect_type, multiplier, duration_ticks }
PlayerBuffEvent    { player_id, buff_type, multiplier, duration_ticks }
SpellCastEvent     { caster_id, spell_form, element, damage, position }
WorldEvent         { event_type, position, params }
```

工业系统（`MachineSystem`）订阅 `MachineBlessEvent`，自行处理附魔计算。战斗系统订阅 `SpellCastEvent` 处理伤害。

### 8.3 世界机制系统（MechanismManager）

仪式引发的世界事件（遗迹门开关、天气改变等）通过现有的 `MechanismManager` 标记系统管理。与 Sun Altar → Ruin Gate 的现有机制完全兼容。

### 8.4 玩家系统（PlayerController + Inventory + Equipment）

- `ManaPool` 挂在 `GDPlayerEquipment` 旁边，随玩家数据序列化
- 法力药水/符文/魔符是标准的 `MaterialItem`，存入 `PlayerInventory`
- `SpellBook` 是装备物品，放入主手槽后右键触发施法（而非 E 键交互）
- 法力条 UI 在 `PlayerController.gd` 的 HUD 组件中渲染
- 数字键 1-4 切换魔法书预设法术

### 8.5 存档系统（ChunkSerializer）

- `ManaPool` 数据序列化到 `PlayerData` 中
- 祭坛台座上放置的符文作为世界状态序列化到对应 Chunk
- 永久仪式效果（传送链接、法力扩容）记录在 WorldData

## 11. 设计约束

1. **零 Godot 依赖**：`src/core/magic/` 下所有代码为纯 C++20，不引入 `godot-cpp` 头文件。
2. **最小化耦合**：魔法核心与工业核心只通过 `EventBus` 事件和 `MaterialRegistry` 间接交互，不互相 `#include`。
3. **存档前向兼容**：`ManaPool` 字段序列化新增，旧存档加载时设默认值（0 mana / 100 max）。
4. **可测试性**：核心逻辑可在独立 C++ 测试中验证（`tests/`），无需启动 Godot。
5. **扩展预留**：`RitualEffect::params` 使用 JSON 串行化，可后续追加参数字段而不改数据结构定义。
