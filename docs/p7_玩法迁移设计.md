# P7 — 玩法迁移设计

> P7.2.1 状态：已完成。game-owned `MachineTickSystem`、炉子内容定义、运行态事件边界和热重载配方快照测试均已接入；其余四类机器按同一运行态契约迁移。

> P7.3 状态：任务定义、`QuestRegistry`、值快照和 AngelScript objective 注册已实现。P7.4 的 per-player 任务进度文件 provider、严格格式校验、恢复备份、dedicated-server 玩家生命周期，以及 client/server 共用的世界 load/bootstrap/save 生命周期已实现；真实玩家/机器事件 producer、reward sink 和机器运行态持久化尚未实现。

> 本文是 P7 的实施契约和任务系统唯一设计说明；总体边界和当前能力以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准。

## 1. 决策与边界

- 玩法内容统一使用 AngelScript `.as`，由 `ScriptManager` 在运行时编译；**不使用 DLL**。这样没有 ABI、卸载顺序和符号冲突问题，热重载只需要重建 AngelScript 模块。
- 迁移只保留自研引擎最新接口。旧 Godot / GDScript / `godot-cpp` 接口不新增兼容层。
- C++ 继续负责性能敏感、确定性和外部资源边界：机器 tick、region 存档、协议编解码、SteamP2P 传输和 ECS 调度。脚本负责内容定义与可热更玩法规则。
- P7 不修改 P6 UI 工作区的未提交重构；P6 构建恢复后，P7 模块应自然接入同一工程。

## 3. 注册与热重载规则

1. 每项脚本注册都带稳定 `ScriptId`；事件只存 `(ScriptId, callback_id)`，绝不存 AngelScript 函数指针。
2. 内容 ID 是全局唯一。脚本可覆盖同 ID 的内建定义；两个脚本争抢同一 ID 会失败，避免加载顺序影响玩法。
3. 重载先调用 `begin_reload`，临时恢复内建定义；脚本重新编译并执行入口后调用 `commit_reload`。
4. 编译、绑定或入口执行失败时调用 `rollback_reload`，原脚本的注册内容原样恢复。
5. `StateStore` 按 `ScriptId` 隔离，跨重载保留；仅引擎会话关闭时清空。玩家持久化状态不放入脚本全局变量。

## 5. P7.2 机器运行契约

- `MachineRuntimeComponent` 是 game-owned ECS 持久运行态：输入物品、输出槽、能量、进度与 `MachineRecipeSnapshot` 都不保存脚本函数、模块或 VM 指针。
- 机器按稳定 `EntityGuid` 顺序 tick。开始任务时先验证输出容量、预扣一份输入，并复制当前 `RecipeDefinition` 为快照；脚本重载只影响之后启动的任务。
- 配方完成前输出槽满会进入 `WaitingForOutput`，不会丢失已预扣输入或已完成进度；能量不足会进入 `WaitingForEnergy`，不会前进进度。
- `IMachineTickEventSink` 已作为 UI、任务、存档脏标记和 replication 的跨模块事件接口预留。当前没有监听者时系统仍独立运行。
- 日志仅在“无匹配配方 / 能量不足 / 输出阻塞”及其恢复时记录，避免每 tick、每个配方完成一次的高频日志。

## 6. P7.3 任务运行契约

- `GameContentRegistry` 只拥有可热重载 `QuestDefinition`；`QuestRegistry` 按 `QuestPlayerId + quest id` 拥有 `QuestProgressRecord`。定义重载绝不覆写玩家 count、完成状态或 orphan record。
- objective 使用稳定 id 和明确 kind：`acquire_item`、`craft_item`、`mine_block`、`place_machine`、`reach_tick`、`custom`；不解析字符串编码规则。
- `QuestProgressRecord` / `IQuestProgressPersistence` 位于独立的 `quest_progress.h`，使 `snt_game_world_save` 只依赖稳定值，不反向链接 `snt_game_simulation`。`QuestRegistry::load_player_progress()` / `save_player_progress()` 只在显式玩家生命周期边界调用 provider，fixed tick 不做文件 I/O。
- `QuestPlayerId` 直接使用认证后的 game account id：`steam:<SteamID64>` 与 `local-name:<validated UTF-8 name>` 互不重叠。未来 player/server lifecycle 不得以显示名重新构造任务 key；本地同名登录共享该 key 并遵循会话接管语义。
- `ScienceAndTheologySimulationSession` 在脚本 reload 后的 simulation main-thread fixed tick 同步 quest revision；任务状态变换记录低频日志，objective 增量不逐条记录。

### 6.1 所有权

| 域 | 所有者 | 当前状态 |
| --- | --- | --- |
| 热重载定义 | `GameContentRegistry` | 稳定 quest id、结构化 objective、`ScriptId` 事务和 revision 已实现 |
| 玩家运行态 | `game/quest/QuestRegistry` | 按稳定 game account id + quest id 保存状态和计数 |
| 任务效果 | `IQuestRewardSink` | 接口已声明，尚无生产 reward provider |
| UI、replication、save 脏标记 | `IQuestLifecycleSink` | 接口已声明，只传递值事件 |
| 跨重启持久化 | `IQuestProgressPersistence` / `GameSaveManager` | `SNTQ` v1 provider 与 dedicated-server 玩家生命周期已实现 |

`QuestProgressRecord` 只保存 quest id、状态、objective count、完成 tick、完成次数和奖励领取位；不保存 AngelScript 对象、ECS handle、原始指针或源码路径。

### 6.2 内容接口与状态机

AngelScript 在 `snt_register()` 中以稳定 ID 注册定义：

```angelscript
snt_register_quest("stone_age.smelt_iron", "Smelt Iron", "Craft iron ingots");
snt_add_quest_objective(
    "stone_age.smelt_iron", "craft.iron", "craft_item", "iron_ingot", 2);
snt_add_quest_prerequisite("bronze_age.forge_tool", "stone_age.smelt_iron");
```

同一任务内的 objective id 必须唯一；除 `reach_tick` 外都必须提供 target id。状态只按下列转换，未来玩家、机器或网络 producer 必须先携带认证后的 stable account id：

```text
Locked -- prerequisites met --> Available -- accept / auto_start --> InProgress
InProgress -- all objectives met --> Completed
Completed -- reset_repeatable --> Available
```

### 6.3 重载和持久化边界

- 相同 quest/objective id 的定义重载保留既有 count 和完成状态；新 objective 从零开始；暂时移除的任务进度作为 orphan record 保留，恢复同 id 时继续使用。
- `record_progress()` 与 `update_inventory()` 只接收按值事件或查询，不能以显示名替代 account id。
- `QuestRegistry::load_player_progress()` 与 `save_player_progress()` 只通过 provider 交换值。任务状态转换记录低频 Info；缺失 prerequisite 在内容 revision 变化时仅告警一次，objective 每次增量不写 Info。

## 7. P7.4 存档契约

- `ScienceAndTheologySimulationSession` 是当前世界持久化的唯一编排者。`persistence.world_save_enabled` 为真时，它把相对 `persistence.universe_save_dir` 解析到 `user_root`，以 `world_dimension_id`、试玩 seed 和 `universe_mode` 构造 `GameWorldPersistenceLifecycle`；图形 client 与无头 server 因而使用同一条 load/bootstrap/shutdown-save 路径。
- `load_existing()` 只在空的 chunk/sidecar registry 上运行，并在 demo bootstrap 前验证 universe header、dimension header、seed、mode 和 dimension id。没有任何 header 才表示首次启动；已识别但没有 region 的空 universe 也算已加载，不能被 demo 数据覆盖。损坏或不匹配会使会话创建失败，`world_ready_` 保持 false，关闭时不会写入空替代存档。
- 受控关闭只在 world ready 后保存；顺序为当前 dimension 的 region/sidecar，随后写入 universe header。当前格式仍是 game payload v9 + raw region framing v2，不维护旧格式读取或自动迁移。

- 每个 universe save 根下使用 `players/player_{hex_player_id}.quest`。hex 文件名避免玩家 id 进入路径，文件内仍保存并校验原 player id。
- 格式为 `SNTQ` v1：task state、计数、完成 tick、完成次数和 reward claimed 只使用值类型；读取要求当前版本、合法字段和 EOF，尾随字节或损坏内容直接拒绝。
- 写入先完成 `.tmp`，再暂存旧 primary 为 `.bak` 并提升新文件。只有 primary 缺失时才读取 `.bak`，并记录一条低频 Warn；primary 损坏不会被旧备份静默遮掩。
- `ScienceAndTheologyServerSession` 在网络启用时将 `persistence.universe_save_dir` 解析到 `user_root`，再构造 `GameServerPlayerLifecycle`。首次认证加载该 account 的进度；同账号顶号只转移内存状态，正常断线和关服保存全部 resident account。fixed tick 不做文件 I/O；机器运行态的存档策略仍由后续生命周期实现。
- `MachineRuntimeComponent` 目前不写入世界存档。它虽有稳定 `EntityGuid`，但尚无稳定的 `EntityGuid -> ChunkKey/block-entity` 空间锚点，也没有能从该锚点重建 ECS 实体和机器组件的生产工厂；现在写独立机器文件会留下无法恢复到空间玩法状态的数据。

## 8. 选择说明

原阶段计划中的“示例脚本 DLL”与 AngelScript 的 `.as` 运行时编译互相矛盾。两种路线的取舍如下：

- `.as` 模块（采用）：重载无需 ABI 处理、可以直接执行内容脚本；缺点是需要维护 C++→AngelScript binding。
- 原生 DLL：可复用现有 C++，但跨平台装载、符号、内存归属和卸载安全复杂，且与 AngelScript 主线重复。

机器持久化尚未选择实现，候选边界如下：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| chunk sidecar 锚定（推荐） | 机器与其方块/ChunkKey 的空间所有权一致；随 region 载入、卸载和保存 | 需要补 placement/recovery 映射与 ECS 重建工厂 |
| universe 级独立机器文件 | 初始序列化较短，不改 chunk sidecar 格式 | 脱离空间所有权，难以处理 chunk 卸载、损坏恢复和跨维度；不建议采用 |

P7 采用前者；原生 DLL 不进入玩法内容管线。

任务不迁回旧 `src/core/quest/QuestSystem`，也不进入 `snt_engine` 通用模块：前者会重新引入 Godot event bus、旧序列化和旧命名，后者会把游戏玩法语义泄漏到引擎。玩家进度采用 `players/*.quest`，而不是写进每个 `planet_data.bin` 或一个 universe 共用文件，避免多玩家互相覆盖并保持空间存档与玩家状态解耦。
