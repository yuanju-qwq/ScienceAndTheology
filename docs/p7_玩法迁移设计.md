# P7 — 玩法迁移设计

> P7.2.1 状态：已完成。`MachineTickSystem`、炉子内容定义、运行态事件边界和热重载配方快照测试均已接入；其余四类机器按同一运行态契约迁移。

> 本文是 P7 的实施契约；总体边界和当前能力以 `自研引擎架构设计.md` 为准，本文件只维护玩法迁移顺序和已冻结接口。

## 1. 决策与边界

- 玩法内容统一使用 AngelScript `.as`，由 `ScriptManager` 在运行时编译；**不使用 DLL**。这样没有 ABI、卸载顺序和符号冲突问题，热重载只需要重建 AngelScript 模块。
- 迁移只保留自研引擎最新接口。旧 Godot / GDScript / `godot-cpp` 接口不新增兼容层。
- C++ 继续负责性能敏感、确定性和外部资源边界：机器 tick、region 存档、协议编解码、SteamP2P 传输和 ECS 调度。脚本负责内容定义与可热更玩法规则。
- P7 不修改 P6 UI 工作区的未提交重构；P6 构建恢复后，P7 模块应自然接入同一工程。

## 3. 注册与热重载规则

1. 每项脚本注册都带稳定 `ScriptId`；事件只存 `(ScriptId, callback_id)`，绝不存 AngelScript 函数指针。
2. 内容 ID 是全局唯一。脚本可覆盖同 ID 的内建定义；两个脚本争抢同一 ID 会失败，避免加载顺序影响玩法。
3. 重载先调用 `begin_reload`，临时恢复内建定义；脚本重新编译并执行入口后调用 `commit_reload`。
4. 编译、绑定或入口执行失败时调用 `rollback_reload`，原脚本的注册内容原样恢复。
5. `StateStore` 按 `ScriptId` 隔离，跨重载保留；仅引擎会话关闭时清空。玩家持久化状态不放入脚本全局变量。

## 5. P7.2 机器运行契约

- `MachineRuntimeComponent` 是 ECS 持久运行态：输入物品、输出槽、能量、进度与 `MachineRecipeSnapshot` 都是引擎拥有的数据；其中绝不保存脚本函数、模块或 VM 指针。
- 机器按稳定 `EntityGuid` 顺序 tick。开始任务时先验证输出容量、预扣一份输入，并复制当前 `RecipeDefinition` 为快照；脚本重载只影响之后启动的任务。
- 配方完成前输出槽满会进入 `WaitingForOutput`，不会丢失已预扣输入或已完成进度；能量不足会进入 `WaitingForEnergy`，不会前进进度。
- `IMachineTickEventSink` 已作为 UI、任务、存档脏标记和 replication 的跨模块事件接口预留。当前没有监听者时系统仍独立运行。
- 日志仅在“无匹配配方 / 能量不足 / 输出阻塞”及其恢复时记录，避免每 tick、每个配方完成一次的高频日志。

## 6. 选择说明

原阶段计划中的“示例脚本 DLL”与 AngelScript 的 `.as` 运行时编译互相矛盾。两种路线的取舍如下：

- `.as` 模块（采用）：重载无需 ABI 处理、可以直接执行内容脚本；缺点是需要维护 C++→AngelScript binding。
- 原生 DLL：可复用现有 C++，但跨平台装载、符号、内存归属和卸载安全复杂，且与 AngelScript 主线重复。

P7 采用前者；原生 DLL 不进入玩法内容管线。
