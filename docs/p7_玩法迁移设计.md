# P7 — 玩法迁移设计

> P7.2.4 状态：已完成。game-owned `MachineTickSystem` 已支持多输入槽和显式手动激活；炉子、pit kiln、charcoal pit、bloomery 与 anvil 的配方内容、运行态事件边界、v13 持久化和热重载配方快照测试均已接入。手动任务以稳定认证 account id 归属，完成事件携带 tick、account 与结构化产物并在提交后清空归属；`MachineInteractionService` 以内容定义的覆盖、点火、结构和工具 tag 要求校验 main-thread value context 后排队激活；`MachineStructureValidator` 已按旧 bloomery 的 3x3x2/17-cell 规则从 `ChunkRegistry` fail-closed 地生成结构结论。认证玩家、库存扣除、方块编辑和已锚定机器收集已由 `GameServerPlayerInteractionService` 接入；协作联机路径把客户端前置条件作为可信提示，不为反作弊重新生成这些结论。

> P7.3 状态：任务定义、`QuestRegistry`、结构化 reward、值快照和 AngelScript objective/reward 注册已实现。`GameServerQuestEventService` 已将已提交的挖矿、方块放置/机器产物领取库存变化和带认证 account 的机器完成事件接入任务进度，并作为 item reward 的主机库存事务 sink；领取已由 `SNTG` v7 的专用 `QuestClaimReward` 命令显式触发。`SNTG` v8 已将认证账号的 `QuestBookSnapshot` 经 `GameServerQuestBookReplication`、per-peer value baseline 和 `GameClientQuestBookState` 复制到客户端；任务书 UI 仍未实现，且只能消费这一只读快照。P7.4 的 per-player 任务进度文件 provider、严格格式校验、恢复备份、dedicated-server 玩家生命周期、修订号门控的受控自动存档、client/server 共用的世界 load/bootstrap/save 生命周期，以及 chunk-sidecar 锚定的机器运行态恢复/保存已实现。

> 本文是 P7 的实施契约和任务系统唯一设计说明；总体边界和当前能力以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准。

## 1. 决策与边界

- 玩法内容统一使用 AngelScript `.as`，由 `ScriptManager` 在运行时编译；**不使用 DLL**。这样没有 ABI、卸载顺序和符号冲突问题，热重载只需要重建 AngelScript 模块。
- 迁移只保留自研引擎最新接口。旧 Godot / GDScript / `godot-cpp` 接口不新增兼容层。
- C++ 继续负责性能敏感、确定性和外部资源边界：机器 tick、region 存档、协议编解码、SteamP2P 传输和 ECS 调度。脚本负责内容定义与可热更玩法规则。
- P7 不修改 P6 UI 工作区的未提交重构；P6 构建恢复后，P7 模块应自然接入同一工程。

## 3. 注册与热重载规则

1. 每项脚本注册都带稳定 `ScriptId`；事件只存 `(ScriptId, callback_id)`，绝不存 AngelScript 函数指针。
2. 内容 ID 是全局唯一。脚本可覆盖同 ID 的内建定义；两个脚本争抢同一 ID 会失败，避免加载顺序影响玩法。
3. 重载先调用 `begin_reload`，临时恢复内建定义；脚本重新编译并执行入口后调用 `commit_reload`。
4. 编译、绑定或入口执行失败时调用 `rollback_reload`，原脚本的注册内容原样恢复。
5. `StateStore` 按 `ScriptId` 隔离，跨重载保留；仅引擎会话关闭时清空。玩家持久化状态不放入脚本全局变量。

## 5. P7.2 机器运行契约

- `MachineRuntimeComponent` 是 game-owned ECS 持久运行态：输入槽、输出槽、能量、进度、待激活标记、手动任务的稳定 account owner 与 `MachineRecipeSnapshot` 都不保存脚本函数、模块、peer id 或 VM 指针。
- 机器按稳定 `EntityGuid` 顺序 tick。开始任务时先验证输出容量、预扣全部配方输入，并复制当前 `RecipeDefinition` 为快照；脚本重载只影响之后启动的任务。
- `MachineDefinition::requires_manual_activation` 为 pit kiln、charcoal pit、bloomery 和 anvil 提供显式启动边界；`MachineActivationRequirements` 以值定义 cover、ignition、valid structure 和可选 tool tag。输入满足时它们进入 `WaitingForActivation`；生产 player-command 路径在 simulation main thread 通过 `GameServerPlayerInteractionService` 调用 `MachineInteractionService::request_manual_activation()` 排队一次请求，worker 仍只消费该一次性标记。客户端的 cover/ignition/structure/tool 提示是选定联机模型中的可信输入：服务端只做会话、dimension/reach、加载 chunk、库存和 sidecar 完整性检查，不为反作弊重扫结构或重算 tool tag。`MachineStructureValidator` 仍保留给内容/bootstrap 校验和需要严格玩法约束的未来模式；当前协作路径不以它复算客户端提示。
- 每个持久化机器都由同 chunk sidecar 的 `BlockEntityPlacement(MACHINE)` 锚定。`MachineRuntimePersistenceRecord` 保存 anchor `EntityId`、稳定 `EntityGuid`、手动任务 owner 与值快照；simulation session 只在主线程启动/关闭边界恢复或捕获，worker 不访问 sidecar。
- 配方完成前输出槽满会进入 `WaitingForOutput`，不会丢失已预扣输入或已完成进度；能量不足会进入 `WaitingForEnergy`，不会前进进度。
- `IMachineTickEventSink` 是 UI、任务、存档脏标记和 replication 的跨模块值事件接口。dedicated server 以 `GameServerQuestEventService` 消费完成事件中的 tick、认证 account 和产物，推进对应 `craft_item`；没有监听者时系统仍独立运行。
- 日志仅在“无匹配配方 / 能量不足 / 输出阻塞”及其恢复时记录，避免每 tick、每个配方完成一次的高频日志。
- `game/scripts/p7_bootstrap.as` 已注册炉子自动烧炼；pit kiln 手动烧制 bowl/jug（8000 ticks）、crucible（12000 ticks）和 brick（6000 ticks）；charcoal pit 手动将 `wood_dust x16` 转为 `charcoal x8`（24000 ticks）；bloomery 手动将 `iron_crushed x5 + charcoal x5` 转为 `iron_bloom`（12000 ticks）；anvil 手动将 `iron_bloom` 锻为 `wrought_iron_ingot`（1 tick）。它们均为零能耗。pit kiln/charcoal pit 要求 cover + ignition，bloomery 要求 ignition + valid structure，anvil 要求 `hammer` tool tag。
- AngelScript 使用 `snt_register_recipe(..., first_input_item, first_input_count, ...)` 注册首个输入，使用 `snt_add_recipe_input(recipe_id, item_id, count)` 追加输入；`snt_register_machine(..., requires_manual_activation)` 创建机器后以 `snt_set_machine_activation_requirements(machine_id, requires_cover, requires_ignition, requires_valid_structure, required_tool_tag)` 声明启动前置条件，不保留旧签名。

## 6. P7.3 任务运行契约

- `GameContentRegistry` 只拥有可热重载 `QuestDefinition`；`QuestRegistry` 按 `QuestPlayerId + quest id` 拥有 `QuestProgressRecord`。定义重载绝不覆写玩家 count、完成状态或 orphan record。
- objective 使用稳定 id 和明确 kind：`acquire_item`、`craft_item`、`mine_block`、`place_machine`、`reach_tick`、`custom`；不解析字符串编码规则。
- reward 使用 `QuestRewardKind::kItem` 或 `kUnlockQuest` 与独立 target id/count；`QuestRegistry::claim_reward()` 只处理显式领取，item 必须由主机 reward sink 整笔提交，unlock 由 registry 更新稳定任务状态。
- `QuestProgressRecord` / `IQuestProgressPersistence` 位于独立的 `quest_progress.h`，使 `snt_game_world_save` 只依赖稳定值，不反向链接 `snt_game_simulation`。`QuestRegistry::load_player_progress()` / `save_player_progress()` 只在显式玩家生命周期边界调用 provider，fixed tick 不做文件 I/O。
- `QuestPlayerId` 直接使用认证后的 game account id：`steam:<SteamID64>` 与 `local-name:<validated UTF-8 name>` 互不重叠。未来 player/server lifecycle 不得以显示名重新构造任务 key；本地同名登录共享该 key 并遵循会话接管语义。
- `ScienceAndTheologySimulationSession` 在脚本 reload 后的 simulation main-thread fixed tick 同步 quest revision；任务状态变换记录低频日志，objective 增量不逐条记录。

### 6.1 所有权

| 域 | 所有者 | 当前状态 |
| --- | --- | --- |
| 热重载定义 | `GameContentRegistry` | 稳定 quest id、结构化 objective、`ScriptId` 事务和 revision 已实现 |
| 玩家运行态 | `game/quest/QuestRegistry` | 按稳定 game account id + quest id 保存状态和计数 |
| 任务效果 | `QuestRegistry` + `IQuestRewardSink` | `kUnlockQuest` 由 registry 改变进度状态；`GameServerQuestEventService` 以单个主机库存事务提交全部 `kItem`，失败时不设置 `reward_claimed` |
| UI、replication、save 脏标记 | `IQuestLifecycleSink` | 接口已声明，只传递值事件 |
| 跨重启持久化 | `IQuestProgressPersistence` / `GameSaveManager` | `SNTQ` v1 provider 与 dedicated-server 玩家生命周期已实现 |

`QuestProgressRecord` 只保存 quest id、状态、objective count、完成 tick、完成次数和奖励领取位；不保存 AngelScript 对象、ECS handle、原始指针或源码路径。

### 6.2 内容接口与状态机

AngelScript 在 `snt_register()` 中以稳定 ID 注册定义：

```angelscript
snt_register_quest("stone_age.smelt_iron", "Smelt Iron", "Craft iron ingots");
snt_add_quest_objective(
    "stone_age.smelt_iron", "craft.iron", "craft_item", "iron_ingot", 2);
snt_add_quest_prerequisite("bronze_age.forge_tool", "stone_age.smelt_iron");
snt_add_quest_item_reward("stone_age.smelt_iron", "wrought_iron_ingot", 1);
snt_add_quest_unlock_reward("stone_age.smelt_iron", "bronze_age.forge_tool");
```

同一任务内的 objective id 必须唯一；除 `reach_tick` 外都必须提供 target id。状态只按下列转换，未来玩家、机器或网络 producer 必须先携带认证后的 stable account id：

```text
Locked -- prerequisites met --> Available -- accept / auto_start --> InProgress
InProgress -- all objectives met --> Completed
Completed -- explicit claim --> Completed (reward_claimed = true)
Completed -- reset_repeatable --> Available
```

### 6.3 重载和持久化边界

- 相同 quest/objective id 的定义重载保留既有 count 和完成状态；新 objective 从零开始；暂时移除的任务进度作为 orphan record 保留，恢复同 id 时继续使用。
- `record_progress()` 与 `update_inventory()` 只接收按值事件或查询，不能以显示名替代 account id。
- `GameServerPlayerInteractionService` 只在方块编辑或机器产物领取已经提交后发布 event；bridge 对挖矿额外记录 `mine_block`，对挖矿/放置/领取以认证玩家的主机库存刷新 `acquire_item`。`MachineTickSystem` 只在完成后以手动任务 owner 的 account 和产物推进 `craft_item`。
- 领取不会随完成或接取自动发生：未来任务书专用命令调用 `claim_reward()`；若 item transaction 无法整笔提交，`reward_claimed` 保持 false，玩家可在库存腾出空间后重试。
- `QuestRegistry::load_player_progress()` 与 `save_player_progress()` 只通过 provider 交换值。任务状态转换记录低频 Info；缺失 prerequisite 在内容 revision 变化时仅告警一次，objective 每次增量不写 Info。

## 7. P7.4 存档契约

- `ScienceAndTheologySimulationSession` 是当前世界持久化的唯一编排者。`persistence.world_save_enabled` 为真时，它把相对 `persistence.universe_save_dir` 解析到 `user_root`，以 `world_dimension_id`、试玩 seed 和 `universe_mode` 构造 `GameWorldPersistenceLifecycle`；图形 client 与无头 server 因而使用同一条 load/bootstrap/shutdown-save 路径。
- `load_existing()` 只在空的 chunk/sidecar registry 上运行，并在 demo bootstrap 前验证 universe header、dimension header、seed、mode 和 dimension id。没有任何 header 才表示首次启动；已识别但没有 region 的空 universe 也算已加载，不能被 demo 数据覆盖。损坏或不匹配会使会话创建失败，`world_ready_` 保持 false，关闭时不会写入空替代存档。
- 受控关闭只在 world ready 后保存；它先捕获所有已锚定 ECS 机器到 sidecar，再保存当前 dimension 的 region/sidecar，随后写入 universe header。当前格式是 game payload v13 + raw region framing v2；v13 保存 in-flight 手动机器任务的认证 owner，任一 region 或 chunk payload 被拒绝都会使 dimension load 失败，不维护旧格式读取或自动迁移。

- 每个 universe save 根下使用 `players/player_{hex_player_id}.quest`。hex 文件名避免玩家 id 进入路径，文件内仍保存并校验原 player id。
- 格式为 `SNTQ` v1：task state、计数、完成 tick、完成次数和 reward claimed 只使用值类型；读取要求当前版本、合法字段和 EOF，尾随字节或损坏内容直接拒绝。
- 写入先完成 `.tmp`，再暂存旧 primary 为 `.bak` 并提升新文件。只有 primary 缺失时才读取 `.bak`，并记录一条低频 Warn；primary 损坏不会被旧备份静默遮掩。
- `ScienceAndTheologyServerSession` 在网络启用时将 `persistence.universe_save_dir` 解析到 `user_root`，再构造 `GameServerPlayerLifecycle`。首次认证加载该 account 的进度；同账号顶号只转移内存状态，正常断线和关服强制保存全部 resident account。`QuestRegistry` 为每个玩家维护只在持久化值变化时递增的内存 revision；server 在 `after_fixed_tick` barrier 按 `persistence.player_progress_autosave_interval_ticks`（默认 1200，即 20 TPS 下约 60 秒）检查并只写入 revision 已变化的 resident account。值为 0 时关闭定时 autosave，断线和关服保存仍保留；QuestRegistry 的 fixed tick 自身不做文件 I/O。
- `GameMachineRuntimePersistence` 只接受同 chunk sidecar 内的 `BlockEntityPlacement(MACHINE)` 作为 anchor；启动时验证 anchor、root chunk、record、重复 Guid 和冲突 ECS entity 后恢复，关闭时拒绝未锚定的 live `MachineRuntimeComponent`，不会静默丢失状态。`create_anchored_machine()` / `remove_anchored_machine()` 是未来放置/拆除系统唯一应调用的 lifecycle 边界。

## 8. 选择说明

原阶段计划中的“示例脚本 DLL”与 AngelScript 的 `.as` 运行时编译互相矛盾。两种路线的取舍如下：

- `.as` 模块（采用）：重载无需 ABI 处理、可以直接执行内容脚本；缺点是需要维护 C++→AngelScript binding。
- 原生 DLL：可复用现有 C++，但跨平台装载、符号、内存归属和卸载安全复杂，且与 AngelScript 主线重复。

机器持久化已选择并实现 chunk-sidecar 锚定；备选方案保留为已拒绝的设计记录：

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| chunk sidecar 锚定（已选择） | 机器与其方块/ChunkKey 的空间所有权一致；随 region 载入、卸载和保存 | placement/recovery 必须经显式 lifecycle API，未锚定机器会拒绝保存 |
| universe 级独立机器文件（未采用） | 初始序列化较短，不改 chunk sidecar 格式 | 脱离空间所有权，难以处理 chunk 卸载、损坏恢复和跨维度 |

P7 采用前者；原生 DLL 不进入玩法内容管线。

任务不迁回旧 `src/core/quest/QuestSystem`，也不进入 `snt_engine` 通用模块：前者会重新引入 Godot event bus、旧序列化和旧命名，后者会把游戏玩法语义泄漏到引擎。玩家进度采用 `players/*.quest`，而不是写进每个 `planet_data.bin` 或一个 universe 共用文件，避免多玩家互相覆盖并保持空间存档与玩家状态解耦。

`QuestRegistryTest` 覆盖接取、进度、完成、前置解锁、显式 item/unlock reward、无 sink 时不置 claimed、自动接取、库存/时间 objective、repeatable reset、热重载保留、snapshot restore、文件化重启、尾随数据拒绝和 backup recovery。`GameServerQuestEventServiceTest` 覆盖已提交交互、机器产物、认证玩家库存事务和显式奖励领取；`GameServerPlayerLifecycleTest` 覆盖真实 `SNTQ` 文件的顶号内存转移、断线保存、新 server 生命周期加载，以及到期/变化 revision 才写入的自动保存；`GameQuestBookReplicationTest` 覆盖严格 task-book codec、尾随数据拒绝和客户端 revision 门控，`GameServerQuestBookReplicationTest` 覆盖认证账号 source，`GameServerPlayerReplicationTest` 覆盖 value baseline 的无变更抑制、upsert/remove 与断线清理。`MachineTickSystemTest` 覆盖稳定 recipe id 选择、多输入、service 排队后的手动启动和完成事件的 tick/account/output，`MachineInteractionServiceTest` 覆盖 reachability、cover、ignition、structure、tool tag、重复请求和非手动机器拒绝，`MachineStructureValidatorTest` 覆盖跨 chunk 的完整 bloomery、缺 chunk、material mismatch 和 air material 拒绝；`GameMachineRuntimePersistenceTest` 覆盖 chunk anchor 创建/移除、恢复、capture、手动任务 owner round-trip 与未锚定拒绝，`GameSimulationSessionTest` 覆盖真实 world save 后的机器重启恢复；`P7ScriptApiTest` 覆盖 AngelScript 结构化 objective/reward 和机器 activation requirement 注册。

下一步实现只消费 `GameClientQuestBookState` 的任务书 UI/presentation；领取继续通过专用 `QuestClaimReward` command，不得借用任务接取命令或通用方块交互 payload，也不得让客户端直接写 world/inventory/save。

机器真实交互已具备 server-owned player state、基础 reach/loaded-chunk 检查、背包原子转移，以及 activation/output collect 入口。`GameServerPlayerInteractionService` 必须始终是网络 payload 与 `MachineInteractionService` 之间的唯一适配层：它先解析 typed command、执行低成本共享状态完整性检查，再把客户端可信的 `MachineActivationContext` 提示传入机器服务。后续仍需实现机器放置、chunk/machine replication source 和任务/UI 消费者；不能让任意 opaque payload 直接调用机器服务或写入 world/inventory。
