# ScienceAndTheology 游戏网络协议设计

> 当前基线，更新于 2026-07-16。本文描述 `game/network/` 的游戏协议边界；可复用 socket/Steam transport 以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准。

## 1. 目标和所有权

`snt_network` 只负责传输可靠性、TCP/UDP 关联、wire framing 和 fixed-tick phase。它不知道玩家、方块、任务、配方、登录凭据或 AOI。

`snt_game_network` 负责最新版本的游戏 payload、玩家准入和后续玩法同步契约。它保持无头闭包，只依赖 `snt_core`、`snt_network`、`snt_ecs` 和 `snt_voxel_data`，不链接 SDL、Vulkan、Steamworks 或 client runtime。

多人交互采用“主机权威、客户端可信提示”：客户端先完成本地射线、动画、挖掘进度、所选物品与机器前置条件判断，并把目标和提示送给主机；主机是世界、库存和持久化的唯一写入者。项目不以反作弊为目标，因此协议不要求主机逐格重放 raycast、重算挖掘时间或重新验证 cover/ignition/structure/tool tag。主机仍拒绝无认证会话、跨维度、超基础距离、未加载区块、目录外 material、负库存和会破坏墓碑/machine sidecar 的提交，以维护多人共享状态与存档一致性。

| 层 | 格式 | 版本策略 | 责任 |
| --- | --- | --- | --- |
| 引擎 transport | `SNTR` 20-byte wire header | `kCurrentReplicationProtocolVersion` | frame 长度、通道、server tick、TCP/UDP/Steam adapter |
| 游戏 payload | `SNTG` v9、12-byte envelope | `kCurrentGameReplicationProtocolVersion` | 游戏消息类型、身份 provider、登录与玩法同步语义 |

两个版本独立拒绝不匹配值，不提供旧协议探测、转换或兼容 wrapper。

## 2. 当前 envelope

`SNTG` 使用 network byte order：

```text
magic:u32 | game_protocol_version:u16 | kind:u8 | reserved:u8 | payload_length:u32 | payload
```

`reserved` 必须为零，payload length 必须与实际字节完全匹配。除 movement intent 外，当前声明的游戏消息必须走 reliable ordered channel；`ClientMovementInput` 是唯一已显式登记的 UDP/unreliable kind，新增 UDP gameplay 消息仍必须先新增 kind 并在 `validate_game_replication_channel` 中登记。

| 方向 | 已声明消息 | 当前实现 |
| --- | --- | --- |
| client -> server | `LoginRequest` | 已实现 typed codec、客户端连接会话和认证状态转换 |
| client -> server | `Command` | 已实现 typed codec；只有 `QuestClaimReward` 和 `BlockInteraction` 在 dedicated server 的确定性命令顺序中应用，任务进度由服务器事件自动推进 |
| client -> server | `ClientMovementInput` | 已实现 v5 引入的 typed codec；在当前 v7 中仍是唯一 UDP/unreliable 消息，只含玩家意图，不含位置 |
| client -> server | `InterestUpdate`、`SnapshotAcknowledgement` | 仅声明，当前明确拒绝 |
| server -> client | `LoginAccepted` | 已实现，scheduler barrier 后发送，客户端严格校验返回 account id |
| server -> client | `Snapshot`、`Delta` | 已实现严格值 codec、per-peer budget 校验、可注入 provider 调度和客户端有界值队列；production host 已注入 concrete player AOI/source 与 remote-player value application，chunk/machine source 尚未实现 |
| server -> client | `Notice` | 仅声明，当前明确拒绝 |

登录名最大 48 UTF-8 bytes，稳定 player id 最大 128 bytes，凭据最多 1024 bytes，通用命令 payload 最多 64 KiB，`QuestClaimReward.quest_id` 最多 512 bytes。凭据是 opaque bytes，日志不得记录它或其派生值。

任务没有客户端接取命令。`QuestRegistry` 在服务器 main-thread tick 中根据前置任务图把 `Locked` 自动转为 `InProgress`，已提交的玩法事件直接推进进行中的任务；客户端只能读取复制后的任务书快照。未知 command type、错误 payload 和不递增的同 peer sequence 会作为协议错误拒绝；未完成、重复领奖、空间不足或事务失败属于低频游戏规则拒绝，不会让过时客户端终止整个 server。

v4 增加严格的 server-originated 值 codec。`Snapshot` 为 `snapshot_id:u64 | chunk_count:u16 | chunk* | entity_count:u16 | entity*`；每个 chunk 是 `dimension_id | chunk_x:i32 | chunk_y:i32 | chunk_z:i32 | payload_length:u32 | payload`，每个 entity 是 `guid:u64 | payload_length:u32 | payload`。`Delta` 为 `base_snapshot_id:u64 | sequence:u64 | chunk_count:u16 | chunk_block_changes* | entity_count:u16 | entity*`；每个 block change 是 `local_index:u16 | material:u8 | flags:u32`。dimension id、chunk/entity payload、数量、local index 和 envelope 总大小均有上限；重复 chunk、重复 entity、重复 block local index、空或越界 blob、trailing bytes 全部拒绝。chunk/entity blob 的具体 schema 仍由 game provider 拥有，不在 transport 或 generic codec 中解释。

v5 增加 `ClientMovementInput` 的固定 16-byte payload：`client_sequence:u64 | forward_axis:i8 | strafe_axis:i8 | flags:u8 | reserved:u8=0 | yaw_centidegrees:i16 | pitch_centidegrees:i16`。轴只能为 `-1/0/1`，flags 仅允许 sprint/jump，yaw 为 `[-18000,18000)`、pitch 为 `[-8900,8900]`，sequence 必须非零并与同 peer 的可靠 command 共享严格递增序列。客户端永远不提交 position、velocity、碰撞结果、grounded 状态或背包上下文；服务端在 main thread 合并每 peer 最新 intent 后积分、碰撞并复制权威 block-grid 位置到 replication source。

当前 `SNTG` v9 定义 `command_type=1` 的 `BlockInteraction`。它仍包在外层 `client_sequence:u64 | command_type:u16 | payload_length:u32 | payload` 中，内层 payload 为 `action:u8 | client_hints:u8 | expected_material:u16 | dimension_id | block_x:i32 | block_y:i32 | block_z:i32 | selected_item_id`，其中两个标识符均带 `u16` 长度前缀、长度上限且不得含 NUL。`action` 只允许 mine、place、use、activate machine 和 collect machine output；`expected_material=256` 表示客户端不提供乐观的旧材质保护。cover、ignition、structure 与 tool 提示仅允许随 machine activation 发送。客户端可先完成本地射线、挖掘进度和机器前置条件判断；主机不重放这些反作弊式检查，而以自身内容目录解析最终材质、以主机库存完成原子扣除/授予，并保护已有墓碑和 machine sidecar。当前服务只对已存在的机器锚点执行 activation/collect；机器放置和 chunk delta source 仍是后续工作。

当前 `SNTG` v9 定义 `command_type=2` 的 `QuestClaimReward`。其 payload 是严格的 `quest_id_length:u16 | quest_id`。客户端不携带 player id、物品、数量或背包数据；服务端从认证 peer 得到 account id，并在稳定命令顺序中调用 `QuestRegistry::claim_reward()`。item reward 仍由主机 inventory transaction 整笔提交，未完成、重复领取、无 active peer、空间不足或事务失败只属于低频游戏规则拒绝，且不得设置 `reward_claimed`。该命令不替代任务书快照同步或 UI。

当前 concrete player entity blob 为 game-owned v1：`kind:u8=1 | version:u8=1 | operation:u8 | reserved:u8`；`upsert` 再携带认证 identity provider、account id、display name、dimension、三个 block-grid `i32` 位置和六个装备槽的 item id，`remove` 只保留 header。它只表达联机展示所需的值，不传背包、实例数据、器官 payload、tool tag、peer id 或 ECS handle。未知 entity kind 仍由 future game source 拥有，不由 generic protocol 猜测。

## 3. 服务端时序

```text
transport poll
  -> GameServerReplicationHandler inbound validation/authentication/player lifecycle/command intake
  -> GameServerCommandSink stable command ordering and latest movement-intent forwarding
  -> QuestRegistry automatic prerequisite reconciliation and gameplay-event progress
  -> game fixed tick
  -> SystemScheduler barrier
  -> GameServerReplicationHandler outbound login/snapshot/delta emission
```

认证通过后，`LoginAccepted` 被排入同一 tick 的 outbound 队列，因此不会在 scheduler barrier 前发送“已进入世界”的假状态。命令入口只在 simulation main thread 调用；`GameServerCommandSink` 在 gameplay fixed tick 前按 account id、peer、client sequence 稳定排序 `QuestClaimReward` 和 `BlockInteraction`。领取命令调用 `QuestRegistry::claim_reward()`；方块命令经 typed codec 转给 `GameServerPlayerInteractionService`，由该服务以客户端可信的目标、所选物品和 machine 提示驱动主机 world/inventory 事务。`QuestRegistry` 随后的固定 tick 根据前置图自动解锁任务，已提交交互与机器事件直接推进进行中任务。交互服务只做认证 peer、当前 dimension、基础 reach、已加载 chunk、内容目录、库存非负和 sidecar 完整性检查，不重放 raycast、挖掘时间或机器前置条件。movement intent 以同一 shared sequence 验证、每 peer 只保留最新值，再交给 `GameServerPlayerMovement`。后者在游戏 fixed tick 前执行速度、跳跃、重力、体素碰撞和 stale-input timeout，客户端 position 不会进入任何写入路径。断线或同账号顶号会取消尚未应用的旧 peer 命令、movement intent 和 sequence 状态。`GameServerPlayerLifecycle` 在首次认证时加载 `SNTQ` 任务进度和 `SNTP` 玩家值，顶号时只转移同一 account 的 live entity，正常断线和关服时保存；床设置、死亡重生和墓碑领取只标记玩家状态，并在 barrier 后的配置化 checkpoint 写入。所有这些文件操作都不在 fixed tick 中执行。

当 `GameServerReplicationHandler` 同时注入 `IGameReplicationInterestProvider` 和 `IGameReplicationSnapshotSource` 时，它在 outbound barrier 为每个认证 peer 计算 interest。source 的首个非空 `build_initial_snapshot()` 结果作为 snapshot；之后调用 `build_deltas()`。handler 会重新解析并重新编码 source 返回的每条消息，只接受当前 phase 的 `Snapshot` 或 `Delta`，并按每 peer 的 reliable bytes、snapshot chunks、entity snapshots、typed values、block deltas 预算验证整个 batch。任一条超预算或非法时整个 batch 不入 outbound 队列，不会发送部分同步帧；仍有未发送 snapshot/delta 时不会继续生产新数据。当前 `ScienceAndTheologyServerSession` 将 `GameServerPlayerReplication` 同时注入为 interest provider 和 snapshot source：它从 `GameServerPlayerState` 读取玩家值，并聚合 `IGameReplicationValueSource` 的认证账号值；强制快照包含观察者自身，并按同 dimension、可配置的水平圆形 AOI、垂直范围和稳定距离/Guid 顺序选择其他玩家；`server_replication` 同时控制可见人数和 bytes/entity/value/chunk/block budget。source 在首个 snapshot 保存 per-observer baseline，之后发 player upsert/remove 与 current value 的 upsert/remove；normal disconnect 与同账号顶号都会清除旧 peer baseline。`GameServerQuestBookReplication` 是当前的 account value source，只从 `QuestRegistry` 读取进度。当前 source 不产生 chunk 或 machine payload，也不接收 client `InterestUpdate`。

当前 `ScienceAndTheologyServerSession` 安装 `GameAccountPeerAuthenticator`：

- `local_name` 请求不带凭据，服务端以校验后的 UTF-8 名称构造 `local-name:<name>`。
- 同一个 account id 已在线时，后登录 peer 接管该会话。handler 撤销旧 peer 尚未发送的登录成功消息，并在同一 tick 的 outbound 阶段经 `ReplicationService` 请求断开旧 peer；这就是本地同名的“顶号”语义。
- `steam` 请求只携带 opaque ticket，协议内没有可相信的 client SteamID 字段。服务端必须让 `ISteamSessionTicketVerifier` 返回 SteamID64，再构造 `steam:<SteamID64>`。当前可执行 server 尚未注入 Steamworks verifier，Steam 请求会明确拒绝，不会退化为本地账号。

相同显示名不会合并账号：Steam `Alex` 与本地输入的 `Alex` 分别是 `steam:<SteamID64>` 和 `local-name:Alex`。同一 SteamID64 的多次登录也会自然落入同一 account id；不同 Steam 用户不会因显示名相同而发生冲突。

## 4. 客户端本地身份

`snt_game_identity` 不依赖 SDL、网络或 Steamworks。客户端启动按以下优先级解析身份：

1. 已注入的 `ISteamPlayerIdentityProvider` 返回经过 Steam 本地认证的 SteamID64 时，使用 `steam:<SteamID64>`。
2. 否则读取 `user_root/profile/local_player_identity.bin` 的 `SNTI` v1 本地资料。
3. 本地资料不存在时打开 SDL 名字输入窗口，校验名字并持久化；资料损坏时直接报错，不静默改名或覆盖。

本地资料只保存本地名字，不复制 Steam 身份。写入使用 `.tmp -> .bak -> primary`，且只在 primary 缺失时读取 backup；因此损坏的 primary 不会被旧 backup 静默遮掩。当前客户端尚未链接 Steamworks SDK，所以尚无 provider 被注入，非 Steam 路径是实际可用启动流程。图形客户端在 `client_network.enabled` 时以 `GameClientReplicationSession` 组合 TCP+UDP transport：它在 fixed tick 前轮询 inbound、在 scheduler barrier 后发出 outbound，连接后自动提交 local-name 登录请求。认证后它严格解析 `Snapshot`/`Delta` 并通过 `drain_replication_updates()` 交出有序值更新；Delta 必须匹配已接收 snapshot id，sequence 必须严格增长。会话以 UDP 发送已认证后的 movement intent，server 端每 peer 只保留最新 intent。队列上限为 64 个 replication update 且总 wire bytes 不超过一个最大 `SNTG` envelope；越界或错序会断开连接。该会话不创建远端世界、不持有 ECS World，也不伪造玩家状态；图形宿主将已 drain 的 player v1 value 应用到 `GameRemotePlayerWorld`，并把本账号的权威 block-grid position 写到相机，在本地断线时清空 value world。当前仍无客户端 prediction、reconciliation smoothing 或 remote avatar render entity。

## 5. 预留接口

以下接口已声明在 `game/network/game_replication_services.h`。认证器、玩家生命周期和命令 sink 已有生产使用；AOI/source 的 generic handler integration、budget enforcement 和 concrete player provider 已实现，chunk/machine provider 仍不存在：

- `IGamePeerAuthenticator`：把凭据验证和持久玩家身份映射留在游戏层。
- `IGamePlayerSessionLifecycle`：把认证后的玩家状态 load、同账号替换、断线和关服 save 留在 dedicated-server composition。当前 `GameServerPlayerLifecycle` 组合 `QuestRegistry`、`GameServerPlayerState`、`SNTQ`/`SNTP` persistence 与 dirty checkpoint，由 `persistence.universe_save_dir` 解析到 `user_root`；它不持有 transport 或 ECS World。
- `IGameReplicationCommandSink`：认证后按 shared client sequence 接收可靠 command 和 unreliable movement intent，并在 simulation main thread 排队确定性 mutation。`GameServerCommandSink` 当前支持 `QuestClaimReward`、`BlockInteraction` 和最新 intent 合并；领取只借用 `QuestRegistry`，方块命令只依赖窄的 `IGameServerPlayerInteractionService`，不持有 transport 或 ECS World，并在 disconnect/takeover 时取消旧 peer 队列。
- `IGameReplicationInterestProvider`：以当前 `ChunkKey`、稳定 `EntityGuid` 和机器实体集合计算 AOI。`GameServerPlayerReplication` 当前实现 player-only 同 dimension AOI；ChunkKey/机器集合保留给后续 source。
- `IGameReplicationSnapshotSource`：按每 peer 的 chunk/entity/block budget 产出 server-originated snapshot 或 delta，并可在 disconnect/takeover 时释放 observer baseline。`GameServerPlayerReplication` 当前只产出 player v1 entity blob。
- `IGameReplicationValueSource`：为认证观察者生成当前的 server-owned presentation value；aggregate snapshot source 负责把它与 per-peer baseline 比较、预算、upsert/remove 和 disconnect 清理。当前 `GameServerQuestBookReplication` 只提供认证账号的 `QuestBookSnapshot`。

`GameClientReplicationSession` 是独立的无头 client composition：它可接收任意 `IReplicationTransport`，以显式 `server_peer` 支持未来 Steam P2P peer id；`connect_tcp_udp()` 只是当前图形宿主的便利工厂。会话只向 server peer 接受 server-originated `SNTG` payload，LoginAccepted 必须与本地已认证 provider/account id 一致。它提供通用严格递增 command queue、typed `enqueue_quest_claim_reward()`、`enqueue_block_interaction()`、unreliable `enqueue_player_movement_input()` 和 `drain_replication_updates()` 值边界；当前图形宿主已把 player value 交给 `GameRemotePlayerWorld`、认证账号的 task-book value 交给 `GameClientQuestBookState`，MUI 任务书以只读快照展示任务图并发送领奖请求；remote avatar rendering 和其它 remote-world source 仍未接入。

身份模块还预留了：

- `ISteamPlayerIdentityProvider`：客户端启动时读取已认证的 SteamID64 和显示名。
- `ISteamSessionTicketVerifier`：服务端验证 opaque Steam ticket，并返回唯一的 SteamID64 与服务端认可的显示名。
- `ILocalPlayerNamePrompt`：不依赖 SDL 的本地名字输入边界；SDL 窗口只是当前 client host 的一个实现。

`IReplicationFrameSink::disconnect()` 是 game handler 在 outbound 阶段请求主动断开的唯一通道。它仍由 `ReplicationService` 持有并调用实际 transport，handler 自己清理 session state，避免 game 层直接持有 TCP/UDP 或 Steam transport。

`GameReplicationInterest` 目前使用现有 `ChunkKey`。统一宇宙迁移到 Sector 坐标时，直接更新该最新接口和协议版本，不保留 ChunkKey 到 Sector 的兼容桥。

## 6. 认证取舍

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 将共享口令硬编码到 server config | 最快，LAN 调试简单 | 秘密管理和明文配置会成为长期 API，难以迁移 |
| 让客户端直接提交 SteamID64 | 编码最短 | 可伪造身份，不能用于服务端权威世界 |
| 仅 Steam 身份 | 唯一性强 | dedicated server、局域网调试和非 Steam 启动被锁定 |
| 可注入 Steam ticket verifier + 本地名字账号（已选择） | Steam 保持平台唯一标识；非 Steam 可启动并持久化；两套名字空间互不冲突 | Steamworks client/server adapter 仍需实现；本地同名故意具有接管风险 |

选择最后一种与 `snt_network` 的 Steam SDK-neutral adapter 一致。`GameAccountPeerAuthenticator` 已为 local-name 生产路径实现完整映射和会话接管；Steam 的身份和票据验证接口已经冻结，但不能在没有 Steamworks backend 时伪造为已可登录。
