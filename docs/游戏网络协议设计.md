# ScienceAndTheology 游戏网络协议设计

> 当前基线，更新于 2026-07-15。本文描述 `game/network/` 的游戏协议边界；可复用 socket/Steam transport 以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准。

## 1. 目标和所有权

`snt_network` 只负责传输可靠性、TCP/UDP 关联、wire framing 和 fixed-tick phase。它不知道玩家、方块、任务、配方、登录凭据或 AOI。

`snt_game_network` 负责最新版本的游戏 payload、玩家准入和后续玩法同步契约。它保持无头闭包，只依赖 `snt_core`、`snt_network`、`snt_ecs` 和 `snt_voxel_data`，不链接 SDL、Vulkan、Steamworks 或 client runtime。

| 层 | 格式 | 版本策略 | 责任 |
| --- | --- | --- | --- |
| 引擎 transport | `SNTR` 20-byte wire header | `kCurrentReplicationProtocolVersion` | frame 长度、通道、server tick、TCP/UDP/Steam adapter |
| 游戏 payload | `SNTG` v4、12-byte envelope | `kCurrentGameReplicationProtocolVersion` | 游戏消息类型、身份 provider、登录与玩法同步语义 |

两个版本独立拒绝不匹配值，不提供旧协议探测、转换或兼容 wrapper。

## 2. 当前 envelope

`SNTG` 使用 network byte order：

```text
magic:u32 | game_protocol_version:u16 | kind:u8 | reserved:u8 | payload_length:u32 | payload
```

`reserved` 必须为零，payload length 必须与实际字节完全匹配。所有当前声明的游戏消息必须走 reliable ordered channel；要添加 UDP gameplay 消息，必须先新增 kind 并在 `validate_game_replication_channel` 中显式登记。

| 方向 | 已声明消息 | 当前实现 |
| --- | --- | --- |
| client -> server | `LoginRequest` | 已实现 typed codec、客户端连接会话和认证状态转换 |
| client -> server | `Command` | 已实现 typed codec；`QuestAccept` 在 dedicated server 转为权威任务状态 |
| client -> server | `InterestUpdate`、`SnapshotAcknowledgement` | 仅声明，当前明确拒绝 |
| server -> client | `LoginAccepted` | 已实现，scheduler barrier 后发送，客户端严格校验返回 account id |
| server -> client | `Snapshot`、`Delta` | 已实现严格值 codec、per-peer budget 校验、可注入 provider 调度和客户端有界值队列；production host 尚未注入具体 AOI/source |
| server -> client | `Notice` | 仅声明，当前明确拒绝 |

登录名最大 48 UTF-8 bytes，稳定 player id 最大 128 bytes，凭据最多 1024 bytes，通用命令 payload 最多 64 KiB，`QuestAccept.quest_id` 最多 512 bytes。凭据是 opaque bytes，日志不得记录它或其派生值。

v3 定义第一个实际权威命令：`QuestAccept`。它保留外层 `client_sequence:u64 | command_type:u16 | payload_length:u32 | payload` 的 `Command` 编码，`command_type=1` 时 payload 是严格的 `quest_id_length:u16 | quest_id`。未知 command type、错误 payload 和不递增的同 peer sequence 会作为协议错误拒绝；任务不存在或当前状态不可接取属于游戏规则拒绝，只做低频聚合 Warn，不会让恶意或过时客户端终止整个 server。

v4 增加严格的 server-originated 值 codec。`Snapshot` 为 `snapshot_id:u64 | chunk_count:u16 | chunk* | entity_count:u16 | entity*`；每个 chunk 是 `dimension_id | chunk_x:i32 | chunk_y:i32 | chunk_z:i32 | payload_length:u32 | payload`，每个 entity 是 `guid:u64 | payload_length:u32 | payload`。`Delta` 为 `base_snapshot_id:u64 | sequence:u64 | chunk_count:u16 | chunk_block_changes* | entity_count:u16 | entity*`；每个 block change 是 `local_index:u16 | material:u8 | flags:u32`。dimension id、chunk/entity payload、数量、local index 和 envelope 总大小均有上限；重复 chunk、重复 entity、重复 block local index、空或越界 blob、trailing bytes 全部拒绝。chunk/entity blob 的具体 schema 仍由 game provider 拥有，不在 transport 或 generic codec 中解释。

## 3. 服务端时序

```text
transport poll
  -> GameServerReplicationHandler inbound validation/authentication/player lifecycle/command intake
  -> GameServerCommandSink stable command ordering and QuestRegistry mutation
  -> game fixed tick
  -> SystemScheduler barrier
  -> GameServerReplicationHandler outbound login/snapshot/delta emission
```

认证通过后，`LoginAccepted` 被排入同一 tick 的 outbound 队列，因此不会在 scheduler barrier 前发送“已进入世界”的假状态。命令入口只在 simulation main thread 调用；当前 `GameServerCommandSink` 在 gameplay fixed tick 前按 account id、peer、client sequence 稳定排序 `QuestAccept`，并调用 `QuestRegistry::accept()`。断线或同账号顶号会取消尚未应用的旧 peer 命令并清除其 sequence 状态。`GameServerPlayerLifecycle` 在首次认证时加载任务进度，顶号时只转移同一 account 的内存状态，正常断线和关服时保存；这些文件操作不在 fixed tick 中执行。

当 `GameServerReplicationHandler` 同时注入 `IGameReplicationInterestProvider` 和 `IGameReplicationSnapshotSource` 时，它在 outbound barrier 为每个认证 peer 计算 interest。source 的首个非空 `build_initial_snapshot()` 结果作为 snapshot；之后调用 `build_deltas()`。handler 会重新解析并重新编码 source 返回的每条消息，只接受当前 phase 的 `Snapshot` 或 `Delta`，并按每 peer 的 reliable bytes、snapshot chunks、entity snapshots、block deltas 预算验证整个 batch。任一条超预算或非法时整个 batch 不入 outbound 队列，不会发送部分同步帧；仍有未发送 snapshot/delta 时不会继续生产新数据。当前 `ScienceAndTheologyServerSession` 没有注入 concrete provider，所以可执行 server 仍只发送 `LoginAccepted`，不产生实际世界同步。

当前 `ScienceAndTheologyServerSession` 安装 `GameAccountPeerAuthenticator`：

- `local_name` 请求不带凭据，服务端以校验后的 UTF-8 名称构造 `local-name:<name>`。
- 同一个 account id 已在线时，后登录 peer 接管该会话。handler 撤销旧 peer 尚未发送的登录成功消息，并在同一 tick 的 outbound 阶段经 `ReplicationService` 请求断开旧 peer；这就是本地同名的“顶号”语义。
- `steam` 请求只携带 opaque ticket，协议内没有可相信的 client SteamID 字段。服务端必须让 `ISteamSessionTicketVerifier` 返回 SteamID64，再构造 `steam:<SteamID64>`。当前可执行 server 尚未注入 Steamworks verifier，Steam 请求会明确拒绝，不会退化为本地账号。

相同显示名不会合并账号：Steam `Alex` 与本地输入的 `Alex` 分别是 `steam:<SteamID64>` 和 `local-name:Alex`。同一 SteamID64 的多次登录也会自然落入同一 account id；不同 Steam 用户不会因显示名相同而发生冲突。

## 4. 客户端本地身份

`snt_game_identity` 不依赖 SDL、网络或 Steamworks。客户端启动按以下优先级解析身份：

1. 已注入的 `ISteamPlayerIdentityProvider` 返回经过 Steam 本地认证的 SteamID64 时，使用 `steam:<SteamID64>`。
2. 否则读取 `user_root/profile/local_player_identity.bin` 的 `SNTI` v1 本地资料。
3. 本地资料不存在时打开 SDL 名字输入窗口，校验名字并持久化；资料损坏时直接报错，不静默改名或覆盖。

本地资料只保存本地名字，不复制 Steam 身份。写入使用 `.tmp -> .bak -> primary`，且只在 primary 缺失时读取 backup；因此损坏的 primary 不会被旧 backup 静默遮掩。当前客户端尚未链接 Steamworks SDK，所以尚无 provider 被注入，非 Steam 路径是实际可用启动流程。图形客户端在 `client_network.enabled` 时以 `GameClientReplicationSession` 组合 TCP+UDP transport：它在 fixed tick 前轮询 inbound、在 scheduler barrier 后发出 outbound，连接后自动提交 local-name 登录请求。认证后它严格解析 `Snapshot`/`Delta` 并通过 `drain_replication_updates()` 交出有序值更新；Delta 必须匹配已接收 snapshot id，sequence 必须严格增长。队列上限为 64 个 update 且总 wire bytes 不超过一个最大 `SNTG` envelope；越界或错序会断开连接。该会话不创建远端世界、不持有 ECS World，也不伪造玩家状态。

## 5. 预留接口

以下接口已声明在 `game/network/game_replication_services.h`。认证器、玩家生命周期和命令 sink 已有生产使用；AOI/source 的 generic handler integration 和 budget enforcement 已实现，但 concrete gameplay provider 仍不存在：

- `IGamePeerAuthenticator`：把凭据验证和持久玩家身份映射留在游戏层。
- `IGamePlayerSessionLifecycle`：把认证后的玩家状态 load、同账号替换、断线和关服 save 留在 dedicated-server composition。当前 `GameServerPlayerLifecycle` 只持有 `QuestRegistry` 与 `GameSaveQuestProgressPersistence`，由 `persistence.universe_save_dir` 解析到 `user_root`；它不持有 transport 或 ECS World。
- `IGameReplicationCommandSink`：认证后按 client sequence 接收命令，并在 simulation main thread 排队确定性 mutation。`GameServerCommandSink` 是当前实现，只支持 `QuestAccept`；它不持有 transport 或 ECS World，并在 disconnect/takeover 时取消旧 peer 队列。
- `IGameReplicationInterestProvider`：以当前 `ChunkKey`、稳定 `EntityGuid` 和机器实体集合计算 AOI。
- `IGameReplicationSnapshotSource`：按每 peer 的 chunk/entity/block budget 产出 server-originated snapshot 或 delta。

`GameClientReplicationSession` 是独立的无头 client composition：它可接收任意 `IReplicationTransport`，以显式 `server_peer` 支持未来 Steam P2P peer id；`connect_tcp_udp()` 只是当前图形宿主的便利工厂。会话只向 server peer 接受 server-originated `SNTG` payload，LoginAccepted 必须与本地已认证 provider/account id 一致。它提供通用严格递增 command queue、typed `enqueue_quest_accept()` 和 `drain_replication_updates()` 值边界；当前图形 UI 还没有任务面板或 remote-world consumer，因此没有生产输入/应用调用。

身份模块还预留了：

- `ISteamPlayerIdentityProvider`：客户端启动时读取已认证的 SteamID64 和显示名。
- `ISteamSessionTicketVerifier`：服务端验证 opaque Steam ticket，并返回唯一的 SteamID64 与服务端认可的显示名。
- `ILocalPlayerNamePrompt`：不依赖 SDL 的本地名字输入边界；SDL 窗口只是当前 client host 的一个实现。

`IReplicationFrameSink::disconnect()` 是 game handler 在 outbound 阶段请求主动断开的唯一通道。它仍由 `ReplicationService` 持有并调用实际 transport，handler 自己清理 session state，避免 game 层直接持有 TCP/UDP 或 Steam transport。

`GameReplicationInterest` 目前使用现有 `ChunkKey`。统一宇宙迁移到 Sector 坐标时，直接更新该最新接口和协议版本，不保留 ChunkKey 到 Sector 的兼容桥。

## 6. 认证取舍

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 将共享口令硬编码到 server config | 最快，LAN 调试简单 | 秘密管理和明文配置会成为长期 API，难以迁移 |
| 让客户端直接提交 SteamID64 | 编码最短 | 可伪造身份，不能用于服务端权威世界 |
| 仅 Steam 身份 | 唯一性强 | dedicated server、局域网调试和非 Steam 启动被锁定 |
| 可注入 Steam ticket verifier + 本地名字账号（已选择） | Steam 保持平台唯一标识；非 Steam 可启动并持久化；两套名字空间互不冲突 | Steamworks client/server adapter 仍需实现；本地同名故意具有接管风险 |

选择最后一种与 `snt_network` 的 Steam SDK-neutral adapter 一致。`GameAccountPeerAuthenticator` 已为 local-name 生产路径实现完整映射和会话接管；Steam 的身份和票据验证接口已经冻结，但不能在没有 Steamworks backend 时伪造为已可登录。

## 7. 下一步

1. 在独立 Steamworks package 实现 `ISteamPlayerIdentityProvider`、Steam session ticket 创建和 `ISteamSessionTicketVerifier`，由 client/server package 显式注入，不修改引擎 transport。
2. 已完成 client host 的连接会话与 local-name 登录请求创建；Steam ticket 仍待注入，客户端不会伪造该凭据。
3. 已完成 v3 `QuestAccept` command id、`GameServerCommandSink` 到确定性 `QuestRegistry` mutation 的映射，以及 dedicated-server 任务持久化 lifecycle。后续 command 必须先拥有服务端玩家/实体状态；移动、方块编辑和奖励领取不得借用当前任务接取接口伪实现。
4. 为 server-owned player/entity state 实现 concrete AOI 和 snapshot/delta provider，定义 blob schema 与 client remote-world consumer；`InterestUpdate` 和 snapshot acknowledgement 继续保持未实现，不能由客户端声明 AOI 或确认权威状态。

验证基线：身份测试覆盖 Steam/local 同名隔离、本地资料持久化、损坏拒绝、backup 恢复和首启解析；`GameReplicationProtocolTest`、`GameReplicationHandlerTest` 和 `GameAccountPeerAuthenticatorTest` 覆盖 v4 envelope、snapshot/delta roundtrip、重复值拒绝、可靠通道、provider 预算的原子拒绝、Steam verifier 边界、本地同名接管、登录确认、生命周期回调与未认证命令拒绝；`GameServerCommandSinkTest` 覆盖 QuestAccept、同 peer sequence 去重和断线取消；`GameServerPlayerLifecycleTest` 覆盖真实 `SNTQ` 顶号转移、断线保存和重启加载；`GameClientReplicationSessionTest` 覆盖 LoginAccepted 身份校验、snapshot/delta 值 drain、错 snapshot base 拒绝，以及真实 localhost TCP+UDP 的 QuestAccept 端到端应用；`snt_game_server --network --tcp-port 0 --udp-port 0 --ticks 3` 覆盖无头网络宿主启动与关闭。
