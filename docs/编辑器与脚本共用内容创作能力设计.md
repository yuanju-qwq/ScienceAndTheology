# 编辑器与脚本共用内容创作能力设计

> 状态：目标架构。本文定义编辑器、Authoring Script 和无头内容工具的共同内容操作边界；不描述具体 MUI 页面实现。
>
> 更新：2026-07-17。

## 1. 决策

编辑器不是游戏客户端中的调试菜单，AngelScript 也不是只能注册少量静态定义的旁路。两者都是同一套内容创作能力的前端：

```text
MUI 编辑器 ────────┐
Authoring Script ──┼──> ContentCommandBus / ContentTransaction
无头内容工具 ──────┘                    │
                                      DocumentStore
                                  校验 / 依赖图 / Undo
                                      │
                                  ContentCompiler
                                      │
                              CompiledContentBundle
                                      │
                         客户端、服务端、预览 Runtime
```

所有创建、修改、删除、连线、校验、预览和导出操作必须通过同一个命令模型完成。UI 事件和脚本调用不得分别修改不同的数据结构，也不得存在“编辑器能做、脚本不能做”或同名操作语义不同的情况。

本项目尚未发布正式版本，因此直接采用新边界并删除旧接口；不保留旧的内容注册兼容层。

## 2. 目标与非目标

### 2.1 目标

- 可视化编辑任务书、配方、模型配置、动画配置和脚本关联。
- AngelScript 可使用与编辑器等价的内容操作，支持批量生成、重构和自动化检查。
- 编辑器、脚本和 CI 使用完全相同的校验器、依赖分析器和内容编译器。
- 内容源文件可在 Git 中审查和合并；运行时只读取已编译的内容包。
- 保持当前 `snt_simulation_runtime` 的无 SDL/Vulkan 闭包，不让编辑器依赖反向污染服务端。
- 所有失败诊断包含内容 GUID、源文件、字段路径和操作来源。

### 2.2 非目标

- 不自研 Blender、UV 编辑、雕刻、绑定或 FBX 工作流。美术源资产首发只接受 glTF 2.0。
- 不让运行中的多人游戏脚本直接修改工作区源文件或提交内容包。
- 不以生成 AngelScript 注册调用作为任务书、配方或动画配置的唯一真源。
- 不再引入 Dear ImGui；编辑器 UI 继续使用 `snt::ui` retained MUI。

## 3. 脚本运行域

“脚本与编辑器具有共同操作能力”指 Authoring Script 与编辑器共享内容创作 API，不等于所有游戏内脚本都可任意写入资产。

| 运行域 | 运行位置 | 内容操作权限 | 主要用途 |
| --- | --- | --- | --- |
| `AuthoringScript` | `snt_editor`、`snt_game_contentc`、受控预览 | 完整 `ContentAuthoringApi` | 批量创建、重构、校验、导入配置、生成任务和配方 |
| `GameplayScript` | 客户端、服务端、模拟预览 | 只读已编译内容，写玩法状态 | 事件回调、特殊任务条件、机器行为、UI 行为 |

`GameplayScript` 不能获得 `ContentDocumentStore`、文件写入器、内容提交器或编辑器选择状态。这样避免服务端、客户端和存档因运行时脚本修改内容定义而出现不一致。

## 4. 模块与所有权

### 4.1 新增目标

| 目标 | 所有者 | 职责 | 禁止依赖 |
| --- | --- | --- | --- |
| `snt_authoring_core` | 引擎 | 通用命令、事务、文档版本、撤销、诊断、变更订阅接口 | MUI、Vulkan、游戏任务/配方类型 |
| `snt_game_authoring` | 游戏 | 任务、配方、模型、动画的 schema、领域命令、校验和编译 | 客户端窗口、运行时存档 |
| `snt_game_contentc` | 游戏 | 无头加载、校验、编译、CI 入口 | SDL、Vulkan、MUI |
| `snt_editor` | 游戏 | 编辑器可执行程序、`EditorSession`、MUI 页面、视口和预览编排 | 对游戏运行时私有状态的直接写入 |
| `snt_game_content_runtime` | 游戏 | 加载 `CompiledContentBundle` 并建立运行时只读定义 | 编辑器 UI、源文档写入 |

`snt_editor` 是游戏仓库中的宿主可执行程序，使用 `snt_client_runtime` 获得窗口、输入、Vulkan 视口和 MUI。它不是 `snt_engine` 的可执行程序，也不得成为 `snt_game_client` 的启动模式。

### 4.2 目录布局

```text
snt_engine/authoring/                 # snt_authoring_core
game/authoring/                       # snt_game_authoring
game/tools/contentc/                  # snt_game_contentc
game/editor/                          # snt_editor
game/content-src/                     # 可编辑、可审查源内容
game/content-cooked/                  # 生成物，运行时内容包
```

`content-cooked/` 可以由构建生成并忽略版本控制；`content-src/` 与外部 glTF/纹理源文件必须进入版本控制。编辑器布局、导入缓存、预览缓存和崩溃恢复文件只写入 `<user>/editor/`。

## 5. 共同内容模型

### 5.1 稳定身份与引用

每个内容文档都有不可变的 `AssetGuid`，采用引擎现有 UUID 能力保存。显示名称、文件名和可选 slug 都不是跨内容引用的主键。

```cpp
struct ContentAssetId {
    snt::core::Uuid guid;
    ContentKind kind;
};

template <typename T>
struct AssetRef {
    ContentAssetId id;
};
```

任务前置条件、奖励目标、配方物品和机器、模型引用、动画状态目标及脚本引用都使用强类型 `AssetRef<T>`。编译阶段拒绝类型不匹配、缺失引用和循环依赖；运行时不再以路径或显示名称猜测目标。

### 5.2 源内容与运行时内容

源内容使用版本化 JSON 文档，便于审查、合并和定位字段。编译器输出不可变的二进制 `CompiledContentBundle`，供客户端、服务端和预览 Runtime 读取。

```text
game/content-src/
  quests/<guid>.sntquest.json
  recipes/<guid>.sntrecipe.json
  models/<guid>.sntmodel.json
  animations/<guid>.sntanimgraph.json
  scripts/<guid>.as

game/content-cooked/
  content.bundle
  content.manifest
  imported-assets/
```

源文档中的编辑器布局元数据，例如任务图节点位置、折叠状态和动画图画布坐标，可与运行时定义同文件保存，但编译器不会把它写入 `CompiledContentBundle`。

## 6. 共同操作模型

### 6.1 命令与事务

编辑器与脚本均通过 `ContentAuthoringApi` 创建命令。任何可改变内容的调用都必须位于事务中；事务要么整体提交，要么整体回滚。

```cpp
class ContentAuthoringApi {
public:
    [[nodiscard]] ContentTransaction begin_transaction(
        ContentOperationContext context);

    [[nodiscard]] Expected<ContentAssetId> create_asset(
        ContentTransaction&, ContentKind, std::string display_name);
    [[nodiscard]] Expected<void> delete_asset(
        ContentTransaction&, ContentAssetId);
    [[nodiscard]] Expected<void> apply(
        ContentTransaction&, const ContentCommand&);
    [[nodiscard]] ValidationReport validate(const ContentSnapshot&) const;
    [[nodiscard]] Expected<ContentCommit> commit(ContentTransaction&);
    void rollback(ContentTransaction&) noexcept;
};
```

`ContentCommand` 是封闭的、可序列化的值类型，不携带 MUI View、AngelScript VM 指针、ECS 指针或文件句柄。第一批命令包含：

- `CreateAsset`、`DeleteAsset`、`RenameAsset`、`SetProperty`。
- `InsertListItem`、`RemoveListItem`、`MoveListItem`。
- `ConnectGraphEdge`、`DisconnectGraphEdge`、`MoveGraphNode`。
- `SetModelImportSettings`、`AssignMaterial`、`SetAnimationTransition`。
- `ReplaceScriptSource`、`SetScriptReference`。

领域操作可在底层由上述结构命令表示，但公开 API 优先提供语义化名称，例如 `add_recipe_input`、`add_quest_prerequisite` 和 `add_animation_state`，避免脚本和 UI 依赖脆弱的任意字符串字段路径。

### 6.2 操作上下文

每个事务都带有来源，供撤销、审计、日志和权限检查使用：

```cpp
enum class ContentOperationOrigin {
    EditorUi,
    AuthoringScript,
    CommandLine,
    Importer,
};

struct ContentOperationContext {
    ContentOperationOrigin origin;
    std::string author_id;
    std::string label;
    bool allow_source_write = true;
};
```

一次编辑器拖放是一个事务；一次 Authoring Script 顶层执行也是一个事务。脚本抛错、校验失败或编译预检失败时，整批命令回滚。脚本提交成功后，编辑器将其显示为一个可撤销的历史项，而不是一组不可见的文件改动。

### 6.3 Undo、Redo 与变更订阅

`ContentCommandBus` 保存已提交事务的反向命令或提交前快照。Undo/Redo 也通过命令总线执行，因此会向编辑器、脚本宿主、依赖图和预览 Runtime 发出同一条 `ContentChangeSet`。

变更事件只携带内容 GUID、文档 revision、影响字段、依赖失效集合和诊断摘要。订阅者不得保留可变文档指针；后台导入和预览任务只消费不可变 snapshot。

## 7. 领域扩展点

每种内容领域必须同时提供数据、操作、校验、编译、脚本绑定和编辑器工具，而不是让其中一端自行补丁。

```cpp
class IContentDomain {
public:
    virtual ContentKind kind() const noexcept = 0;
    virtual Expected<void> apply(ContentDocument&, const ContentCommand&) = 0;
    virtual ValidationReport validate(const ContentSnapshot&) const = 0;
    virtual Expected<void> compile(const ContentSnapshot&, CompiledContentWriter&) const = 0;
};

class IScriptDomainBinding {
public:
    virtual Expected<void> register_authoring_api(asIScriptEngine&) = 0;
};

class IEditorDomainTool {
public:
    virtual void build_workspace(EditorContext&) = 0;
};
```

`IContentDomain` 不依赖 MUI 或 AngelScript。脚本绑定和编辑器工具是对同一领域命令的薄适配层，不能重新实现校验和状态修改。

### 7.1 任务书

任务书编辑器显示节点和边；脚本可创建任务、添加目标、添加奖励和连接前置条件。编译器检查：

- GUID 唯一、任务图无环、无自引用。
- 目标和奖励引用存在且类型匹配。
- 解锁任务可达，隐藏任务具有合理的显示路径。
- 客户端展示字段与服务端权威进度字段分离。

现有 `QuestDefinition`、目标和奖励值类型可迁移为编译器输出；`QuestRegistry` 只消费编译后的只读定义。

### 7.2 配方与机器

配方编辑器使用机器、输入、输出和条件面板；脚本具有相同的机器/配方创建和修改能力。编译器检查：

- 输入和输出数量为正，引用物品、机器和标签存在。
- 手动激活条件只能配置到允许手动激活的机器。
- 重复输入、冲突机器配置和无法生产的配方给出明确诊断。
- 进度循环仅在明确标记为允许时通过，并以 warning 输出。

### 7.3 模型与动画配置

引擎编辑器负责导入、预览和配置，不负责替代 DCC 工具。首发支持 glTF 2.0：静态网格、PBR 材质、纹理、骨骼、蒙皮和动画片段。

模型领域管理导入设置、材质覆盖、碰撞、LOD 和资源引用；动画领域管理 clip、状态、过渡、混合和事件标记。UI 图编辑和脚本调用必须落到同一组 `SetAnimationTransition`、`SetAnimationParameter` 等命令。

当前资产上传只实现 OBJ 静态网格，场景只序列化 Transform、MeshRef、Camera。因此模型和动画领域的完整编辑器必须在 glTF、骨骼、蒙皮、动画采样和 `AnimatorComponent` 运行时链完成后交付，不能只先做空 UI。

### 7.4 脚本内容

Authoring Script 提供与领域命令一一对应的 AngelScript facade：

```angelscript
void create_smelting_content() {
    auto tx = authoring.begin_transaction("Create smelting content");
    auto furnace = authoring.machines.create(tx, "Furnace");
    auto recipe = authoring.recipes.create(tx, furnace, "Iron Smelting");
    authoring.recipes.add_input(tx, recipe, items.iron_ore, 1);
    authoring.recipes.add_output(tx, recipe, items.iron_ingot, 1);
    authoring.commit(tx);
}
```

该代码只说明 API 形状；实际绑定不得向脚本暴露 C++ 容器、裸指针或可变文档引用。编辑器也可调用受控的 Authoring Script 来创建内容，脚本结果必须以普通事务显示、校验和撤销。

`GameplayScript` 保留事件和行为 API，可读取 `CompiledContentBundle` 的定义并订阅玩法事件；它不包含上述 `authoring` namespace。

## 8. 当前接口的替换

当前 `GameContentRegistry` 同时承担脚本内容注册、运行时查询和 reload 所有权。该职责需要拆分：

1. 将任务、配方、机器的源定义和校验迁移到 `snt_game_authoring`。
2. 移除 `register_builtin_*`、`register_script_*` 和 `snt_register_*` 这类直接修改 runtime registry 的入口。
3. `snt_game_contentc` 将源文档编译为 `CompiledContentBundle`。
4. `snt_game_content_runtime` 读取 bundle，向 `QuestRegistry`、机器系统和 Gameplay Script 提供不可变查询。
5. 保留 `ScriptManager` 的模块加载、事务性编译和热重载能力，但它只管理脚本模块和行为回调，不再成为静态内容真源。

不保留旧注册 API 的包装层。所有现有 `p7_bootstrap.as` 中的静态机器和配方定义直接迁移到源文档；其后删除旧绑定、测试和兼容代码。

## 9. 预览、并发与持久化

### 9.1 预览

编辑器预览使用独立 `SimulationRuntime + ClientRuntime` 会话或等价隔离实例，加载提交前的不可变内容 snapshot。预览不能读取或写入正常游戏存档，也不能向现有游戏进程注入热重载。

“应用预览”只重新构建受影响的 bundle 分区并重启受影响的 preview subsystem；生产客户端和服务器依旧只在明确加载内容包时接收新定义。

### 9.2 并发

`ContentDocumentStore` 的提交和脚本执行固定在 authoring 主线程，保证命令顺序、Undo/Redo 和诊断稳定。导入、缩略图生成、glTF 解析和内容编译可在 worker 上运行，但必须：

- 只读取不可变 snapshot。
- 用 revision 检查结果是否过期。
- 通过新的提交事务写回结果。
- 不持有 MUI、AngelScript、ECS 或 Vulkan 对象。

### 9.3 持久化

保存时先将源文档写入同目录临时文件，完成 JSON 校验和 fsync 后原子替换。导入结果和 bundle 同样使用临时输出目录加原子切换，避免编辑器崩溃留下半个内容包。

## 10. 诊断、日志与测试

### 10.1 诊断与日志

所有校验和编译错误使用结构化 `ContentDiagnostic`：

```text
severity, code, asset_guid, asset_kind, source_path,
field_path, source_range, operation_origin, message
```

日志只在以下低频事件输出：事务提交/回滚、导入完成/失败、编译摘要、预览启动/停止、脚本编译/执行失败和依赖图异常。禁止逐帧、逐控件或逐节点输出日志。

### 10.2 自动化验收

- 同一组逻辑操作分别经 MUI 和 Authoring Script 执行，必须生成字节等价的源 snapshot 和 bundle。
- 事务失败、脚本异常、校验失败和导入失败均不得留下部分修改。
- Undo/Redo 后的文档、依赖图和编译结果必须与命令回放一致。
- `snt_game_contentc --validate` 在无 SDL、Vulkan、MUI 的构建闭包中运行。
- `GameplayScript` 无法解析或调用 `authoring` namespace。
- 模型/动画导入对无骨骼、损坏 glTF、缺失纹理、循环状态机和无效事件标记提供确定性错误。

## 11. 交付顺序

### P0：共同边界

建立 `snt_authoring_core`、GUID/引用、源文档、事务、命令日志、诊断和无头校验入口。此阶段先声明所有公开接口，再补实现。

### P1：任务书与配方

实现任务/配方领域命令、校验和编译；迁移 `p7_bootstrap.as` 的静态定义；提供最小 MUI 列表、Inspector、任务图、配方编辑和 Authoring Script facade。

### P2：编辑器工作区

完成文本编辑、树/虚拟列表、拖放、属性控件、Undo/Redo、搜索、内容浏览器、诊断面板和隔离预览。MUI 基础控件先于复杂图编辑器闭环。

### P3：模型与动画资产链

实现 glTF 导入、资源缓存、材质/骨骼/动画运行时、Animator 组件和模型/动画编辑工具。此阶段不引入 FBX 或自研 DCC 功能。

### P4：批处理与协作准备

提供 Authoring Script 批处理入口、命令回放、内容差异报告和可复现 bundle 构建。多人实时协作不是本阶段前提，但命令日志、GUID 和 revision 为其保留接口。

## 12. 完成定义

以下条件同时满足时，编辑器与脚本共用内容创作能力才算落地：

1. 编辑器和 Authoring Script 对任务书、配方、模型配置、动画配置均通过同一命令类型修改内容。
2. 两侧操作获得相同的校验结果、诊断、预览结果和编译 bundle。
3. 脚本操作可在编辑器中观察、撤销和重做；UI 操作也可通过命令日志回放。
4. 客户端、服务端和 Gameplay Script 只读取已编译内容，无法写入源文档。
5. `snt_simulation_runtime`、无头服务端和无头内容编译器不链接编辑器 UI、SDL 或 Vulkan。
6. 旧的静态内容脚本注册接口已删除，没有兼容层或第二条内容写入路径。
