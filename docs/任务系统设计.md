# ScienceAndTheology 任务系统设计

> 当前基线，更新于 2026-07-14。任务定义、运行态与存档职责均归 `game/`；`snt_engine` 不解释任务、奖励或玩家进度。

## 1. 所有权

| 域 | 所有者 | 当前状态 |
| --- | --- | --- |
| 热重载定义 | `GameContentRegistry` | 已实现：稳定 quest id、结构化 objective、ScriptId 事务和 revision |
| 玩家运行态 | `game/quest/QuestRegistry` | 已实现：按稳定 game account id + quest id 保存状态和计数 |
| 任务效果 | `IQuestRewardSink` | 接口声明，尚无生产 reward provider |
| UI/replication/save 脏标记 | `IQuestLifecycleSink` | 接口声明，状态变化仅提供值事件 |
| 跨重启持久化 | `IQuestProgressPersistence` / `GameSaveManager` | 已实现：P7.4 per-player `SNTQ` v1 文件 provider，调用时机由玩家生命周期拥有 |

`QuestProgressRecord` 只包含 quest id、状态、objective count、完成 tick、完成次数和奖励领取位；它不含 AngelScript 对象、ECS handle、原始指针或 source tree 路径。`QuestPlayerId` 直接接收认证后的 account id：Steam 是 `steam:<SteamID64>`，本地名字是 `local-name:<validated UTF-8 name>`。因此显示名相同的 Steam/local 玩家不会共享任务，完全相同的本地名字会有意落入同一任务存档。

## 2. 内容接口

AngelScript 内容在 `snt_register()` 中按以下顺序注册：

```angelscript
snt_register_quest("stone_age.smelt_iron", "Smelt Iron", "Craft iron ingots");
snt_add_quest_objective(
    "stone_age.smelt_iron", "craft.iron", "craft_item", "iron_ingot", 2);
snt_add_quest_prerequisite("bronze_age.forge_tool", "stone_age.smelt_iron");
```

当前 objective kind：`acquire_item`、`craft_item`、`mine_block`、`place_machine`、`reach_tick` 和 `custom`。每个 objective id 在同一任务中必须唯一；非 `reach_tick` objective 必须提供 target id。定义重载时仅替换内容值，既有 progress record 不会被清空。

## 3. 状态机

```text
Locked -- prerequisites met --> Available -- accept / auto_start --> InProgress
InProgress -- all objectives met --> Completed
Completed -- reset_repeatable --> Available
```

`QuestRegistry` 在 simulation main thread 的 fixed tick 中观察 `GameContentRegistry` revision、同步定义并推进 `reach_tick` objective。`record_progress()` 和 `update_inventory()` 只接受按值事件/查询；未来玩家、机器或网络命令系统必须携带认证后的 stable account id 后再调用它，不能以显示名作为替代 key。

状态转换记录低频 Info 日志；每次 objective 增量不会写 Info 日志。缺失 prerequisite 在内容 revision 变化时只告警一次，并保持任务锁定。

## 4. 重载与存档边界

内容重载使用定义 id 和 objective id 作为稳定键：

- 同 id 的 quest/objective 保留既有 count 和完成状态。
- 新 objective 初始 count 为零。
- 暂时从内容中移除的 quest progress 作为 orphan record 保留在玩家快照中；同 id 再出现时继续使用。
- `QuestProgressRecord` 和 `IQuestProgressPersistence` 位于 `game/quest/quest_progress.h`，不依赖 `QuestRegistry` 或内容脚本。`GameSaveQuestProgressPersistence` 位于 `snt_game_world_save`，因而存档目标只依赖值类型，避免反向链接 simulation target。
- P7.4 在显式 universe save 根下使用 `players/player_{hex_player_id}.quest`：文件内部保存原 player id，读取时校验 id、`SNTQ` v1、字段上限和 EOF；损坏、非当前版本与尾随字节一律拒绝，不维护旧格式读取。
- 写入以 `.tmp -> .bak -> primary` 顺序保留可恢复的旧文件。只有 primary 缺失时读取 `.bak` 并发出一条低频 Warn；primary 损坏不会静默回退，避免把数据错误伪装成成功。
- `QuestRegistry::load_player_progress()` / `save_player_progress()` 只通过 provider 交换值。玩家登入/离开、save slot 和自动保存频率尚未由当前 session 决定，fixed tick 不做 I/O。

## 5. 方案取舍

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 迁移旧 `src/core/quest/QuestSystem` | 初始代码复用多 | 依赖旧 Godot event bus、旧序列化和旧命名，继续污染当前主链 |
| 放入 `snt_engine` 通用模块 | 其他游戏可能复用 | 任务 objective、奖励和玩家身份属于本游戏玩法，不是引擎语义 |
| game-owned `QuestRegistry` + 值快照（已选择） | 热重载定义与玩家进度分离；headless/联机可复用；P7.4 可直接持久化 | 当前仍需为机器、玩家和网络补充 stable player id 事件生产者 |
| 将任务写入每个 `planet_data.bin` | 文件较少，读取一个维度时可同时读取状态 | 多玩家会重写同一维度元数据，任务不属于空间维度，且 `save_dimension()` 易覆盖状态 |
| 一个 universe 共用任务文件 | 初始实现短 | 多玩家写入相互阻塞，局部存档会重写全部玩家数据 |
| per-player `players/*.quest`（已选择） | 玩家状态与 chunk/planet 解耦，按玩家加载/保存，路径可安全编码 | 多一个小文件目录，调用方必须拥有明确的 universe save 根与玩家生命周期 |

## 6. 验证与后续
