# 内容数据架构

本项目把“会频繁调数值、扩展条目、影响玩法节奏”的内容放在 Godot 脚本或资源中，把“需要稳定、性能、严格校验和跨系统一致性”的规则留在 C++。

## 放在 GD/资源中的内容

- 配方：手工合成、机器处理、未来的多方块/仪式配方。
- 掉落表：地形、方块、实体、战利品权重。
- 燃料数值：燃烧时间、热值、可用机器类型。
- 机器定义：显示名、可用 tier、配方 map 名、默认端口、UI 数据。
- 物品显示数据：名称、本地化、图标、颜色、堆叠上限、百科描述。
- 进度/解锁表：任务、科技、成就、可见性条件。

这些内容的原则是：改数据就能调玩法，不需要重新编译 GDExtension。

## 留在 C++ 中的核心

- 核心模拟：机器 tick、功率网络、物品/流体管线、AE2 自动合成、世界数据。
- 性能敏感查询：配方匹配、网络遍历、库存与资源统计。
- 校验器和注册表：检查 item/fluid/key 是否有效，拒绝重复或非法内容。
- 跨系统稳定接口：GDExtension binding、命令服务器、序列化 ID 规则。
- 确定性规则：材料 ID、form ID、非材料 item key、流体 key 等基础命名体系。

C++ 不应该继续承载大批“内容条目”。如果必须先放 C++，应该作为过渡，并暴露同等的 GD 注册接口。

## 内容引用规范

GD 内容默认使用 `item_key`，不要直接写魔法数字 item id。

- 材料物品 key 使用 `form.material`，例如 `ingot.iron`、`plate.wood`、`dust.copper`。
- 非材料物品 key 使用稳定蛇形名，例如 `gt_hammer`、`workbench`、`machine_hull_basic`。
- C++ 的 `ItemRegistry::get_item_id_by_key()` 同时支持材料物品和非材料物品。
- 只有在桥接旧系统或测试精确 ID 时才使用 `item_id`。

GD 内容入口目前是 `res://scripts/content/ContentDatabase.gd`。它负责收集内容表并调用 C++ 注册接口；C++ 负责校验、存储和运行时查询。

## Mod / Content Pack 系统

第三方内容通过 content pack 扩展游戏。pack 既可以来自本地目录（开发期），也可以来自 Steam Workshop（分发期）。所有 pack 共用同一套 manifest + entry script 约定，并通过 `ModLoader` autoload 统一加载。

### Pack 磁盘格式

- **开发期**：文件夹形式，放在 `user://mods/<mod_id>/` 或 `res://mods/<mod_id>/`（只读）。
- **分发期**：ZIP 包，由 `ModLoader.install_pack_zip()` 解压到 `user://mods/` 后按文件夹形式加载。

每个 pack 根目录必须包含：

- `manifest.json`：元数据、依赖、加载顺序约束。
- `manifest.json` 中 `entry_script` 指向的 GD 脚本（默认 `mod.gd`），继承 `ModEntryPoint`。

### manifest.json 字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `mod_id` | 是 | 小写蛇形，全局唯一，例如 `example_expansion`。 |
| `display_name` | 是 | 玩家可见名称。 |
| `version` | 是 | 语义化版本 `MAJOR.MINOR.PATCH`。 |
| `author` | 否 | 作者。 |
| `description` | 否 | 描述。 |
| `entry_script` | 否 | 入口脚本相对路径，默认 `mod.gd`。 |
| `dependencies` | 否 | 依赖数组，形如 `["beta_mod>=1.0.0", "gamma_mod"]`。 |
| `load_before` | 否 | 必须在本 pack 之前加载的 mod_id 列表。 |
| `load_after` | 否 | 必须在本 pack 之后加载的 mod_id 列表。 |
| `steam_workshop_id` | 否 | Steam Workshop item ID（整数）。 |
| `min_game_version` | 否 | 要求的最低游戏版本。 |

### 加载流程

1. `ModLoader` 在 `ContentDatabase` 之后、`NEIIndex` 之前运行（见 `project.godot` autoload 顺序）。
2. 通过配置的 `ModPackSource`（默认 `LocalPackSource` + `WorkshopPackSource`）发现 pack。
3. 解析每个 pack 的 `manifest.json`，重复 `mod_id` 去重（保留首个，记录冲突）。
4. 基于依赖 + `load_before`/`load_after` 做拓扑排序（Kahn 算法）；检测环和缺失依赖并跳过相关 pack。
5. 按顺序对每个 pack 实例化 entry script，调用 `register_content(registrar)`。
6. `ModRegistrar` 以 pack 为作用域包装 C++ 注册接口（`GDCraftingManager` / `GDRecipeDatabase` / `ItemDatabase`），记录每个 pack 注册的 recipe 名，检测跨 pack 重名冲突。
7. 所有 pack 注册完成后触发 `mods_loaded` 信号，`NEIIndex` 等下游索引据此构建。
8. 输出汇总报告：加载 pack 数、新增配方/处理数、问题列表。

### 命名空间约定

为避免跨 pack 冲突，pack 注册的 recipe 名建议加 `<mod_id>:` 前缀，例如 `example_expansion:craft_gilded_iron_block`。`ModRegistrar` 会以此前缀做冲突归因。

### 相关文件

- `scripts/mod/ModLoader.gd`：autoload，加载管线 + ZIP 安装工具。
- `scripts/mod/ModManifest.gd`：manifest 解析与校验、版本约束匹配。
- `scripts/mod/ModEntryPoint.gd`：pack 入口脚本基类。
- `scripts/mod/ModRegistrar.gd`：pack 作用域内容注册 + 冲突报告。
- `scripts/mod/ModPackSource.gd`：pack 发现源基类，含 `LocalPackSource` 与 `WorkshopPackSource`。
- `mods/example_expansion/`：参考用示例 pack。
- `scripts/test/test_mod_loader.gd`：manifest 解析、拓扑排序、环/冲突检测的单元测试。

## Mod 系统 v2（全面内容 + 原生玩法扩展）

在 v1（仅 recipe/物品）基础上，v2 让 mod 能够注册几乎所有内容类型，并通过自定义方块实体、事件总线和生命周期钩子提供接近 Minecraft 模组的原生玩法扩展能力。

### v2 可注册内容

`ModRegistrar` 现在提供以下注册 API（均在 pack 作用域内统计与冲突归因）：

| API | 说明 | 后端 C++ 注册表 |
|-----|------|----------------|
| `add_crafting_recipe` / `add_processing_recipe` | v1 配方 | `GDCraftingManager` / `GDRecipeDatabase` |
| `add_item(def)` | 动态非材料物品（ID 段 `0x80000000+`） | `GDItemRegistry` → `ItemRegistry::register_mod_item` |
| `add_fluid(def)` | 流体 | `GDFluidRegistry` → `FluidRegistry` |
| `add_fuel(def)` | 燃料 | `GDFuelRegistry` → `FuelRegistry` |
| `add_loot_table(key, entries)` | 加权掉落表 | `GDLootTableRegistry` → `LootTableRegistry` |
| `add_material` / `add_tree_species` / `add_crop_species` / `add_ore_vein` / `add_biome_rule` | 世界生成内容 | `GDTerrainContentRegistry`（共享实例） |
| `add_machine_type(def)` | 机器类型定义（GUI/槽位/功率） | `GDMachineDefinitionRegistry` → `MachineDefinitionRegistry` |
| `add_block_entity_type(key, tick, ser, deser)` | 自定义方块实体类型 | `GDCustomBlockEntityRegistry` |
| `subscribe_event(name, cb)` | 订阅模拟事件 | `ModEventBus` |
| `register_command(name, cb)` | 自定义游戏命令 | `GDGameCommandServer` |
| `register_ui_screen(id, scene)` | UI 界面 | `ModLoader._ui_screens` |

### 自定义方块实体（原生玩法扩展）

`BlockEntityType::CUSTOM`（枚举值 9）允许 mod 注册带独立状态和 GDScript tick 回调的方块实体，实现类似 MC 模组的"额外原生玩法"。

- **C++ 侧**：`BlockEntityRegistry::register_custom_entity(...)` 注册根方块 + 占用单元，状态以不透明 JSON 存储（`CustomBlockEntityState::state_json`）。
- **GD 侧**：`GDCustomBlockEntityRegistry.register_type(type_key, tick_cb, serialize_cb, deserialize_cb)` 注册类型与回调。
- **GDExtension**：mod 也可通过 GDExtension 提供原生 C++ 回调，绕过 GDScript 性能瓶颈；C++ 端 `CustomBlockEntityState` 的 JSON 状态对两种扩展方式透明。

### 事件总线（ModEventBus）

桥接 `GDTickSystem` 的 C++ 信号到 mod 的 GDScript 回调。已知事件名镜像 `GDTickSystem` 的信号列表（`creature_killed`、`player_mined_block`、`machine_started` 等）。

- `ModLoader.bind_event_bus(tick_system)` 在 tick 系统就绪后绑定。
- `registrar.subscribe_event(name, cb)` 订阅，回调签名匹配对应信号。
- 事件分发带崩溃隔离：回调失败时禁用该 mod 的事件订阅，游戏继续运行。

### 生命周期钩子

`ModEntryPoint` 现提供以下钩子（全部由 `ModCrashGuard` 包装）：

| 钩子 | 时机 |
|------|------|
| `register_content(registrar)` | 加载阶段，按拓扑顺序调用（v1 已有） |
| `on_all_packs_loaded(loader)` | 所有 pack 注册完成后（v1 已有） |
| `on_world_loaded(registrar)` | 世界加载完成 |
| `on_world_saving(registrar)` | 世界即将保存 |
| `on_mod_unloading(registrar)` | mod 即将卸载（热重载/关机） |

`ModLoader.notify_world_loaded()` / `notify_world_saving()` / `unload_mod(mod_id)` / `unload_all_mods()` 由世界管理器在对应时机调用。

### 崩溃隔离（ModCrashGuard）

GDScript 无原生 try/catch，`ModCrashGuard` 通过以下方式实现隔离：

1. 调用前校验对象/Callable 有效性；
2. 调用后再次校验对象是否被释放；
3. 失败则 `disable_mod(mod_id, reason)`，记录崩溃日志，跳过该 mod 后续所有生命周期与事件回调。

`ModEventBus` 与 `ModCrashGuard` 共享同一份禁用集合（单一事实来源），保证事件回调和生命周期钩子的禁用状态一致。

### 存档兼容（未知方块占位）

当 mod 被卸载后存档中残留的自定义方块实体，由 `ModLoader.record_unknown_block(type_key, state_json, pos)` 记录为"未知方块"占位，保留原始 type_key 和不透明状态 JSON。重新加载该 mod 后可恢复功能，避免存档损坏。

- `ModLoader.has_unknown_blocks()` / `get_unknown_blocks()` 查询占位列表。
- 世界加载器在遇到 `BlockEntityType::CUSTOM` 但 `GDCustomBlockEntityRegistry.has_type()` 为 false 时调用 `record_unknown_block`。

### ZIP 自动扫描

`ModLoader._ready()` 在发现 pack 前先调用 `scan_incoming_zips()`，扫描 `user://mods/incoming/` 下的 `.zip`，逐个调用 `install_pack_zip()` 解压到 `user://mods/<mod_id>/`，成功后删除原 ZIP，失败则保留供人工排查。Steam Workshop 订阅的 pack 可由集成层写入 `incoming/` 触发自动安装。

- `src/bindings/fluid/gd_fluid_registry.h/cpp`：流体注册 GD 绑定。
