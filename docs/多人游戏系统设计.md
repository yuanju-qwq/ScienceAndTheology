# 多人游戏系统设计

## 1. 架构概述

采用 **方案 B：C++ `snt_server` 模块 + 自定义二进制协议**。权威服务端整体下沉到纯 C++ 模块，自定义 TCP（命令 + 可靠同步）+ UDP（高频位置）协议。Godot 进程无论是 host 还是 dedicated，都只是 `snt_server` 的宿主 + 渲染客户端。

Steam 版本的内置局域网世界可以额外提供 Steam P2P / Relay 传输候选，用于 Steam 账号之间的连接辅助和 NAT 打洞。这里的 P2P 仅是底层传输，不改变权威服务端模型：开局域网的一方仍运行本地 `ServerCore`，其他玩家仍作为客户端提交命令。

```text
┌─ snt_server (纯 C++ 模块, src/server/) ──────────────────┐
│  ServerCore                                                │
│   ├── PlayerManager (per-player state)                     │
│   ├── CommandQueue (FIFO, per-tick drain)                  │
│   ├── TickSystem (复用 snt_core)                           │
│   ├── UniverseManagerCore (多星球并发)                     │
│   ├── StateSyncServer (per-observer delta)                 │
│   ├── SaveManager (独占存档)                               │
│   └── NetworkTransport                                     │
│         ├── TcpListener (命令 + 可靠同步)                  │
│         ├── UdpSocket (高频位置 + LAN 广播)                │
│         ├── SteamP2PTransport (仅内置 LAN host 可选)       │
│         └── ProtocolCodec (二进制帧编解码)                 │
└────────────────────────────────────────────────────────────┘
            ▲ TCP/UDP            ▲ TCP/UDP
┌───────────┴────────────────────┴───────────────┐
│ Godot Client (任意进程, 包括 host 本机)         │
│  ├── GDNetworkClient (GDExtension Node)        │
│  │     ├── connect(host, port)                 │
│  │     ├── submit_command(cmd) → TCP           │
│  │     ├── poll_deltas() → apply to WorldData  │
│  │     └── poll_player_positions() → 渲染      │
│  ├── ChunkRendererBridge (本地渲染)            │
│  └── PlayerController (本地输入 → command)     │
└────────────────────────────────────────────────┘
```

## 2. 设计约束

- 客户端只拥有：输入、预测、UI、渲染状态、本地探索迷雾。
- 权威世界状态只能由 C++ 服务端命令执行器修改。
- 命令协议保持 `Dictionary`（可序列化）。
- 存档权属于权威服务端，客户端不写存档。
- 同一 GDExtension 二进制可同时用于客户端和服务端进程。
- 不做客户端权威 P2P 架构，不做客户端写权限下放。
- Steam P2P / Relay 只作为 Steam 账号之间的内置局域网世界传输候选，不用于专用服务器。
- 高频动作通过预测+回滚解决延迟，不开放客户端直接写 WorldData。
- 不做反作弊；LAN 朋友局为主，支持可选密码。
- 玩家数量上限 20 人。

## 3. 核心数据模型

### 3.1 Runtime PlayerHandle 与 PlayerUuid

```cpp
// src/core/player/player_handle.hpp
using PlayerHandle = uint64_t;
constexpr PlayerHandle kInvalidPlayerHandle = 0;
constexpr PlayerHandle kSinglePlayerHandle = 1;
```

- `PlayerHandle` 只是服务端进程内的运行时 slot id，远程客户端连接时分配。
- 单人模式：固定 `PlayerHandle = 1`。
- 断线后 ID 不回收，允许 ID 空隙；重连后可以获得新的 `PlayerHandle`。
- 玩家长期身份使用 `player_uuid`，格式为 RFC 4122 UUID 字符串，作用类似 Minecraft UUID。
- Steam 账号的 `player_uuid` 由 `steam:<steam_id>` 确定性生成；offline/GitHub 版本使用首次启动时写入本地身份文件的 UUID。
- 世界存档内玩家数据写入 `players/<player_uuid>.json`，不使用运行时 `PlayerHandle` 作为文件名。

### 3.2 玩家状态容器

```cpp
class PlayerManager {
public:
    PlayerHandle create_player(/* inventory dims, spawn dim/pos */);
    void destroy_player(PlayerHandle id);
    bool has_player(PlayerHandle id) const;
    PlayerState* get_player(PlayerHandle id);
    std::vector<PlayerHandle> all_ids() const;
private:
    std::unordered_map<PlayerHandle, std::unique_ptr<PlayerState>> players_;
};
```

`PlayerState` 聚合该玩家的 inventory / equipment / mana / spell_book / source_law_data / current_dimension / current_chunk / 生命 / 饱食等。

### 3.3 多玩家活跃集

```cpp
// TickSystem API:
void add_player_chunk(PlayerHandle, const std::string& dim, int cx, int cy, int cz);
void remove_player_chunk(PlayerHandle, const std::string& dim, int cx, int cy, int cz);
void clear_player_chunks(PlayerHandle);
std::string get_player_dimension(PlayerHandle id) const;
```

- `rebuild_chunk_sets()` 计算所有玩家活跃集的并集。
- 距离分级（NEAR/MID/FAR）按"到任意玩家"的最小距离计算。
- 单人模式：只有一个 PlayerHandle，行为与单玩家完全一致。

## 4. 协议设计

### 4.1 通道

| 通道 | 协议 | 包类型 | 说明 |
|---|---|---|---|
| 命令 | TCP, reliable, ordered | `CMD_LOGIN` / `CMD_COMMAND` / `CMD_LOGOUT` | 客户端→服务端 |
| 同步 | TCP, reliable, ordered | `SYNC_SNAPSHOT` / `SYNC_DELTA` / `SYNC_PLAYER_STATE` | 服务端→客户端 |
| 位置 | UDP, unreliable | `POS_UPDATE` | 双向，高频（20Hz） |
| 发现 | UDP broadcast | `DISCOVER_REQ` / `DISCOVER_REPLY` | LAN 服务器发现 |
| Steam LAN | Steam P2P / Relay | 同一套二进制帧 | 仅 Steam 客户端加入 Steam 玩家开的局域网世界 |

### 4.1.1 LAN 传输策略

内置局域网世界使用传输候选列表：

| 候选 | 使用者 | 说明 |
|---|---|---|
| `direct_lan` | Steam host + Steam/offline 客户端 | 普通 TCP/UDP，同网段发现和连接；保证 GitHub/offline 客户端仍可加入 Steam 玩家开的局域网世界。 |
| `steam_p2p` | Steam host + Steam 客户端 | 通过 Steam Networking / Relay 建连，客户端优先选择；只用于内置局域网世界，不用于专用服务器。 |
| `direct_dedicated` | 专用服务器 | 独立服务器进程直接监听端口，不检查 Steam，不使用 `steam_p2p`。 |

Steam host 广播/展示局域网世界时应同时发布 direct LAN endpoint 和 Steam lobby/P2P endpoint。Steam 客户端优先尝试 `steam_p2p`，失败后可回退 direct LAN；offline 客户端只能使用 `direct_lan`。

### 4.2 二进制帧格式

`[magic][version][type][player_handle][payload_len][payload][crc32]`

这里的 `player_handle` 是连接进入该服务端后的运行时 slot，只用于本局路由。登录包必须携带 `player_uuid`，服务端用它加载和保存世界内玩家数据。

### 4.3 命令协议

`submit_command` 字典字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `player_handle` | int | 本局运行时 slot id，服务端按此路由玩家状态 |
| `player_uuid` | string | 登录阶段必填，稳定玩家身份，用于世界内玩家存档 |
| `client_tick` | int | 可选，客户端预测 tick，用于回滚对账 |

服务端拒绝时返回 `{"accepted": false, "reason": "...", "player_handle": ...}`。

服务端在响应中回显 `client_tick`，供客户端对账确认。

## 5. 同步机制

### 5.1 Per-observer Delta

```cpp
StateDelta compute_delta_for(PlayerHandle observer);
void register_observer(PlayerHandle, const std::vector<ChunkKey>& view);
void update_observer_view(PlayerHandle, std::vector<ChunkKey> view);
```

每个客户端只收到自己视野内 chunk 的 delta。本机客户端通过 `StateSyncClient` 直接 apply；远程客户端通过序列化 delta 发包。

### 5.2 批量 Delta 计算

```cpp
std::vector<std::pair<PlayerHandle, StateDelta>> compute_deltas_batch(
    const std::vector<std::pair<PlayerHandle, std::vector<ChunkKey>>>& observer_views);
```

多观察者在同一维度时，必须使用 batch API：先为所有观察者计算 delta，再统一清除脏标记。单观察者 `compute_delta_for()` 会在首个观察者后清脏，导致后续观察者丢失变更。

### 5.3 多星球并发

- 服务端维护 `std::unordered_map<dimension_id, PlanetRuntime>`，按玩家所在星球引用计数。
- 任一玩家在某星球 → 该星球 chunks 加载并 tick。
- 所有玩家离开 → 延迟卸载（沿用现有 unload 距离策略）。
- `VirtualPlanetSimulator` 继续推进无玩家星球的虚拟生产。
- Delta 按玩家当前维度过滤：只下发玩家所在星球的变更。
- 客户端形态不变：只渲染自己所在星球。

## 6. 客户端预测 + 回滚

### 6.1 预测范围

- **允许预测**：移动、挖掘等高频动作。
- **必须等服务端确认**：合成、机器交互等低频动作。

### 6.2 PredictionBuffer

```cpp
class PredictionBuffer {
public:
    void predict(uint64_t client_tick);
    std::vector<uint64_t> confirm(uint64_t client_tick);
    std::vector<uint64_t> reject_and_rollback(uint64_t client_tick) const;
    uint64_t last_confirmed_tick() const;
    size_t pending_count() const;
    const std::vector<uint64_t>& pending_ticks() const;
    void clear();          // 重置 last_confirmed_ 为 0（传送/切星球/全量同步时用）
    void clear_pending();  // 保留 last_confirmed_，仅清 pending（服务端全量快照时用）
};
```

### 6.3 预测流程

1. 客户端在本地预测前调用 `predict(client_tick)`，记录状态快照。
2. 客户端本地应用预测，同时将命令（含 `client_tick`）提交给服务端。
3. 服务端执行命令，在响应中回显 `client_tick`。
4. 客户端收到响应后调用 `confirm(client_tick)`，丢弃对应快照。
5. 若服务端拒绝（结果与预测不符），客户端调用 `reject_and_rollback(client_tick)`，回滚到 `last_confirmed_tick` 状态，重放所有 `tick >= client_tick` 的预测。

`PredictionBuffer` 仅跟踪 tick 元数据；实际状态快照和重放由 GDScript 宿主处理。

## 7. Godot 集成

- `GDNetworkServer`（GDExtension Node）：host/dedicated 用，封装 `ServerCore`。
- `GDNetworkClient`（GDExtension Node）：客户端用，封装 `NetworkClient`。
- `GDPredictionBuffer`（GDExtension RefCounted）：客户端预测跟踪。
- `GDGameCommandServer` 在网络模式下退化为本地命令转发器：把本机客户端命令投递给 `GDNetworkServer`，不再直接写 WorldData。
- `scripts/network/GodotSteamP2PAdapter.gd`：GodotSteam legacy P2P API 适配层，负责 `sendP2PPacket` / `readP2PPacket` / session request 等插件差异。
- `scripts/network/SteamP2PTransport.gd`：Steam LAN transport，负责 host/client 角色、peer 管理、可靠/不可靠通道、`PackedByteArray` 原始帧收发。
- `scripts/network/SteamLanSessionBridge.gd`：Godot 侧临时桥接层，把 Steam P2P 包封装为登录、命令、同步事件和位置包，并把远端玩家数据保存到当前世界的 `players/` 目录。

Steam LAN host/client 示例：

```gdscript
var transport := SteamP2PTransport.new()
add_child(transport)
transport.packet_received.connect(_on_steam_frame)

# Host:
transport.start_host()

# Client:
transport.connect_to_host(host_steam_id)

func _process(_delta: float) -> void:
	transport.poll()
```

`SteamP2PTransport` 只搬运二进制帧，不解析世界命令。接入 `ServerCore` 时应复用同一套 `ProtocolCodec` 帧，把可靠命令/同步帧放 `CHANNEL_RELIABLE`，位置类低频容忍丢包数据放 `CHANNEL_UNRELIABLE`。

## 8. Dedicated Server

- `ServerScene.tscn` headless 启动，无渲染节点。
- `godot --headless --server` 可被 LAN 客户端连接。
- 存档目录由服务端指定，支持多存档槽。
- 后续需支持跨世界物流和能源（激光输电站、轨道能量包）。

## 9. ME 网络跨星球

ME 网络需要后期结构支持跨星球物流。玩家 A 在星球 1、玩家 B 在星球 2 时，可通过 ME 网络跨星球传输物品/流体。
