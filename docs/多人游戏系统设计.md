# 多人游戏系统设计（LAN / Dedicated Server）

## 1. 任务定位

对应 [项目架构总览](项目架构总览.md) 阶段 13：**多人游戏：LAN / Dedicated Server**。

本文档不直接落地实现，而是给出**多个候选方案**、它们的取舍，以及一个推荐路线。最终方案在评审后确定，再拆分为具体任务。

## 2. 现状与不变量

### 2.1 已经具备的基础

| 基础 | 位置 | 多人化意义 |
|---|---|---|
| 服务端权威命令入口 | [GDGameCommandServer](../src/bindings/simulation/gd_game_command_server.h) | 命令已是 `Dictionary`，可序列化；校验/写入集中 |
| StateSync 增量同步 | [StateSyncServer](../src/core/simulation/state_sync_server.hpp) / [StateSyncClient](../src/core/simulation/state_sync_client.hpp) | 脏 chunk 跟踪、delta、snapshot 已实现 |
| 引擎无关核心 | `src/core/**` | 不依赖 Godot，可独立跑无头服务端 |
| 命令/同步边界约束 | [模拟与事件系统设计.md](模拟与事件系统设计.md) §8 | 客户端不得直接写玩法状态，已按服务端模型组织 |
| 多人方向约束 | [模拟与事件系统设计.md](模拟与事件系统设计.md) §8.4 | 单人=本地权威服务端；LAN=同服务端加网络监听；Dedicated=同服务端无本地客户端 |

### 2.2 当前隐含的"单人假设"（所有方案必须重构）

这些是阻塞多人化的硬约束，与具体网络方案无关：

1. **单玩家状态资源**
   - `GDPlayerInventory` / `GDPlayerEquipment` / `GDManaPool` / `GDPlayerSourceLawData` / `GDSpellBook` 都是单实例 Resource。
   - `GDGameCommandServer` 持有 `inventory_` / `equipment_` 单指针。
   - **必须改为按 `player_id` 索引的容器。**

2. **单玩家活跃集**
   - `GDTickSystem.set_player_chunk(dim, cx, cy, cz)` 只接受一个玩家位置。
   - `TickSystem` 的 active/sleeping chunk 集合由单点推导。
   - **必须改为多玩家活跃集的并集，且按玩家分桶计算距离分级。**

3. **单玩家所在星球**
   - `UniverseManager` 跟踪单个 `active_planet`，`ChunkRendererBridge` 单个 `active_dimension`。
   - 多人时玩家可能分布在多个星球。
   - **服务端必须支持多星球同时活跃；客户端只关心自己所在星球。**

4. **命令缺少玩家身份**
   - `submit_command(command)` 字典里没有 `player_id` 字段。
   - 服务端无法区分谁提交、是否拥有该动作权限。
   - **命令协议必须加入 `player_id`，服务端按 id 路由到对应玩家状态。**

5. **同步信号是进程内 Signal**
   - `terrain_cell_synced` / `inventory_synced` / `furnace_synced` 等是 Godot Signal，直接连到本进程渲染。
   - **必须改为可路由：本机客户端走 Signal，远程客户端走网络包。**

6. **存档单点写入**
   - `UniverseSaveManager` 假设单进程独占存档目录。
   - 多人下只有权威服务端写存档，客户端不写。
   - **当前已经满足约束，只需确保客户端不触发写。**

### 2.3 不变量（所有方案必须遵守）

- 客户端只拥有：输入、预测、UI、渲染状态、本地探索迷雾。
- 权威世界状态只能由 C++ 服务端命令执行器修改。
- 命令协议保持 `Dictionary`（可序列化）。
- 存档权属于权威服务端。
- 同一 GDExtension 二进制可同时用于客户端和服务端进程。

## 3. 共享重构（所有方案的前置工作）

无论选哪个网络方案，以下重构必须先做。这部分独立于网络层，且对单人模式零行为变化。

### 3.1 PlayerId 与玩家会话

```cpp
// src/core/player/player_id.hpp
namespace science_and_theology {
using PlayerId = uint64_t;
constexpr PlayerId kInvalidPlayerId = 0;
}
```

- `PlayerId` 在服务端进程内单调递增；远程客户端连接时分配。
- 单人模式：固定分配 `PlayerId = 1`，本地客户端直接使用。

### 3.2 玩家状态容器

新增 C++ `PlayerManager`（`src/core/player/player_manager.hpp`）：

```cpp
class PlayerManager {
public:
    PlayerId create_player(/* inventory dims, spawn dim/pos */);
    void destroy_player(PlayerId id);

    bool has_player(PlayerId id) const;
    PlayerState* get_player(PlayerId id);
    const PlayerState* get_player(PlayerId id) const;

    std::vector<PlayerId> all_ids() const;

private:
    std::unordered_map<PlayerId, std::unique_ptr<PlayerState>> players_;
};
```

`PlayerState` 聚合该玩家的 inventory / equipment / mana / spell_book / source_law_data / current_dimension / current_chunk / 当前生命/饱食等。

`GDGameCommandServer` 改为持有 `PlayerManager*`，命令执行时按 `command["player_id"]` 取 `PlayerState`，再校验"该玩家是否真的在那个位置/那个星球/拥有那个背包"。

### 3.3 多玩家活跃集

`TickSystem` 改造：

```cpp
// 旧：set_player_chunk(dim, cx, cy, cz)
// 新：
void add_player_chunk(PlayerId, const std::string& dim, int cx, int cy, int cz);
void remove_player_chunk(PlayerId, const std::string& dim, int cx, int cy, int cz);
void clear_player_chunks(PlayerId);
```

- `rebuild_chunk_sets()` 计算所有玩家活跃集的并集。
- 距离分级（NEAR/MID/FAR）按"到任意玩家"的最小距离计算。
- 单人模式：只有一个 PlayerId，行为与现状完全一致。

### 3.4 命令协议升级

`submit_command` 字典新增字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `player_id` | int | 必填，服务端按此路由玩家状态 |
| `client_tick` | int | 可选，客户端预测 tick，用于回滚对账 |

服务端拒绝时返回 `{"accepted": false, "reason": "...", "player_id": ...}`。

### 3.5 同步路由

`StateSyncServer` 扩展为按观察者计算 delta：

```cpp
// 旧：compute_delta(observed_chunks)
// 新：
StateDelta compute_delta_for(PlayerId observer);
void register_observer(PlayerId, const std::vector<ChunkKey>& view);
void update_observer_view(PlayerId, std::vector<ChunkKey> view);
```

每个客户端只收到自己视野内 chunk 的 delta。本机客户端通过 `StateSyncClient` 直接 apply；远程客户端通过序列化 delta 发包。

### 3.6 多星球并发

`UniverseManager` 服务端形态：
- 维护 `std::unordered_map<dimension_id, PlanetRuntime>`，按玩家所在星球引用计数。
- 任一玩家在某星球 → 该星球 chunks 加载并 tick。
- 所有玩家离开 → 延迟卸载（沿用现有 unload 距离策略）。
- `VirtualPlanetSimulator` 继续推进无玩家星球的虚拟生产。

客户端形态不变：只渲染自己所在星球。

## 4. 候选方案

下面给出三个差异明显的方案，覆盖"最少新代码"到"最大长期可控性"的光谱。

### 方案 A：Godot 高层 Multiplayer API（ENet + RPC）

**核心思路**：完全使用 Godot 内置 `SceneMultiplayer` + `ENetMultiplayerPeer`，命令和同步通过 `@rpc` 注解传输。服务端就是同一个 `WorldMap.tscn` 以 `--server` 启动；客户端以 `--client` 启动并 `connect_to_host`。

**架构**：

```text
┌─ Host / Dedicated Server (Godot 进程) ────────────────────┐
│  WorldMap.tscn                                             │
│   ├── GameCommandServer (GDGameCommandServer, authority=1) │
│   │     └── @rpc(any_peer) submit_command(cmd)             │
│   ├── TickSystem (drives C++ sim)                          │
│   ├── ChunkRendererBridge (仅 host 本机客户端启用)         │
│   ├── PlayerController × N (每个 peer 一个，host 本机=1)   │
│   └── NetworkSyncBus (新增, 把 C++ delta 广播给所有 peer)  │
│         └── @rpc(authority, unreliable_ordered) apply_delta│
└────────────────────────────────────────────────────────────┘
            ▲ ENet (LAN)  ▲ ENet (LAN)
┌───────────┴──────────────┴───────────┐
│ Client (Godot 进程)                   │
│  WorldMap.tscn (无 GameCommandServer) │
│   ├── PlayerController (本地)         │
│   ├── ChunkRendererBridge (本地渲染)  │
│   └── NetworkSyncBus (apply_delta)    │
└───────────────────────────────────────┘
```

**关键点**：
- `multiplayer.is_server()` 区分服务端/客户端。
- 服务端 `submit_command` 走 `@rpc("any_peer", "call_local", "reliable")`，本机客户端也走 RPC（保持路径一致）。
- 服务端每 tick 收集 `StateSyncServer` 的脏 chunk，按观察者计算 delta，序列化为 `Dictionary` 通过 `@rpc("authority", "unreliable_ordered")` 下发。
- 玩家位置/朝向高频更新走单独 `@rpc("any_peer", "unreliable")`，服务端广播给其他客户端。
- LAN 发现：用 Godot `MultiplayerSpawner` + 自定义 UDP 广播（端口 8910）。
- Dedicated Server：`godot --headless --server --main-scene res://WorldMap.tscn`，跳过 `ChunkRendererBridge` 初始化。

**优点**：
- 最少新代码：复用 Godot 传输、可靠/不可靠通道、LAN 发现、连接管理。
- 单人模式天然兼容：`ENetMultiplayerPeer` 也支持 `create_server(1)` 单人自连，或直接走离线 `OfflineMultiplayerPeer`。
- 调试方便：Godot 内置多人调试器、协议可见。
- 客户端预测/回滚框架可后续接入 `SceneMultiplayer` 的 `process_packet` 钩子。

**缺点**：
- 协议绑死 Godot RPC：未来想做纯 C++ headless 服务端、或非 Godot 客户端（如手机原生壳）需要再抽象一层。
- GDScript ↔ C++ 边界每帧多次 Dictionary 序列化，高频同步可能成为瓶颈。
- 反作弊弱：RPC 调用本身不校验，全靠服务端 `submit_command` 校验。
- 多玩家活跃集、多星球并发等核心重构（§3）仍要做，方案 A 不省这部分。

**适用场景**：LAN 联机为主、2–8 人小局、不追求独立商业服务端、希望最快出原型。

---

### 方案 B：C++ `snt_server` 模块 + 自定义二进制协议

**核心思路**：把权威服务端整体下沉到纯 C++ 模块 `snt_server`，自定义 TCP（命令 + 可靠同步）+ UDP（高频位置）协议。Godot 进程无论是 host 还是 dedicated，都只是 `snt_server` 的一个"宿主 + 渲染客户端"。长期目标：`snt_server` 可编译为独立无头二进制，不依赖 Godot。

**架构**：

```text
┌─ snt_server (纯 C++ 模块, src/server/) ──────────────────┐
│  ServerCore                                                │
│   ├── PlayerManager (per-player state)                     │
│   ├── CommandQueue (FIFO, per-tick drain)                  │
│   ├── TickSystem (复用 snt_core)                           │
│   ├── UniverseManagerCore (多星球并发)                     │
│   ├── StateSyncServer (per-observer delta)                 │
│   ├── SaveManager (独占存档)                               │
│   └── NetworkTransport                                     │
│         ├── TcpListener (命令 + 可靠同步)                  │
│         ├── UdpSocket (高频位置 + LAN 广播)                │
│         └── ProtocolCodec (二进制帧编解码)                 │
└────────────────────────────────────────────────────────────┘
            ▲ TCP/UDP            ▲ TCP/UDP
┌───────────┴────────────────────┴───────────────┐
│ Godot Client (任意进程, 包括 host 本机)         │
│  ├── GDNetworkClient (新增 GDExtension Node)   │
│  │     ├── connect(host, port)                 │
│  │     ├── submit_command(cmd) → TCP           │
│  │     ├── poll_deltas() → apply to WorldData  │
│  │     └── poll_player_positions() → 渲染      │
│  ├── ChunkRendererBridge (本地渲染)            │
│  └── PlayerController (本地输入 → command)     │
└────────────────────────────────────────────────┘
```

**协议设计**（简述）：

| 通道 | 协议 | 包类型 | 说明 |
|---|---|---|---|
| 命令 | TCP, reliable, ordered | `CMD_LOGIN` / `CMD_COMMAND` / `CMD_LOGOUT` | 客户端→服务端 |
| 同步 | TCP, reliable, ordered | `SYNC_SNAPSHOT` / `SYNC_DELTA` / `SYNC_PLAYER_STATE` | 服务端→客户端 |
| 位置 | UDP, unreliable | `POS_UPDATE` | 双向，高频（20Hz） |
| 发现 | UDP broadcast | `DISCOVER_REQ` / `DISCOVER_REPLY` | LAN 服务器发现 |

二进制帧格式：`[magic][version][type][player_id][payload_len][payload][crc32]`。

**Godot 集成**：
- 新增 `GDNetworkClient`（GDExtension Node）：客户端用，封装 `snt_server::NetworkClient`。
- 新增 `GDNetworkServer`（GDExtension Node）：host/dedicated 用，封装 `snt_server::ServerCore`。
- `GDGameCommandServer` 在网络模式下退化为"本地命令转发器"：把本机客户端命令投递给 `GDNetworkServer`，不再直接写 WorldData。

**优点**：
- 长期可控：协议、内存、线程模型全部自管，可做纯 C++ 无头服务端。
- 性能上限高：二进制协议、零拷贝、自定义压缩（chunk delta 可用 RLE/LZ4）。
- 反作弊强：协议不暴露 Godot RPC 表面，所有写入走命令校验。
- 可扩展：未来支持非 Godot 客户端（手机壳、WebAssembly）、跨版本兼容、录制回放。
- 与 `snt_core` 引擎无关原则一致，是 §8.4 文档"长期目标"的直接落地。

**缺点**：
- 新代码量最大：协议编解码、连接管理、重传、心跳、LAN 发现、版本协商都要自己写。
- 调试工具需自建（协议抓包、回放器）。
- 短期看不到收益：单人/LAN 小局用 Godot RPC 就够。
- 必须先完成 §3 共享重构，且要把 `UniverseManager` 的 GDScript 逻辑下沉到 C++。

**适用场景**：计划做长期商业服务端、需要反作弊、需要支持非 Godot 客户端、玩家规模可能超过 16 人。

### 方案 C：混合方案（Godot ENet 传输 + C++ 权威）

**核心思路**：传输层用 Godot `ENetMultiplayerPeer`（省去协议编解码），权威逻辑用 C++ `snt_server` 模块（保持引擎无关）。Godot RPC 只做"二进制 blob 透传"，不暴露玩法语义。

**架构**：

```text
┌─ Godot Host / Dedicated (进程) ───────────────────────────┐
│  WorldMap.tscn                                             │
│   ├── GDNetworkServerNode (GDExtension, 持有 snt_server)   │
│   │     └── 内部: PlayerManager + CommandQueue + TickSystem│
│   ├── GameCommandServer (退化为转发器)                      │
│   │     └── submit_command → GDNetworkServerNode.enqueue   │
│   ├── NetworkSyncBus (GDScript)                            │
│   │     ├── @rpc(any_peer, reliable) submit_command_blob   │
│   │     └── @rpc(authority, unreliable_ordered) apply_blob │
│   └── ChunkRendererBridge (仅 host 本机客户端)             │
└────────────────────────────────────────────────────────────┘
            ▲ ENet (Godot)        ▲ ENet (Godot)
┌───────────┴─────────────────────┴──────────────┐
│ Godot Client                                    │
│  ├── NetworkSyncBus                             │
│  │     ├── submit_command_blob (RPC to server)  │
│  │     └── apply_blob (RPC from server)         │
│  ├── GDNetworkClientCodec (解码 blob → delta)   │
│  └── ChunkRendererBridge / PlayerController     │
└────────────────────────────────────────────────┘
```

**关键点**：
- `submit_command` 字典在客户端用 `var_to_bytes` 序列化为 `PackedByteArray`，通过 `@rpc("any_peer", "reliable")` 发送；服务端 `bytes_to_var` 还原后投递给 `snt_server::CommandQueue`。
- 服务端每 tick 产出 per-observer `StateDelta`，C++ 侧序列化为 `PackedByteArray`，通过 `@rpc("authority", "unreliable_ordered")` 下发；客户端 `bytes_to_var` 还原后交给 `StateSyncClient::apply_delta`。
- 高频玩家位置同样走 ENet unreliable，但用紧凑二进制（不走 `var_to_bytes`，自定义 32 字节/玩家）。
- Dedicated Server：`godot --headless`，加载 `ServerScene.tscn`（只含 `GDNetworkServerNode` + `NetworkSyncBus`，无渲染节点）。
- LAN 发现：复用方案 A 的 UDP 广播。

**优点**：
- 复用 Godot ENet 传输（可靠/不可靠/顺序/重传/连接管理），不重复造轮子。
- 权威逻辑在 C++，保持引擎无关，未来可平滑迁移到方案 B（只需替换传输层）。
- 协议表面是 `PackedByteArray`，Godot RPC 不暴露玩法 API，反作弊介于 A 和 B 之间。
- 单人模式：`OfflineMultiplayerPeer` 或 `ENetMultiplayerPeer.create_server(1)` 自连，路径统一。

**缺点**：
- 两层抽象：Godot RPC 层 + C++ 权威层，调试时要跨层追。
- `var_to_bytes` / `bytes_to_var` 的 Dictionary 序列化有开销，但比方案 A 的逐字段 RPC 少。
- 仍依赖 Godot 进程跑服务端，纯 C++ 无头二进制还要再走一步（迁移到方案 B）。
- 共享重构 §3 仍要做。

**适用场景**：希望平衡短期落地速度和长期可控性、LAN + 小规模 dedicated、未来可能升级到方案 B。

## 5. 方案对比

| 维度 | 方案 A (Godot RPC) | 方案 B (C++ 协议) | 方案 C (混合) |
|---|---|---|---|
| 新代码量 | 少 | 多 | 中 |
| 协议可控性 | 低（绑 Godot RPC） | 高（自定义二进制） | 中（blob 透传） |
| 性能上限 | 中（Dictionary 序列化） | 高（二进制 + 压缩） | 中高（blob + 紧凑位置） |
| 反作弊 | 弱 | 强 | 中 |
| 纯 C++ 无头服务端 | 不支持 | 支持 | 不支持（需迁移到 B） |
| 非 Godot 客户端 | 不支持 | 支持 | 不支持 |
| LAN 发现 | Godot 内置 | 自建 | Godot 内置 |
| 单人兼容 | 天然 | 需离线模式适配 | 天然 |
| 与 §8.4 长期目标契合度 | 低 | 高 | 中 |
| 调试难度 | 低 | 高 | 中 |
| 共享重构 §3 依赖 | 必须 | 必须 | 必须 |

## 6. 推荐路线

### 6.1 推荐：方案 C 作为初始落地，保留向方案 B 演进的路径

**理由**：
1. §8.4 已明确"Godot headless 第一阶段、C++ `snt_server` 第二阶段、纯 C++ headless 长期目标"。方案 C 正好对应"第一阶段 + 第二阶段前期"：传输用 Godot，权威用 C++。
2. 方案 A 把协议绑死 Godot RPC，未来迁向 B 时协议层要重写，弃用成本高。
3. 方案 B 短期投入过大，且 LAN 联机小局用 Godot ENet 完全够用。
4. 方案 C 的 `PackedByteArray` blob 透传设计，使未来迁移到 B 时只需把 ENet 换成自建 TCP/UDP，C++ 权威层和协议编解码可复用。

### 6.2 分阶段计划

| 阶段 | 内容 | 依赖 |
|---|---|---|
| **M1: 共享重构（无网络）** | §3 全部完成。单人模式行为不变，但 `PlayerId=1` 贯穿命令、状态、活跃集、同步路由。 | 无 |
| **M2: 本地多客户端验证** | 同进程内构造 2 个 `PlayerController`，分别提交命令，验证 `PlayerManager` + 多玩家活跃集正确。仍无网络。 | M1 |

## 8. 不做的事

- **不做 P2P 架构**：服务端权威是 §8 已定约束，P2P 会破坏存档单点写入和反作弊。
- **不做客户端写权限下放**：即使为了延迟体验，也通过预测+回滚解决，不开放客户端直接写 WorldData。
- **不做 Godot 高层 Multiplayer 的 `MultiplayerSpawner` 自动同步节点**：玩法状态在 C++，Godot 节点只是表现，不应让引擎自动同步 Node 树。
- **不在 M1 阶段引入网络代码**：共享重构必须先于网络层，且单人模式必须零回归。
