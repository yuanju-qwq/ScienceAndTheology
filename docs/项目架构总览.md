# 科学与神学 — 项目架构总览

## 1. 项目定位

"科学与神学"是一款 **3D 方块/体素工厂自动化 + 探索游戏**。核心玩法受 GregTech 5 的复杂工业链和 Applied Energistics 2 的智能存储/自动合成启发；魔法系统参考「新生魔艺」(Ars Nouveau) 的符文组合理念，并结合祭坛仪式、方块结构和工业自动化。

世界方向已经从早期 2D 原型切换为类 Minecraft 的生成式 3D 方块世界。地表、洞穴、矿脉、遗迹、工厂和魔法结构都应落在同一体素坐标体系中，而不是通过地表/地下双层地图切换表达。

技术栈：**Godot 4.6（3D 渲染/表现层） + C++20 GDExtension（模拟核心）**。

## 2. 整体分层架构

```text
┌─────────────────────────────────────────────────────────┐
│ 表现层 Presentation                                      │
│ Godot Scenes / GDScript                                  │
│ Node3D WorldMap, CharacterBody3D Player, Camera3D, UI    │
│ ChunkRendererBridge3D, WorldObjectRenderer, UI scripts   │
├─────────────────────────────────────────────────────────┤
│ 命令与同步层 Command / Sync                              │
│ GDGameCommandServer, GameCommandServer.gd                │
│ 客户端输入提交 command，服务端校验写入并发 sync signal    │
├─────────────────────────────────────────────────────────┤
│ 绑定层 GDExtension                                       │
│ GDWorldData, GDTerrainContentRegistry, GDWorldGenConfig  │
│ GDTickSystem, GDPlayerInventory, GDMachine, AE2/Magic    │
├─────────────────────────────────────────────────────────┤
│ 核心系统层 snt_core                                      │
│ WorldData, TerrainGenerator, SaveManager, TickSystem     │
│ Machine, Power/Fluid/Item Networks, Crafting, AE2, Magic │
├─────────────────────────────────────────────────────────┤
│ 适配器接口层 snt_adapters                                │
│ IWorldAdapter, IRenderAdapter, IAudioAdapter, IInputAdapter│
├─────────────────────────────────────────────────────────┤
│ Godot 引擎层                                             │
│ godot-cpp, WorkerThreadPool, 3D physics/rendering         │
└─────────────────────────────────────────────────────────┘
```

设计边界：

- Godot 节点只负责表现、输入、UI 和本地预测。
- `GDGameCommandServer` 是当前同进程权威命令入口。
- C++ 核心必须能向无头服务端演进，不依赖相机、UI 或玩家输入节点。
- 世界数据以内容 key 和冻结配置为准，避免玩法内容散落在渲染脚本中。

## 5. 关键场景

`WorldMap.tscn` 当前结构要点：

- 根节点：`WorldMap (Node3D)`。
- 光照：`WorldEnvironment` + `DirectionalLight3D`。
- 世界：`ChunkRendererBridge (Node3D)`。
- 物体：`WorldObjectRenderer (Node3D)`。
- 玩家：`Player (CharacterBody3D)`，带 `CollisionShape3D`、`Head`、`Camera3D`。
- 选中框：`SelectionBox (MeshInstance3D)`。
- UI：`CanvasLayer`，包含准星、状态、目标名、快捷栏、背包、合成、百科、熔炉界面。
- 命令入口：`GameCommandServer (GDGameCommandServer)`。

## 7. Command / Sync 边界

当前单机原型按 server-authoritative 约束组织：

```text
3D PlayerController / UI
    └── submit_command(command)
            ↓
GDGameCommandServer
    ├── 校验距离、工具、背包、目标方块、放置冲突
    ├── 写入 GDWorldData / Inventory / world managers
    └── emit sync signals
            ↓
3D 渲染/UI
    ├── ChunkRendererBridge.on_terrain_cell_synced()
    └── inventory_synced / world_object_synced / furnace_synced
```

客户端脚本不得直接调用玩法写 API，例如：

- `GDWorldData.set_terrain_cell`
- `GDPlayerInventory.add_item`
- `GDPlayerInventory.remove_from_slot`
- `GDPlayerInventory.set_slot`
- `GDPlayerEquipment.equip` / `unequip`
- `WorkbenchManager.place_workbench`
- `FurnaceManager.place_furnace`
- `GDWorldData.set_terrain_cell`（梯子放置）

例外是初始化、加载、资源生成和非玩家玩法写入。新增不确定问题优先加低频日志，让运行日志说明命令来源、拒绝原因和写入前后状态。

## 8. 关键设计决策

### 8.1 3D 体素世界优先

后续地图、工厂、魔法结构和矿脉都优先按方块坐标设计。不要再新增依赖旧 2D 视觉图层的功能。旧 `layer_id` 参数只作为兼容 API 存在，目标是迁移为 `dimension_id + voxel/chunk coord`。

### 8.2 引擎无关核心

`snt_core` 不含 Godot 头文件，可在纯 C++ 单元测试和未来无头服务端中运行。Godot 绑定层只负责类型转换、信号桥接和编辑器/场景集成。

### 8.3 异步生成使用 WorkerThreadPool

Chunk 生成通过 Godot `WorkerThreadPool` 提交 native task。工作线程只读取不可变世界生成快照；结果回到主线程后 emit `chunk_ready`，再由渲染桥构建 3D chunk。

### 8.4 内容数据注册

地形/方块内容通过 GD/资源注册，冻结为 C++ 快照：

- 材质 key、运行时 ID、flags、硬度、工具需求、掉落。
- 生成规则、生物群系、矿脉、结构参数。
- 渲染映射和未来 block model 映射。

C++ enum 仅作为内置 legacy/builtin 兼容层，不再作为内容扩展入口。

### 8.5 神学魔法与工业弱依赖、强联动

魔法和工业可以独立推进；交汇点包括符文自动化生产、机器祝福、祭坛结构、地形/方块改造和高级材料处理。魔法结构也应落在 3D 方块世界中，避免写成独立 2D 玩法层。

## 9. EventBus → Godot Signal

`GDTickSystem` 桥接核心事件到 Godot Signal。常用信号：

| Signal | 来源事件 | 说明 |
|---|---|---|
| `machine_state_changed` | `MACHINE_STATE_CHANGED` | 机器状态变更 |
| `machine_recipe_completed` | `MACHINE_RECIPE_COMPLETED` | 配方完成 |
| `machine_error` | `MACHINE_ERROR` | 机器错误 |
| `power_overload` | `POWER_OVERLOAD` | 电力过载 |
| `chunk_generated` | `CHUNK_GENERATED` | chunk 生成 |
| `chunk_state_changed` | `CHUNK_STATE_CHANGED` | chunk 状态变化 |
| `entity_created` | `ENTITY_CREATED` | 实体创建 |
| `entity_destroyed` | `ENTITY_DESTROYED` | 实体销毁 |
| `terrain_changed` | `TERRAIN_CHANGED` | 方块/地形修改 |
| `item_dropped` | `ITEM_DROPPED` | 掉落生成 |
| `item_picked_up` | `ITEM_PICKED_UP` | 掉落拾取 |
| `player_inventory_changed` | `PLAYER_INVENTORY_CHANGED` | 背包变化 |
| `player_equipment_changed` | `PLAYER_EQUIPMENT_CHANGED` | 装备变化 |

## 11. 废弃项

以下内容是早期原型遗留，不再作为当前架构目标：

- 2D 主地图。
- 地表/地下双层图层切换。
- 双层透视/同屏可视系统。
- 以 TileMap 作为主世界渲染。
- 按图层分别维护小地图探索状态。
- 新增更多 2D 地图层来表达遗迹、工厂层或深矿层。

后续如果需要地图显示，应设计为 3D 世界投影、扫描器、地图物品或区域雷达，而不是恢复旧图层系统。
