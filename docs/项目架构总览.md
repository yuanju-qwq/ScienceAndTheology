# 科学与神学 — 项目架构总览

## 1. 项目定位

一款 2D 工厂自动化 + 探索游戏，核心玩法受 GregTech 5（工业科技）和 Applied Energistics 2（智能存储与自动合成）启发；魔法系统参考「新生魔艺」(Ars Nouveau) 的符文组合理念，采用祭坛仪式玩法。世界探索部分借鉴 Terraria / Core Keeper 的双层互通地图设计。

技术栈：**Godot 4.6（渲染/表现层） + C++20 GDExtension（模拟核心）**。

## 6. 关键设计决策

### 6.1 引擎无关核心

`snt_core` 静态库不含任何 Godot 头文件依赖，可在纯 C++ 单元测试中独立编译运行。这意味着：
- **可移植**：核心逻辑可运行在无头服务器上
- **可测试**：不需要 Godot 编辑器即可运行单元测试
- **代价**：需要维护 GDExtension 绑定层作为桥梁

### 6.2 异步生成使用 Godot WorkerThreadPool

Chunk 异步生成使用 Godot 内置 `WorkerThreadPool::get_singleton()`，通过 `add_native_task()` 提交 C 函数回调到工作线程。结果通过 mutex 保护的队列返回主线程，在 `process_async_results()` 中消费并 emit `chunk_ready` 信号。

### 6.3 自定义噪声保留

自定义 `NoiseGenerator`（Perlin 分形噪声，~140 行）保留，原因：
- 纯 C++ 内联方法，零虚函数开销
- 栈上构造，天然线程安全（每个 chunk 生成线程创建独立实例）
- 与 Godot `FastNoiseLite` 相比，在异步工作线程上使用 Godot Object 子类存在线程安全风险

> 将来若需要 Cellular/Warp 等高级噪声，可引入 [FastNoiseLite 独立 C++ 头文件](https://github.com/Auburn/FastNoiseLite)（非 Godot 绑定版本）直接嵌入核心层。

### 6.4 适配器接口层（当前为桩）

`src/adapters/` 定义了 4 个纯虚接口（`IWorldAdapter` 等），用于将核心层与特定引擎解耦。当前所有接口仅包含虚析构函数，具体方法声明和实现待后续填充。

### 6.5 Server-Authoritative 边界约束

项目从当前单机原型开始按 **Server Simulation / Client Rendering** 约束组织代码。目标联机形态参考 Minecraft：单人、开放 LAN、专用服共用同一个权威服务端模型，不能分裂成三套玩法逻辑。

- **Singleplayer / Integrated Server**：开始单人游戏时自动启动本地权威服务端，本地客户端通过本地传输连接这个服务端；即使同进程，也按“客户端连接服务端”处理。
- **Open to LAN**：房主继续使用同一个本地权威服务端，只是额外打开局域网监听；房主客户端和局域网客户端都通过同一套 command/sync 接口交互。
- **Dedicated Server**：后续可单独打包无头服务端进程，不加载客户端 UI、TileMap、Camera、输入脚本和渲染资源，只运行权威模拟、存档、网络监听和日志。

所有形态必须共用同一套玩法命令语义和 C++ 权威模拟；差异只在启动方式、传输层和是否附带本地客户端表现层。

同进程是当前部署形态，不是耦合边界。客户端脚本仍必须像远程客户端一样提交 command、消费 sync，不允许因为“就在一个进程里”而直接改世界、背包、机器或放置状态。

当前同进程服务端入口是 `GDGameCommandServer`（C++ GDExtension），场景中的 `scripts/simulation/GameCommandServer.gd` 只负责把 `ChunkRendererBridge`、玩家背包/装备、工作台/熔炉/梯子管理节点接到 C++ 执行器：
- **客户端侧**：GDScript UI、输入、渲染脚本只提交命令、读取快照/同步信号，不直接写核心状态。
- **服务端侧**：`GDGameCommandServer` 是唯一允许执行玩法写入的命令入口，可调用 `GDWorldData.set_terrain_cell()`、`GDPlayerInventory.add_item()`、`remove_from_slot()` 以及当前仍待下沉的放置/熔炉管理器写接口。
- **同步侧**：服务端执行命令后必须发同步信号，例如 `terrain_cell_synced`、`inventory_synced`、`world_object_synced`、`furnace_synced`。渲染桥和 UI 只响应这些同步信号刷新表现。
- **拒绝路径**：命令失败必须 `push_warning` 并发 `command_rejected`。遇到不确定问题优先加日志，让运行日志说明现场。

新增玩法规则：
1. 不允许在 `scripts/ui/`、`scripts/interaction/`、玩家输入脚本里直接调用核心写 API。
2. 新增写操作先定义 command 字典字段，再在 `GDGameCommandServer.submit_command()` 中集中校验和执行；`GameCommandServer.gd` 不允许实现命令分发或写状态逻辑。
3. 客户端可做预测/预检查，但服务端必须重复校验距离、资源、工具、目标状态等关键条件。
4. 单人和局域网开放时，本机玩家不能获得特殊写入路径；本机客户端也必须走 command，和远程客户端只在传输方式上不同。
5. 后续迁移 LAN 或专用服务器时，`GameCommandServer` 的命令语义要保持稳定，只替换传输端；放置、熔炉等仍在 GDScript manager 中保存的服务端状态应继续下沉到 C++。
6. 新增服务端能力必须能脱离客户端场景运行；如果必须依赖 UI、TileMap、Camera 或玩家输入节点，说明它还不是服务端能力。

Dedicated Server 演进路线：
1. **近端目标**：Godot headless/server export，可复用 GDExtension 和服务端场景，快速得到可独立启动的局域网/专用服务端包。
2. **中期目标**：新增 `snt_server` C++ 模块，承载命令队列、玩家会话、tick、存档、快照/增量同步；`GDGameCommandServer` 退化为 Godot 包装层。
3. **长期目标**：纯 C++ headless server binary 可链接 `snt_core`/`snt_server` 独立运行；Godot 客户端只是其中一种客户端。
