# 玩家状态与交互产品设计

> 产品基线，更新于 2026-07-16。本文冻结新版 server-owned player 的玩法规则、持久化边界和后续实施顺序。旧 Godot 玩家对象、JSON 存档和 client-trusted 交互只作为迁移来源，不进入新运行时。

## 1. 已确认规则

1. 断线后，服务端立即保存玩家状态并销毁其 ECS 实体；断线玩家不占用 AOI、实体同步或模拟预算。
2. 受控重启、正常断线和后续重新登录必须恢复玩家位置、背包、装备、床重生点和器官状态。连接 peer、entt handle、脚本对象和 UI 状态不得写入存档。
3. 玩家死亡时，除装备外的全部背包物品转入一个不可破坏的墓碑存储体；装备保留在玩家身上。重生优先使用有效床重生点，没有有效重生点时使用世界出生点。
4. 背包和工具遵循 Minecraft 风格：固定槽位、主/副手、头/胸/腿/脚装备槽、物品实例数据不堆叠、普通物品按内容定义的最大堆叠数合并。当前 36 个背包槽和默认 64 堆叠是可配置初始值，不是网络协议常量。
5. 多人共享状态采用“主机权威、客户端可信提示”。主机仍是世界、背包、装备、床点、墓碑和存档的唯一最终提交者；客户端负责本地射线、挖掘进度、所选物品、机器 cover/ignition/structure/tool 提示、预测和表现。项目不以反作弊为目标，主机不重算这些高成本提示，只保留认证会话、同 dimension、基础距离、加载区块、内容目录、库存不为负和跨 world/inventory 事务等低成本一致性检查。
6. 联机展示优先性能：服务端以 AOI 裁剪同步，但 AOI 内必须同步其他玩家。具体半径、更新频率、实体 LOD 和客户端插值策略由运行时配置与性能测试决定，不能写死在玩法协议中。
7. 任务使用独立 MUI 任务书；任务在前置完成后自动解锁并由 `QuestRegistry` 的游戏事件/奖励边界推进，任务书只展示复制快照并显式请求领奖。Steam 成就不作为任务书替代物，后续通过独立 Steam adapter 接收已确认的游戏成就事件。

## 2. 新旧取舍

| 旧行为 | 新版决定 |
| --- | --- |
| 9x4 背包、默认 64 堆叠 | 保留为 Minecraft 风格的默认配置，但用 string item id 和当前格式值存档重建 |
| secondary item 不堆叠 | 保留为 item instance data 非空即不堆叠，不复制旧 `secondary_id` API |
| 7 个装备槽，含旧 `ARM` | 替换为 Minecraft 风格主手、副手、头、胸、腿、脚 6 槽，不保留 `ARM` 兼容层 |
| 普通交互 6 格、攻击 5 格 | 替换为 server config 的通用权威可达性；攻击单独在战斗设计中定义 |
| GDScript 客户端直接改背包、方块和机器 | 废弃；所有跨背包/世界变更必须经过服务端事务 |
| JSON 玩家文件和 Godot Resource 指针 | 废弃；使用严格、版本化、可恢复的当前格式二进制文件 |

## 3. 持久化状态

每个账号拥有一个 `SNTP` 当前格式文件，路径使用账号 id 的 hex 编码，不把显示名写进文件名。它保存：

- 当前 dimension 和 block-grid 位置；
- 可选床重生点；
- 固定槽位背包与装备；
- 器官状态的 schema id、schema version 和有界值 payload；
- 未来由器官网络模块拥有的具体解释，不由网络层或通用存档层猜测字段。

写入采用 `.tmp -> .bak -> primary`。只在 primary 缺失时读取 backup；损坏 primary 必须显式拒绝，不能被旧备份静默覆盖。任务进度仍使用独立 `SNTQ` 文件，两个文件各自原子，不伪造跨文件事务。

## 4. 生命周期

```text
认证成功
  -> 读取 SNTP / 首次创建默认状态
  -> 创建 server-owned ECS player entity
  -> 绑定当前 peer

同账号顶号
  -> 保留同一 live entity
  -> 转移 peer，不写盘、不重建背包

正常断线
  -> 捕获值状态并写 SNTP
  -> 保存 SNTQ
  -> 销毁 player entity，移除 AOI 和 peer 映射

受控关服
  -> 对每个在线玩家捕获并写 SNTP/SNTQ
  -> 销毁 player entity
```

如果写盘失败，服务端仍销毁断线实体，但保留内存中的待写值并在受控关服或同账号重连时重试；失败必须低频记录错误，不能伪装成已保存。床设置、床重置、死亡重生和墓碑领取会在主线程先标记玩家状态为 dirty，并由 `after_fixed_tick` 的配置化 checkpoint 写入 `SNTP`；它们不在游戏 tick 内直接做文件 I/O。

## 5. 死亡与重生

死亡结算是一个服务端事务：先捕获玩家值和预解析重生点，成功创建不可破坏墓碑后才清空非装备背包，随后写入重生位置并清空连续移动缓存；任一步失败都会回滚已创建的墓碑或已清空的背包。墓碑是 game-owned `GameChunkSidecar` 记录、`CUSTOM` block-entity anchor 和不可破坏 terrain cell 的一致组合，不是可由客户端伪造的掉落包。装备、器官状态、床重生点和任务状态不进入墓碑。

重生选择顺序：

1. 仍存在的床 anchor；若已加载且存在安全格，选择床附近安全格，周围填满或 dimension 未加载时仍返回床旁的 canonical 格；
2. 当前世界出生点；
3. 出生点不可用时由世界服务在配置半径内寻找最近安全格，找不到则拒绝死亡结算而不丢失物品。

当前 server service 的墓碑、床和重生行为：

- 墓碑仅所有者/管理员可读取；玩家背包能一次性容纳全部物品时直接取回并删除墓碑，不能容纳时保留墓碑和 contents 值供后续墓碑 UI 分槽领取，没有领回期限；
- 死亡点没有可放置墓碑的安全格时往上寻找可放置墓碑点；
- 床被破坏、就重生在出生点，dimension 卸载或周围被填满时不判断无效依然重生在床附近；
- 死亡不改变器官稳定、畸变、耐久或其他成长数值。

## 6. 权威交互与网络

`GameServerPlayerState` 向命令层提供认证 peer 对应的值快照、当前位置和背包事务。客户端先本地执行射线、动画和预测，并提交目标、所选物品及玩法提示；主机不将这些提示视为反作弊对象，而只把它们作为允许执行共享状态事务的输入。未来的 `PlayerCommandService` 必须按以下顺序工作：

```text
客户端目标、所选物品和可信玩法提示
  -> 认证 peer / 当前 dimension / 基础 reach / loaded chunk 验证
  -> server content catalog 解析允许的最终世界值
  -> 预检 world change 和 inventory transaction
  -> 主线程原子提交 world + inventory
  -> 发布 quest / replication / UI 值事件
```

主机不会为方块编辑逐格重放客户端射线、重算挖掘时长、验证客户端选择的工具 tag，或重新扫描 machine 的 cover/ignition/structure 前置条件；这些都属于可信客户端提示。主机仍不能接受任意 terrain material、负库存、未加载 chunk、跨维度目标或会破坏墓碑/machine sidecar 的写入，因为它们会损坏共享状态而不是单纯影响反作弊效果。客户端应先本地表现，收到主机提交后的 replication 再收敛；这样交互延迟不进入视觉路径。

AOI 和 snapshot source 只读取服务器 ECS/Chunk 值。首个 snapshot 至少含本玩家与 AOI 内其他玩家；delta 只发送有 revision 的位置、装备展示或状态变更。客户端 remote-world consumer 仅应用有序值更新，不能反向拥有权威玩家实体。

### 6.1 当前同步实现

`GameServerPlayerReplication` 已作为 `ScienceAndTheologyServerSession` 的 concrete interest provider 和 snapshot source 接入。它只从 `GameServerPlayerState` 复制已认证实体的值：观察者自身必定包含在快照中；其余玩家必须处于同一 dimension、可配置的水平圆形 AOI 和垂直范围内。候选按“自身、距离、稳定 EntityGuid”排序，并同时受 `server_replication.max_visible_players` 与每 peer entity/byte budget 限制。

玩家 entity blob 是 game-owned v1 值格式，只含认证身份、display name、dimension、block-grid position 和六个装备槽的 item id；背包、装备实例数据、器官 payload、tool tag、peer id 和 ECS handle 不会离开服务端。首个 snapshot 建立 observer baseline，之后只有新增、位置/展示装备变化或离开 AOI 的 `upsert/remove` delta；断线和顶号会丢弃旧 peer baseline。图形宿主把有序 Snapshot/Delta 交给 `GameRemotePlayerWorld`，它只维护远端玩家值并在本地网络断线时清空。

当前 `SNTG` v9 保留 v5 引入的唯一 UDP/unreliable `ClientMovementInput`：客户端只发送严格递增 sequence、前后/横移轴、冲刺/跳跃 flag 和 centidegree look，不发送位置。`GameServerCommandSink` 每 peer 合并最新 intent，`GameServerPlayerMovement` 在 fixed tick 主线程执行速度、重力、跳跃、体素碰撞和 stale-input timeout，并把 block-grid 结果写回现有 AOI source。网络 client 不再注册本地 demo physics controller，而是按变化或 heartbeat 上报 intent，并用自身的权威 player value 更新相机。

连续物理目前仍只存在于服务端；保存和 player v1 replication 仍是 block-grid 位置。客户端没有 prediction、reconciliation smoothing、插值或 remote avatar render entity，因此展示优先性能的最终体验仍待这些表现层工作完成。

## 7. 预留接口

- `IGamePlayerStatePersistence`：`SNTP` 值读写。
- `IGamePlayerOrganStateCodec`：器官网络模块解释/验证其独立 payload。
- `IGamePlayerGraveStore`：创建、授权读取和删除不可破坏墓碑；`GameServerPlayerGraveStore` 以 sidecar/custom anchor/terrain cell 实现，跨 inventory/world 的预检仍由 death service 保持主线程事务。
- `IGamePlayerBedLocator`：重生 resolver 只查询当前世界是否仍有 bed anchor；方块编辑服务负责注册/移除 anchor。
- `IGamePlayerRespawnResolver`：按床、世界出生点和安全格规则解析位置；当前实现为 `GameServerPlayerRespawnResolver`。
- `IGameServerPlayerStateCheckpointSink`、`IGameServerPlayerMotionReset`：死亡和床服务分别请求 barrier 后持久化、清除移动缓存，不反向持有 lifecycle 或 physics 内部状态。
- `IQuestBookPresenter`：客户端任务书接收 `QuestRegistry` 值快照，不持有任务可变状态。
- `IGameAchievementSink`：游戏层提交已确认成就事件；Steamworks adapter 后续实现它，任务书不依赖 Steam SDK。

## 8. 实施顺序

1. 已完成：实现 `SNTP`、固定槽位背包/装备/可选床点/器官 payload，接入断线保存与重连恢复。
2. 已完成：AOI 内其他玩家 snapshot/delta、client remote-player value world、v5 movement intent、服务端碰撞/重力和 block-grid 权威位置回写。
3. 已完成后端：床 anchor 注册/移除与重生点设置、死亡事务、不可破坏 sidecar 墓碑、床/出生点 resolver、直接全量领取和 player-state dirty checkpoint；战斗死亡 producer 与墓碑存储 UI 仍未接入。
4. 已完成 `SNTG` v9 `BlockInteraction` 的协议、网络会话 API 和 dedicated-server 事务：主机以可信客户端目标、所选物品和机器提示执行 mine/place/use、床设置、已锚定机器放置与 activation/collect 的跨 world/inventory 事务，并保护墓碑/machine sidecar。机器放置由 game-owned 独立 item-to-machine/material registry、内容提交/启动交叉校验、条件库存事务和 chunk-sidecar anchor 提交。图形客户端的具体鼠标/快捷键绑定与 interaction event 到 chunk/machine delta source 均尚未实现；客户端可预测，但目前不能靠实际 chunk replication 完成重新收敛。
5. 已完成任务书 UI、任务事件/reward effect 和任务书复制消费者；继续实现墓碑分槽 UI、Steam achievement adapter，以及 interaction event 到 chunk/machine 同步/UI 的生产消费者。
