# 模拟与事件系统设计

## 1. 概述

模拟层是连接 C++ 核心游戏逻辑与 Godot 渲染层的桥梁。核心组件：`TickSystem`（统一调度器）、`EventBus`（发布/订阅事件）、`MachineSystem`（机器子系统）、`StateSync`（状态增量同步），以及 `GDTickSystem`（Godot 绑定 + EventBus→Signal 桥接）。

命名空间 `science_and_theology`，位于 `src/core/simulation/`（12 个文件）。

## 2. TickSystem（统一模拟调度器）

### 2.1 核心参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 模拟频率 | 20 TPS | 每秒 20 ticks |
| 活跃 Chunk 半径 | 4（默认） | Chebyshev 距离 |
| 休眠 NEAR 间隔 | 20 ticks（默认） | 每 1 秒 tick 一次，半径 ×1~×2 |
| 休眠 MID 间隔 | 60 ticks（默认） | 每 3 秒 tick 一次，半径 ×2~×3 |
| 休眠 FAR 间隔 | 120 ticks（默认） | 每 6 秒 tick 一次，半径 ×3~×4 |

### 2.2 Tick 划分

```cpp
class TickSystem {
    WorldData* world_data_;
    std::unique_ptr<EventBus> event_bus_;
    std::unique_ptr<ErrorHandler> error_handler_;
    std::unique_ptr<StateSyncServer> state_sync_;
    std::vector<std::unique_ptr<SimulationSystem>> subsystems_;

```

### 2.3 每帧流程

```
tick(delta)
├── 1. rebuild_chunk_sets()
│     ├── 计算 player_cx, player_cy, player_cz Chebyshev 半径内的 Chunk → active_chunks_
│     ├── 半径 ×1~×2 → sleep_near_chunks_
│     ├── 半径 ×2~×3 → sleep_mid_chunks_
│     └── 半径 ×3~×4 → sleep_far_chunks_
├── 2. 按优先级排序 subsystems (0=Power → 1=Machine → 2=Fluid → 3=Logistics → 4=Region)
│     ├── for each subsystem: tick_active()  — 遍历 active_chunks_
│     ├── for each subsystem: tick_sleeping(NEAR) — 每 sleep_near_interval_ ticks
│     ├── for each subsystem: tick_sleeping(MID)  — 每 sleep_mid_interval_ ticks
│     └── for each subsystem: tick_sleeping(FAR)  — 每 sleep_far_interval_ ticks
├── 3. event_bus_->process_queue()  — 双缓冲排出延迟事件
└── 4. state_sync_->set_tick_counter(tick_counter_++)
```

### 2.4 休眠 Chunk 调度（距离分级）

休眠 Chunk 按距离分为三级，每级使用不同的 tick 间隔：

| 级别 | 距离范围 | 默认间隔 | 说明 |
|------|----------|----------|------|
| NEAR | active_radius+1 ~ ×2 | 20 ticks (1s) | 玩家可能即将到达 |
| MID  | ×2+1 ~ ×3 | 60 ticks (3s) | 中等距离 |
| FAR  | ×3+1 ~ ×4 | 120 ticks (6s) | 远距离，最低频模拟 |

所有间隔均可通过 `GDTickSystem` 在 GDScript 中运行时调整。

`should_tick_sleeping(cx, cy, cz, tier)` 使用空间哈希将 Chunk 分散到不同帧：
```
(hash = cx * 73856093 + cy * 19349663 + cz * 83492791)
tick 当 (hash % interval) == (tick_counter / interval) % interval
```

### 2.5 休眠机器同速低频策略

休眠 Chunk 中的机器采用**同速低频**策略：

- 机器运行速度与活跃 Chunk 完全相同
- 只是更新频率更低（每次补上间隔的 tick 数）
- `MachineSystem::tick_sleeping()` 将 `delta * kTicksPerSecond` 换算为 tick 数
- 调用 `Machine::tick_n(ticks)` 一次性推进，避免逐 tick 调用的开销
- `tick_n()` 在终端状态（NO_POWER / OUTPUT_FULL / ERROR）时提前终止
- 仅在最终状态变化时发射一次 EventBus 事件，避免事件风暴

其他子系统的休眠行为：
- BlockPhysicsSystem：事件驱动，与频率无关
- TreeGrowthSystem：休眠 chunk 不生长（设计意图）
- DayNightSystem / SeasonSystem：全局状态，无 per-chunk 工作
- RegionSystem：降频模拟（`delta * 0.25`）

## 3. EventBus（事件总线）

### 3.1 设计

线程安全的 pub/sub 系统，支持两种分发模式：

```cpp
class EventBus {
    using Handler = std::function<void(const GameEvent&)>;
    using HandlerId = uint64_t;

    HandlerId subscribe(GameEventType type, Handler handler);
    void unsubscribe(HandlerId id);

    void emit(const GameEvent& event);      // 立即同步分发（调用线程上执行）
    void enqueue(const GameEvent& event);   // 延迟排队（线程安全）
    void process_queue();                   // 排出延迟队列（双缓冲）
    size_t queue_size() const;
};
```

### 3.2 延迟队列双缓冲

```
enqueue → deferred_queue_ (mutex 保护)
process_queue():
  1. swap(deferred_queue_, processing_queue_)
  2. 遍历 processing_queue_ → emit(event) → 各 handler 执行
  3. processing_queue_.clear()
  # 在 handler 执行期间，新的 enqueue 进入新的 deferred_queue_，不会死锁
```

## 4. EventBus → Godot Signal 桥接（已实现）

### 4.1 架构

`GDTickSystem` 在 `set_world_data()` 后自动调用 `subscribe_to_event_bus()`，将 8 种核心 EventBus 事件桥接到 Godot Signal：

```
EventBus (C++ core)
  └── subscribe(GameEventType, lambda)
      └── lambda → emit_signal("signal_name", args...)
          └── GDScript: tick_sys.signal_name.connect(_handler)
```

### 4.2 已注册的 Godot Signal

| Signal | 参数 | 触发条件 |
|--------|------|----------|
| `machine_state_changed` | machine_id: int, old_state: int, new_state: int | Machine::tick() 状态变更 |
| `machine_recipe_completed` | machine_id: int, recipe_name: String | Machine::try_complete() 成功 |
| `machine_error` | machine_id: int, error_code: String, message: String | Machine 发生错误 |
| `power_overload` | node_id: int, demand: int, capacity: int | PowerNetwork 检测到过载 |
| `chunk_generated` | layer: String, cx: int, cy: int | TerrainGenerator 生成完成 |
| `chunk_state_changed` | layer: String, cx: int, cy: int, old_state: int, new_state: int | ChunkData 状态变更 |
| `entity_created` | entity_id: int, type_name: String, cx: int, cy: int | 新实体注册 |
| `entity_destroyed` | entity_id: int, layer: String | 实体销毁 |

### 4.3 订阅生命周期

```cpp
void GDTickSystem::set_world_data(gd_world) {
    unsubscribe_from_event_bus();     // 解绑旧的
    tick_system_ = new TickSystem(ptr);
    subscribe_to_event_bus();         // 绑定新的
}

GDTickSystem::~GDTickSystem() {
    unsubscribe_from_event_bus();     // 析构时自动解绑
}
```

### 4.4 GDScript 用法

```gdscript
@onready var tick_sys: GDTickSystem = $GDTickSystem

func _ready():
    tick_sys.machine_error.connect(_on_machine_error)
    tick_sys.chunk_generated.connect(_on_chunk_generated)
    tick_sys.power_overload.connect(_on_power_overload)
    tick_sys.entity_created.connect(_on_entity_created)

func _on_machine_error(machine_id: int, error_code: String, message: String):
    show_popup("Machine %d: %s" % [machine_id, message])

func _on_chunk_generated(layer: String, cx: int, cy: int):
    refresh_chunk_view(layer, cx, cy)

func _on_power_overload(node_id: int, demand: int, capacity: int):
    play_explosion_effect(get_node_position(node_id))
```

## 8. Command / Sync 边界（C++ 执行器）

当前单机原型采用同进程 server-authoritative 模式：`GDGameCommandServer`（C++ GDExtension）代表服务端执行玩法写入，Godot 输入/UI 代表客户端提交命令并接收同步。场景里的 `GameCommandServer.gd` 只继承 `GDGameCommandServer` 并完成节点接线，不实现命令处理。

后续联机方向按 Minecraft 式服务端模型演进：单人游戏自动启动本地权威服务端，本地客户端连接这个服务端；开放 LAN 是同一个服务端额外监听局域网；Dedicated Server 是同一个服务端不带本地客户端单独启动。所有形态都通过同一类 command 输入和 sync 输出驱动玩法；当前服务端和客户端可以运行在同一个 Godot 进程中，但设计上必须按进程外通信的边界处理：客户端只拥有输入、预测、UI 与渲染状态；权威世界状态只能由 C++ 服务端命令执行器修改。

### 8.1 命令流

```
Client GDScript
    ├── PlayerController3D / CraftingUI / FurnaceUI
    └── submit_command(command)
            ↓
GDGameCommandServer (C++ GDExtension)
    ├── 校验：目标状态、工具、背包、放置冲突、配方、燃料
    ├── 写入：GDWorldData / GDPlayerInventory / world managers
    └── emit sync signals
            ↓
Client Rendering/UI
    ├── ChunkRendererBridge.on_terrain_cell_synced()
    └── PlayerController.inventory_changed → HotbarUI / InventoryUI
```

### 8.2 当前命令

| Command | 来源 | 服务端写入 | 同步信号 |
|---|---|---|---|
| `mine_block` | `PlayerController` 的 `Camera3D` 射线 | `GDWorldData.set_terrain_cell`、背包掉落入库 | `terrain_cell_synced`、`inventory_synced` |
| `place_object` | `PlayerController` | `WorkbenchManager` / `FurnaceManager` / 地形方块（梯子）、背包消耗 | `world_object_synced` / `terrain_cell_synced`、`inventory_synced` |
| `craft_recipe` | `CraftingUI` | 背包消耗输入、加入输出 | `inventory_synced` |
| `furnace_take_output` | `FurnaceUI` | 熔炉输出槽、玩家背包 | `furnace_synced`、`inventory_synced` |
| `furnace_insert_input` | `FurnaceUI` | 背包消耗、熔炉输入槽 | `furnace_synced`、`inventory_synced` |
| `furnace_insert_fuel` | `FurnaceUI` | 背包消耗、熔炉燃料槽 | `furnace_synced`、`inventory_synced` |

### 8.3 写入约束

客户端脚本不得直接调用以下玩法写 API：
- `GDWorldData.set_terrain_cell`
- `GDPlayerInventory.add_item`
- `GDPlayerInventory.remove_from_slot`
- `GDPlayerInventory.set_slot`
- `GDPlayerEquipment.equip` / `unequip`
- `WorkbenchManager.place_workbench`
- `FurnaceManager.place_furnace`
- `GDWorldData.set_terrain_cell`（梯子放置）

例外：
- 初始化、加载/卸载、资源生成等非玩家玩法写入仍可由桥接/管理节点处理，例如 `ChunkRendererBridge` 请求 chunk 生成。
- `GDGameCommandServer` 是唯一允许集中调用这些写 API 的服务端入口；`GameCommandServer.gd` 只能做路径查找、资源注入和信号接线。
- 纯表现状态、UI 状态、探索迷雾、过场遮罩等客户端本地状态不受此约束。
- `WorkbenchManager` / `FurnaceManager` 目前仍保存少量世界对象状态；C++ 命令器可以临时调用它们，但新增服务端状态应优先落到 C++，不要继续扩展 GDScript manager 的写逻辑。梯子已迁移为地形方块，不再使用独立 manager。

### 8.4 服务端形态约束

局域网开放不是另一套玩法模式，而是把当前本地权威服务端暴露给局域网客户端：
- 单人：自动启动本地权威服务端，本机客户端通过本地传输提交 command。
- 开放到局域网：同一个服务端增加网络监听，房主本机客户端仍通过同一套 command 提交，远程客户端通过网络传输提交同结构 command。
- 服务端：统一在 `GDGameCommandServer` 或后续 C++ 服务端模块中校验、排序、执行命令，并广播 sync。
- 客户端：本机玩家和远程玩家都只消费 sync 更新 UI/渲染，不因房主身份获得直接写状态权限。
- 存档：保存权属于权威服务端，客户端本地只缓存表现/预测状态。

Dedicated Server 是同一权威服务端的独立打包形态，不应复制一套玩法逻辑：
- Godot headless/server export 可作为第一阶段专用服务端，加载最小服务端场景和 GDExtension，不加载客户端 UI、3D 渲染节点、Camera、输入脚本。
- C++ `snt_server` 模块应作为第二阶段承载命令队列、会话、tick、存档、快照和增量同步；`GDGameCommandServer` 只保留 Godot 绑定职责。
- 纯 C++ headless binary 可作为长期目标，直接链接 `snt_core`/`snt_server` 独立运行。
- 新增规则若无法脱离客户端节点运行，必须先拆出服务端依赖，不能直接进入权威模拟层。

### 8.5 日志约束

命令拒绝必须通过 `push_warning("GDGameCommandServer rejected ...")` 输出原因，并发 `command_rejected(command_type, reason)`。遇到不能确定的同步/状态问题，优先在 C++ 命令执行点或同步消费点加日志，让运行日志明确是谁提交、为何拒绝、写入前后状态是什么。

```gdscript
func _process(delta):
    tick_sys.tick(delta)
    tick_sys.set_player_chunk(player.layer, player.cx, player.cy)

    # StateSync
    for chunk in tick_sys.get_dirty_chunks():
        var delta = tick_sys.compute_delta([chunk])
        update_chunk_view(delta)

    # 或使用 process_async_results
    world.process_async_results()

# 错误 UI
func _on_error(machine_id, code, msg):
    $ErrorPanel.show_error(machine_id, msg)
```

## 10. Region Graph + 区域系统

### 10.1 架构

`RegionSystem`（优先级 5）管理 5 种 `RegionGraph`，每种类型一个图：

| RegionType | 索引 | 说明 |
|------------|------|------|
| POWER_GRID  | 0 | 电网子网络（变压器隔离） |
| FLUID       | 1 | 流体管道连通区域 |
| CONNECTED   | 2 | 通用 BFS 可达子图 |
| POLLUTION   | 3 | 污染扩散区域 |
| TEMPERATURE | 4 | 温度计算区域 |

### 10.2 RegionGraph（Union-Find 连通分量）

```cpp
class RegionGraph {
    // Node management
    uint64_t add_node(const RegionNode& node);   // O(1) amortized
    void remove_node(const RegionNode& node);     // marks dirty
    uint64_t link(const RegionNode& a, const RegionNode& b); // Union

    // Rebuild
    vector<uint64_t> rebuild_dirty(adjacency_fn);  // BFS split detection
    vector<uint64_t> rebuild_all(adjacency_fn);     // full recompute

    // Simulation
    void tick_pollution(neighbor_fn, diffusion_rate);
    void tick_temperature(neighbor_fn, conduction_rate);
};
```

核心数据结构：

```cpp
struct RegionNode {
    string dimension_id;
    int block_x, block_y, block_z;
};

struct RegionData {
    uint64_t region_id;
    RegionType type;
    string dimension_id;
    double pollution;      // [0, 1]
    double temperature;    // arbitrary scale
    size_t node_count;
};
```

### 10.3 增量更新策略

- **合并**：`link()` 使用 Union-Find，O(α(n)) 摊还。
- **删除**：`remove_node()` 标记区域为 dirty，不立即重建。
- **分裂检测**：`rebuild_dirty()` 在下次 tick 时对 dirty 区域做 BFS，发现分裂则创建新区域。
- **全量重建**：`rebuild_all()` 从零 BFS，用于初始化或存档加载后。

### 10.4 污染扩散模型

每 tick 对每个区域计算邻域加权平均：

```
new_pollution = pollution + (avg_neighbor_pollution - pollution) * diffusion_rate
```

`diffusion_rate` 默认 0.1，可通过 `RegionSystem::set_pollution_diffusion_rate()` 调节。

### 10.5 温度传导模型

与污染扩散同构，使用 `conduction_rate`（默认 0.05）。

## 11. SimulationSystem 多子系统并行

### 11.1 架构

`TickSystem` 支持两层并行：

| 层级 | 名称 | 说明 |
|------|------|------|
| L1 | 优先级组并行 | 同一优先级的多个子系统并发执行 |
| L2 | Chunk 级并行 | 同一子系统内多个 chunk 并行 tick |

并行执行默认关闭，需通过 `GDTickSystem.set_parallel_enabled(true)` 显式启用。

### 11.2 线程安全声明

`SimulationSystem` 新增 `is_thread_safe()` 虚方法：

```cpp
virtual bool is_thread_safe() const { return false; }
```

- 返回 `true`：TickSystem 可对该子系统使用 L1 和 L2 并行
- 返回 `false`：TickSystem 只在主线程顺序调用该子系统
- 默认 `false`，确保向后兼容

### 11.3 执行流程（并行模式）

```
tick(delta) [parallel_enabled_ = true]
├── rebuild_chunk_sets()
├── 按优先级分组 subsystems
│     ├── Priority 0: [DayNight]           → 顺序（单子系统）
│     ├── Priority 1: [BlockPhysics]       → 顺序
│     ├── Priority 2: [Machine]            → 顺序
│     ├── Priority 5: [Region]             → 顺序
│     ├── Priority 6: [Season]             → 顺序
│     └── Priority 7: [TreeGrowth]         → 顺序
│     （未来同优先级多子系统时，组内并行执行）
├── 每个 subsystem 内：
│     if is_thread_safe() && chunks > 1 && workers > 1:
│         将 chunks 分成 N 份，每份一个 worker 线程
│     else:
│         顺序遍历 chunks
├── event_bus_->process_queue()
└── state_sync_->set_tick_counter()
```

### 11.4 当前线程安全状态

| 子系统 | is_thread_safe() | 原因 |
|--------|-------------------|------|
| DayNightSystem | false | 写全局 state_ |
| BlockPhysicsSystem | false | 读写共享事件队列、修改 terrain |
| MachineSystem | false | Machine 内部状态非线程安全 |
| RegionSystem | false | processed_chunks_ 非线程安全 |
| SeasonSystem | false | 写全局缓存 |
| TreeGrowthSystem | false | 修改 block_entity_registry、terrain |

后续改造子系统使其线程安全后，可逐步标记为 `true`。

### 11.5 GDScript 接口

```gdscript
# Enable parallel execution
tick_sys.set_parallel_enabled(true)
tick_sys.set_max_worker_threads(4)  # 0 = auto

# Query
var parallel = tick_sys.get_parallel_enabled()  # false by default
var workers = tick_sys.get_max_worker_threads()  # 0 = auto
```

### 11.6 Worker 线程管理

- `max_worker_threads_ = 0`（默认）：自动使用 `hardware_concurrency - 1`，最少 1
- 使用 `std::async(std::launch::async, ...)` 启动 worker
- 每个 tick 结束时所有 future `.get()` 等待完成
- 不使用持久线程池（避免生命周期复杂性），未来可优化
