# 旧项目迁移与清理

> 当前清理基线，更新于 2026-07-24。本文只记录旧 Godot/GDExtension 树的保留理由、迁移范围和删除门槛；当前运行时能力以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准，玩法与联机状态分别以 [P7 玩法迁移设计](p7_玩法迁移设计.md) 和 [游戏网络协议设计](游戏网络协议设计.md) 为准。

## 1. 范围和原则

顶层 CMake 只构建 `snt_engine/` 与 `game/`。旧 `src/`、GDScript、Godot 场景和 GDExtension 不参与当前构建或运行包，只能作为迁移来源，迁移过程中不需要保证godot可以编译。

- 行数只用于表示遗留规模，不代表有效代码量、复杂度或工期。
- `src/bindings/` 是 Godot 包装层，不是迁移目标；底层功能替代后直接删除。
- 旧 `tests/core/` 不在当前 CTest 链。删除实现前，将仍有价值的断言迁入 `snt_engine/tests/` 或 `game/tests/`。
- 项目尚未发布，不保留旧 API、旧存档或 Godot 兼容层。每个旧功能必须被标记为 `保留`、`替换` 或 `废弃` 后才可删除对应来源。

## 2. 遗留清单

| 范围 | 规模 | 当前状态 | 处理方式 |
| --- | ---: | --- | --- |
| `src/core/` | 剩余未迁移旧域 | 部分数据与运行时能力已进入新主链，大量玩法域未迁入 | 逐域替换或废弃 |
| `src/bindings/` | 剩余旧包装源 | 旧 Godot GDExtension 包装层；网络包装仍在迁移验收中，预测包装已有原生替代 | 不迁移包装 API；先迁移底层行为并验收 |
| `scripts/` 和示例 mod | 109 个 `.gd`，26,437 行 | 目前只有脚本框架、炉子内容和基础 UI 原型 | 按玩法/UI 域重建 |
| Godot 入口 | 配置、场景、shader、locale | 只保留为迁移参考 | 替代方案验收后统一删除 |

历史 `src/server/` 的 `SNT1` 文件已不在当前工作树；这只说明旧树已被清理过，不构成其功能已经完整迁移的证据。旧 `src/bindings/multiplayer/` 网络包装和 `tests/core/m3_server_network.cpp` 仍作为未迁移网络行为的比对来源；本轮已完成替代的 `src/core/simulation/prediction_buffer.*`、`src/bindings/multiplayer/gd_prediction_buffer.*` 和 `tests/core/m6_prediction.cpp` 已从旧树及其 CMake 注册删除。旧 GDScript/GDExtension API 本身不是新接口；需要迁移和验收的是它承载的服务器、客户端、同步和预测行为。新版交互边界仅指 `SNTG` typed command 与 `GameServerPlayerInteractionService` 的原生 C++ 服务契约，不是 GDScript 兼容层。

本轮验证：Debug `snt_game_tests.exe --gtest_color=no` 通过 255 项测试（83 个套件），其中包含新 prediction/interpolation、远端 avatar ECS 投影、player v2 codec、独立 movement sequence、服务端连续移动与静止玩家 delta 去重断言；Debug 图形 client 可构建；当前串行 CTest 的 616 项已执行测试全部通过，另有 1 项 benchmark 按配置禁用。这些结果继续将 `PredictionBuffer` 与 `m6_prediction.cpp` 保持为可删除；仍标为“部分替代”的网络包装继续保留。后续若决定某项不进入新版范围，必须由产品明确标为废弃，并保留该决定与影响说明；不能仅因旧 API 不再使用就删除源文件。

`src/` 仅保留尚未迁移功能域的比对来源；其规模不代表当前运行时能力或迁移完成度。当前 `scripts/` 共有 223 个文件，其中包括 111 个会随脚本删除的 `.uid` 文件。

## 6. 取舍和下一步

| 路线 | 优点 | 缺点 |
| --- | --- | --- |
| 全功能等价迁移 | 最大限度保留旧玩法和内容资产 | 需要重建大量 Godot UI、GDExtension 边界和旧测试 |
| 按新版范围裁剪 | 最快收敛到无 Godot 的主链，符合不维护兼容的策略 | 需要明确记录被放弃的玩法 |

当前采用“按新版范围裁剪”：先为每个功能域记录唯一新归属和 `保留`、`替换` 或 `废弃` 决策，再按优先级迁移。P7.4 的机器运行态已以 chunk-sidecar anchor、稳定 Guid、v13 多输入/激活/手动任务 owner 状态恢复与未锚定拒绝保存完成，P7.2.4 已补充 cover/ignition/structure/tool tag 的 activation contract 与 bloomery 体素结构校验；玩家任务进度已以 revision-gated autosave 避免无变更定时重写，已提交玩家/机器事件自动推进前置任务图，显式 item reward 已通过主机任务 bridge 接入。

P7.5 已把 `SNTG` v20 的 snapshot/delta 值 codec、per-peer budget、provider 调度、客户端 drain、同 dimension 玩家 AOI、player v2 连续运动 presentation blob、per-observer delta baseline、客户端 `GameRemotePlayerWorld`、`GameClientQuestBookState` 与 typed `BlockInteraction` / `QuestClaimReward` / `InventorySlotTransfer` / `MachineInputSlotTransfer` 建立在无头网络边界内；任务书仅由认证账号的 server source 复制，并以 revision-gated delta 避免无变更重复发送。可靠 command 和 UDP movement 各有严格递增 sequence；server 只保留每 peer 最新 movement intent。客户端通过共享运动内核预测本地输入，收到 `last_processed_input_sequence` 后回放未确认输入并在帧边界平滑修正；远端连续状态进入 `GameRemotePlayerInterpolator`，再由 `GameRemotePlayerPresentationWorld` 每帧采样为带稳定 GUID、显示名、装备展示键、full/LOD mesh 和 yaw 的临时原生 ECS 实体。server-owned loaded-chunk AOI 会按配置选择已加载区块，进入 AOI 时发送 material/flags RLE terrain 全量 payload，退出时显式释放基线；主机已提交方块交互会生成稀疏 block delta，已锚定机器以不含 owner、配方输入或 ECS handle 的 presentation entity 同步。图形客户端的 `GameClientRemoteChunkWorld` 在覆盖本地 bootstrap terrain 前保存原值，收到退出、断线或关闭时恢复，并把变更交给 `ChunkRenderSystem` 重网格；`GameRemoteMachineWorld` 提供值缓存和稳定 anchor 查询。

生产机器 MUI 已读取该机器 presentation 的容量与输入/输出槽，以双向拖放提交 `MachineInputSlotTransfer`，且只在主机私有库存回执及下一帧 presentation 后确认显示；旧 Godot 炉子 manager/data/panel、Dictionary 命令与同步信号已随之删除。认证后客户端以 DDA local target、所选 hotbar item 和本地 cover/ignition/bloomery structure/tool 提示提交 mine/use/place/activate/collect/till/plant/fertilize/harvest，绝不直接写 world/inventory/save。MUI 任务书已消费任务缓存，以章节任务图表达前置关系并只允许领奖请求；它不提供接取入口。服务端已完成 movement/physics，以及方块（包括由内容 registry 映射、条件库存事务和 sidecar anchor 组成的机器放置）/床/已锚定机器和任务奖励的主机提交事务。新版玩家产品规则现由[玩家状态与交互产品设计](玩家状态与交互产品设计.md)冻结：主机权威、客户端可信提示，`SNTP` 状态、断线保存/实体销毁、重连恢复和 dirty-gated checkpoint 已完成；床、死亡、不可破坏 sidecar 墓碑与重生 resolver 已作为服务端后端接入。

战斗死亡 producer、墓碑 UI 和 Steam 成就 adapter 仍待实现；客户端提示不能直接写 world/inventory/save，但不需要主机重放反作弊式 raycast、挖掘进度或机器前置条件。
