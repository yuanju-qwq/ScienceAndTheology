# 旧项目迁移与清理

> 当前清理基线，更新于 2026-07-20。本文只记录旧 Godot/GDExtension 树的保留理由、迁移范围和删除门槛；当前运行时能力以 [项目架构与运行时基线.md](项目架构与运行时.md) 为准，玩法与联机状态分别以 [P7 玩法迁移设计](p7_玩法迁移设计.md) 和 [游戏网络协议设计](游戏网络协议设计.md) 为准。

## 1. 范围和原则

顶层 CMake 只构建 `snt_engine/` 与 `game/`。旧 `src/`、GDScript、Godot 场景和 GDExtension 不参与当前构建或运行包，只能作为迁移来源，迁移过程中不需要保证godot可以编译。

- 行数只用于表示遗留规模，不代表有效代码量、复杂度或工期。
- `src/bindings/` 是 Godot 包装层，不是迁移目标；底层功能替代后直接删除。
- 旧 `tests/core/` 不在当前 CTest 链。删除实现前，将仍有价值的断言迁入 `snt_engine/tests/` 或 `game/tests/`。
- 项目尚未发布，不保留旧 API、旧存档或 Godot 兼容层。每个旧功能必须被标记为 `保留`、`替换` 或 `废弃` 后才可删除对应来源。

### 3.1 P7.5 玩家库存增量迁移（已完成）

旧 Godot 背包拖放、`Dictionary` 同步和客户端直接改槽位不会迁入。当前主链只保留稳定槽位值、条件转移和服务器权威结果：`SNTG` v13 的 `InventorySlotTransfer` 带 request id、观察到的 inventory revision、源/目标槽位与两份期望 stack；`GameServerInventoryReplication` 在主线程调用 `GameServerPlayerState` 的原子条件转移，并在成功前标记 `SNTP` dirty checkpoint。

首次认证和重连发送玩家私有 `GameInventorySnapshot` 完整固定槽位快照；随后 `GameInventoryDelta` 仅发送实际变化槽位与最新请求回执。`GameClientInventoryState` 严格核对 account、outer snapshot id 和可靠 delta 序列，在 UI 边界重建完整快照，`GameplayUiController` 只在匹配回执后确认拖放。外层 provider 预算拒绝时，库存 source 不会推进其 observer baseline，下一批会重试同一 payload，因此不会因低带宽预算丢失状态。生产机器输入槽使用独立的 `MachineInputSlotTransfer`，在同一权威回执流中返回结果；容器/装备专用界面和墓碑分槽 UI 仍按各自产品范围迁移。

| `src/` | 对应功能已替换或正式废弃，关键行为测试已迁入当前 CTest；`src/server/` 已按此门槛删除 |

| `resource/terrain/` | 不删除；当前 `game/CMakeLists.txt` 仍复制它 |
| 其他 `resource/` | 先标记为运行包资产、源素材或废弃物，不能按目录批量删除 |
| 未跟踪 `bin/`、`.godot/` | 停止使用旧 Godot 工作流且当前运行包验证完成后再清理 |

最终删除前必须在干净构建目录完成配置、客户端/服务端构建和当前 CTest，并确认构建、运行包、测试与工具不再引用 `src/`、`scripts/`、`res://`、`godot-cpp`、`.gdextension` 或 Godot 场景路径。

## 6. 取舍和下一步

| 路线 | 优点 | 缺点 |
| --- | --- | --- |
| 全功能等价迁移 | 最大限度保留旧玩法和内容资产 | 需要重建大量 Godot UI、GDExtension 边界和旧测试 |
| 按新版范围裁剪 | 最快收敛到无 Godot 的主链，符合不维护兼容的策略 | 需要明确记录被放弃的玩法 |

当前采用“按新版范围裁剪”：先为每个功能域记录唯一新归属和 `保留`、`替换` 或 `废弃` 决策，再按优先级迁移。P7.4 的机器运行态已以 chunk-sidecar anchor、稳定 Guid、v13 多输入/激活/手动任务 owner 状态恢复与未锚定拒绝保存完成，P7.2.4 已补充 cover/ignition/structure/tool tag 的 activation contract 与 bloomery 体素结构校验；玩家任务进度已以 revision-gated autosave 避免无变更定时重写，已提交玩家/机器事件自动推进前置任务图，显式 item reward 已通过主机任务 bridge 接入。

P7.5 已把 `SNTG` v13 的 snapshot/delta 值 codec、per-peer budget、provider 调度、客户端 drain、同 dimension 玩家 AOI、presentation-only player blob、per-observer delta baseline、客户端 `GameRemotePlayerWorld`、`GameClientQuestBookState` 与 typed `BlockInteraction` / `QuestClaimReward` / `InventorySlotTransfer` / `MachineInputSlotTransfer` 建立在无头网络边界内；任务书仅由认证账号的 server source 复制，并以 revision-gated delta 避免无变更重复发送。server-owned loaded-chunk AOI 会按配置选择已加载区块，进入 AOI 时发送 material/flags RLE terrain 全量 payload，退出时显式释放基线；主机已提交方块交互会生成稀疏 block delta，已锚定机器以不含 owner、配方输入或 ECS handle 的 presentation entity 同步。图形客户端的 `GameClientRemoteChunkWorld` 在覆盖本地 bootstrap terrain 前保存原值，收到退出、断线或关闭时恢复，并把变更交给 `ChunkRenderSystem` 重网格；`GameRemoteMachineWorld` 提供值缓存和稳定 anchor 查询。

生产机器 MUI 已读取该机器 presentation 的容量与输入/输出槽，以双向拖放提交 `MachineInputSlotTransfer`，且只在主机私有库存回执及下一帧 presentation 后确认显示；旧 Godot 炉子 manager/data/panel、Dictionary 命令与同步信号已随之删除。认证后客户端以 DDA local target、所选 hotbar item 和本地 cover/ignition/bloomery structure/tool 提示提交 mine/use/place/activate/collect，绝不直接写 world/inventory/save。MUI 任务书已消费任务缓存，以章节任务图表达前置关系并只允许领奖请求；它不提供接取入口。服务端已完成 movement/physics，以及方块（包括由内容 registry 映射、条件库存事务和 sidecar anchor 组成的机器放置）/床/已锚定机器和任务奖励的主机提交事务。新版玩家产品规则现由[玩家状态与交互产品设计](玩家状态与交互产品设计.md)冻结：主机权威、客户端可信提示，`SNTP` 状态、断线保存/实体销毁、重连恢复和 dirty-gated checkpoint 已完成；床、死亡、不可破坏 sidecar 墓碑与重生 resolver 已作为服务端后端接入。
