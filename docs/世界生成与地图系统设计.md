# 世界生成与地图系统设计

## 1. 概述与设计目标

"科学与神学"的地图方向为 **3D 方块/体素世界**，并在此基础上扩展为 **体素球形星球 + 外太空多星球系统**。

核心设计目标：

- 方块本身保持全局固定立方体，不随球面局部法线旋转。
- 星球整体是球体或类球体，玩家沿一个方向持续移动可以绕星球一圈回到原点附近。
- 重力方向随玩家位置变化，始终指向当前星球球心。
- 玩家在地面附近看到的是标准方块体素；飞到高空后可以看到地平线弯曲，星球整体呈圆形。
- 支持外太空、飞船、空间站、小行星和多个星球。
- 多个星球之间使用独立局部体素世界，不共享一个超大体素坐标系。

核心架构原则保持不变：**Godot 场景不是世界本身**。权威世界数据在 C++ 核心层管理，Godot 负责把 chunk 数据渲染成 3D 场景、接受输入并提交命令。

当前 3D 起点已经落地：

- `WorldMap.tscn` 根节点为 `Node3D`，玩家为 `CharacterBody3D`，相机为 `Camera3D`。
- `scripts/ChunkRendererBridge.gd` 以 `Node3D + MultiMeshInstance3D + StaticBody3D` 渲染方块 chunk。
- `scripts/PlayerController.gd` 通过 `Camera3D` 射线选择方块，左键挖掘，右键放置/交互。
- `scripts/world/WorldObjectRenderer.gd` 用方块风格的 `BoxMesh` 组合渲染工作台、熔炉等对象。梯子由 `ChunkRendererBridge` 作为地形方块渲染。

## 2. 核心结论

推荐方案是：

```text
宇宙层：非体素大尺度空间
星球层：独立局部体素球
表面层：真实方块 chunk
远景层：球形 LOD mesh
重力层：根据当前位置指向最近或最强星球球心
```

一句话概括：

```text
不要把整个宇宙做成一个无限大的体素网格；
而是把多个"局部体素星球"放在一个普通 3D 宇宙空间里。
```

不推荐使用：

- Cube-Sphere 六面球体素。
- 经纬度体素。
- 六边形球面网格。
- 球坐标楔形体素。
- 每个方块跟随球面法线旋转的局部体素。

当前最适合使用：

```text
全局固定方块网格 + 球形裁剪生成 + 球心重力 + 高空球形远景 LOD + 多星球局部坐标
```

## 3. 世界总体架构

世界分成三层：

```text
Universe
├─ Space Scene
│  ├─ Star / Sun
│  ├─ Planet A LOD
│  ├─ Planet B LOD
│  ├─ Asteroid Field
│  ├─ Space Station
│  └─ Player / Ship
│
├─ Planet A
│  ├─ Local Voxel World
│  ├─ Chunk System
│  ├─ Terrain Generator
│  ├─ Gravity Field
│  ├─ Surface LOD
│  └─ Save Data
│
└─ Planet B
   ├─ Local Voxel World
   ├─ Chunk System
   ├─ Terrain Generator
   ├─ Gravity Field
   ├─ Surface LOD
   └─ Save Data
```

其中：

- **Universe** 负责星球、飞船、太阳、空间站等大尺度对象的位置。
- **Planet** 负责自己的局部体素世界。
- **Voxel World** 只在玩家靠近星球时加载。
- **LOD Mesh** 用于玩家远距离观察星球。

## 4. 方块与星球的关系

### 4.1 方块保持全局固定立方体

每个星球内部使用普通三维体素坐标：

```text
x, y, z
```

方块方向固定：

```text
+X / -X
+Y / -Y
+Z / -Z
```

方块不会根据球面弯曲而旋转。

这意味着：

- 方块仍然是标准立方体。
- 多方块结构检测简单。
- 机器、管道、电线、物流连接规则简单。
- 碰撞可以继续使用 AABB 或常规体素碰撞。
- chunk 存储方式与普通 Minecraft 式体素世界接近。

### 4.2 单个星球的体素生成方式

单个星球可以理解为：

```text
一个普通 3D 方块网格 + 只保留球体范围内的方块
```

判断某个方块是否属于星球：

```cpp
bool inside_planet(Vector3i block_pos, Vector3 planet_center, float planet_radius) {
    Vector3 p = block_pos - planet_center;
    return p.length() <= planet_radius;
}
```

如果加入地形起伏：

```cpp
bool is_solid_block(Vector3 block_pos) {
    Vector3 dir = normalize(block_pos - planet_center);
    float terrain_height = noise(dir) * mountain_height;
    float surface_radius = planet_radius + terrain_height;

    float d = length(block_pos - planet_center);
    return d <= surface_radius;
}
```

这样可以生成：球形星球、山脉、峡谷、洞穴、地下矿脉、地核结构。

## 5. 核心数据模型

### 5.1 目标坐标

目标语义采用体素坐标：

```text
VoxelCoord = (planet_id, x, y, z)
ChunkCoord = (planet_id, cx, cy, cz)
BlockLocal = (lx, ly, lz)
```

其中：

- `x/z` 是水平平面坐标。
- `y` 是高度和深度，地表、地下、矿井、遗迹都在同一 3D 世界中表达。
- 地下不是独立图层，而是 `y < surface_height` 的空间。
- 传送门、梯子、电梯、矿井竖井等只是世界对象或结构，不再承担图层切换职责。

### 5.2 当前 C++ 模型（过渡）

```cpp
class WorldData {
    std::unordered_map<ChunkKey, ChunkData> chunks_;

    const ChunkData* get_chunk(layer_id, cx, cy) const;
    ChunkData* get_chunk(layer_id, cx, cy);
    bool has_chunk(layer_id, cx, cy) const;
    void set_chunk(layer_id, cx, cy, ChunkData);
    void remove_chunk(layer_id, cx, cy);
};

struct ChunkData {
    int chunk_x, chunk_y;
    ChunkState state;
    TerrainData terrain;
    std::vector<EntityId> entities;
    std::vector<EntityId> machines;
    std::vector<EntityId> connector_ids;
};

struct TerrainData {
    int size_x, size_y;
    std::vector<TerrainCell> cells;
};
```

这是 3D 迁移的 P0 数据基线：`TerrainData` 暂时是 X/Z 平面，GDScript 把 cell 渲染为单层方块。

### 5.3 目标体素数据模型

下一阶段升级为体素体：

```cpp
struct VoxelChunkData {
    int chunk_x, chunk_y, chunk_z;
    int size_x, size_y, size_z;
    std::vector<TerrainCell> blocks;
    std::vector<BlockEntityPlacement> block_entities;
};
```

### 5.4 目标 ChunkKey

目标 `ChunkKey` 不再依赖视觉图层：

```cpp
struct ChunkKey {
    int planet_id;
    int chunk_x;
    int chunk_y;
    int chunk_z;
};
```

规则：

- 每个星球有独立的 chunk 坐标空间。
- 地下、深层矿区、遗迹层通过 `chunk_y` 和 biome/zone 规则表达。
- 只有真正不同的星球才使用不同的 `planet_id`。

### 5.5 方块实体

普通方块只保存 material/block id。需要状态的内容使用 block entity：

| 类型 | 示例 | 状态 |
|---|---|---|
| 机器 | 熔炉、粉碎机、电炉、组装机 | 输入/输出槽、进度、能量 |
| 网络节点 | 线缆、管道、ME 线缆、接口 | 连接面、通道、缓存 |
| 交互对象 | 工作台、祭坛、台座 | UI 状态、配方状态 |
| 传送/升降 | 梯子（地形方块，攀爬移动）、电梯、传送门 | 目标位置、锁定条件 |

短期原型仍由 `WorkbenchManager` / `FurnaceManager` 持有少量状态，权威写入必须经过 `GDGameCommandServer`。梯子已迁移为地形方块（`snt:ladder`，`TF_CLIMBABLE`），不再由独立 manager 管理。中期应将工作台和熔炉也下沉为 C++ block entity，便于存档、联机和无头服务端运行。

## 6. 世界生成架构

### 6.1 内容注册边界

世界生成采用"**内容定义在 GD/数据层，确定性执行在 C++ 核心层**"：

- GDScript/资源注册方块材质、掉落、硬度、工具需求、渲染映射、生物群系、矿脉和结构参数。
- C++ 核心负责世界数据、确定性噪声、异步 chunk 生成、存档序列化和服务端权威写入。
- WorkerThreadPool 工作线程只读取不可变 `WorldGenConfigSnapshot`，不调用 GDScript，不访问 Godot Object，不读取可变 Resource。
- 内容 ID 通过稳定 key 冻结为运行时 ID，不再靠追加 C++ enum 扩展玩法内容。

### 6.2 注册流程

```gdscript
var registry := GDTerrainContentRegistry.new()

registry.register_material({
    "id": 1,
    "key": "snt:stone",
    "display_name": "Stone",
    "flags": GDTerrainContentRegistry.FLAG_SOLID | GDTerrainContentRegistry.FLAG_MINEABLE,
    "hardness": 1.5,
    "required_tool_tag": "pickaxe",
    "required_mining_level": 1,
    "drops": [{ "item_key": "dust.stone", "count": 1 }],
})

registry.register_base_terrain_rule({
    "dimension": "overworld",
    "mode": "surface_elevation",
    "default_material_key": "snt:dirt",
})

var frozen_config := registry.freeze()
world_data.worldgen_config = frozen_config
```

当前入口是 `scripts/worldgen/BuiltinTerrainContent.gd`，由 `ChunkRendererBridge.initialize()` 创建默认配置并赋给 `GDWorldData.worldgen_config`。

### 6.3 目标生成 Pass

3D 体素世界的目标生成流程：

```text
generate_chunk(planet_id, cx, cy, cz)
├── pass_base_terrain()      # 地形形状、径向分层、基础地貌
├── pass_rock_layer()        # 岩层区域分配（花岗岩区/玄武岩区/深层岩区）
├── pass_biomes()            # 温度/湿度/高度/深度驱动的生态区
├── pass_ore()               # GT 风格矿脉（受岩层约束）
├── pass_surface_objects()   # 树、碎石、装饰、初级交互对象
└── pass_gameplay()          # 游戏玩法元素
```

当前 C++ `TerrainGenerator` 使用确定性三维噪声与异步生成，直接产生体素体积。

### 6.4 矿脉方向

GT 风格复杂系统适合放进 3D：

- 矿脉按 `x/z` 大尺度分布，按 `y` 限定深度。
- 一个矿脉包含主矿、伴生矿、稀有点矿。
- 深层矿需要更高等级工具、能源设备或魔法辅助。
- AE2 陨石/石英矿、魔法晶簇等可作为结构或特殊矿脉 pass。

示例目标数据：

```gdscript
registry.register_ore_vein_rule({
    "key": "gt:chalcopyrite_vein",
    "planet_id": 0,
    "host_material_key": "snt:stone",
    "primary_ore_key": "snt:ore_copper",
    "secondary_ore_keys": ["snt:ore_iron"],
    "min_y": -48,
    "max_y": 24,
    "radius": 18,
    "density": 0.62,
})
```

## 7. Chunk 系统设计

### 7.1 星球局部 chunk 坐标

每个星球内部使用独立 chunk 坐标：

```cpp
struct ChunkCoord {
    int planet_id;
    int cx;
    int cy;
    int cz;
};
```

每个 chunk 建议初期使用：

```text
16 × 16 × 16
```

原因：

- 实现简单。
- 方便调试。
- mesh 重建成本较低。
- 适合小型独立游戏原型。

后续可按性能调优为 `32 × 32 × 32`。

### 7.2 尺寸约定

| 项 | 当前值 | 说明 |
|---|---:|---|
| 方块尺寸 | `1.0` Godot unit | `ChunkRendererBridge.BLOCK_SIZE` |
| 当前 chunk 平面尺寸 | `32 x 32` | 继承旧 `TerrainData`，用于 X/Z 平面 |
| 目标 chunk 尺寸 | `16 x 16 x 16` | 初期推荐值，后续按性能调优 |
| 当前玩家触达 | `6.0` unit | `PlayerController.REACH` |
| 当前可视半径 | `view_radius = 4` chunk | 由 `ChunkRendererBridge` 控制 |
| 当前加载半径 | `loaded_radius = 5` chunk | 生成/模拟半径应大于可视半径 |

### 7.3 只加载玩家附近的 chunk

不应该一次生成整个星球的体素。

例如半径为 1024 方块的星球，体素空间体积大约是：

```text
4/3 × π × 1024³ ≈ 45 亿个体素位置
```

所以必须按需加载：

```text
玩家附近：真实体素 chunk
中距离：简化地表 LOD
远距离：球形星球 LOD
```

推荐加载范围：

```text
玩家周围 8 ~ 16 个 chunk 半径：真实体素
更远区域：LOD mesh
整个星球轮廓：低模球体
```

## 8. 重力系统设计

### 8.1 单星球重力

玩家在星球附近时，重力方向指向球心：

```cpp
Vector3 gravity_dir = normalize(planet_center - player_position);
```

玩家的上方向为：

```cpp
Vector3 player_up = -gravity_dir;
```

玩家的下方向为：

```cpp
Vector3 player_down = gravity_dir;
```

角色控制器需要支持动态 up 方向。

这意味着玩家绕到星球另一侧时，角色会相对于宇宙坐标倒过来，但在玩家视角中仍然是脚踩地面。

### 8.2 多星球重力

多星球情况下，需要判断当前由哪个星球提供主要重力。

推荐初期使用最近星球重力或最强星球重力：

```cpp
Planet* find_gravity_source(Vector3d player_pos) {
    Planet* best = nullptr;
    double best_strength = 0.0;

    for (Planet& planet : planets) {
        double d = distance(player_pos, planet.universe_position);

        if (d < planet.gravity_radius) {
            double strength = planet.mass / (d * d);

            if (strength > best_strength) {
                best_strength = strength;
                best = &planet;
            }
        }
    }

    return best;
}
```

然后：

```cpp
if (Planet* p = find_gravity_source(player_position)) {
    gravity_dir = normalize(p->universe_position - player_position);
} else {
    gravity_dir = Vector3(0, 0, 0); // 外太空无重力
}
```

### 8.3 不建议初期做真实 N 体引力

真实 N 体引力会引入很多问题：轨道调试复杂、飞船控制复杂、玩家容易被多个星球拉扯、浮点误差更明显、游戏性未必更好。

初期建议：

```text
星球附近：使用该星球球心重力
星球之间：无重力或弱重力
飞船：使用简化推进系统
```

## 9. 外太空设计

外太空不需要体素化。

外太空是一个普通三维空间：

```text
玩家位置
飞船位置
星球位置
太阳位置
空间站位置
小行星位置
```

只有当玩家靠近某个星球时，才加载该星球的体素系统。

流程如下：

```text
远离星球：
  显示星球低模球体

进入引力范围：
  启用星球重力

进入近地轨道：
  显示地表 LOD

进入低空：
  加载真实体素 chunk

落地：
  使用标准方块体素世界
```

## 10. 多星球设计

### 10.1 每个星球使用独立局部坐标

不要让整个宇宙所有星球共用一个体素网格。

错误设计：

```text
整个宇宙 = 一个超大 voxel[x][y][z]
```

正确设计：

```text
Universe Space = 大尺度非体素坐标
Planet A = 自己的 voxel[x][y][z]
Planet B = 自己的 voxel[x][y][z]
Planet C = 自己的 voxel[x][y][z]
```

示例结构：

```cpp
struct Planet {
    int planet_id;
    Vector3d universe_position;
    double radius;
    double gravity_radius;
    double mass;

    VoxelWorld voxel_world;
    PlanetGenerator generator;
    PlanetLodRenderer lod_renderer;
};
```

体素坐标：

```cpp
struct VoxelCoord {
    int planet_id;
    int x;
    int y;
    int z;
};
```

### 10.2 多星球切换流程

玩家从星球 A 起飞：

```text
星球 A 地表真实体素
        ↓
高空真实 chunk + 地表 LOD
        ↓
星球 A 远景球体
        ↓
外太空
        ↓
接近星球 B
        ↓
星球 B 远景球体
        ↓
星球 B 地表 LOD
        ↓
星球 B 真实体素 chunk
        ↓
降落到星球 B 地表
```

这个过程可以做成完全无缝、半无缝、用加载动画过渡或用飞船降落动画过渡。

独立游戏初期推荐半无缝加载：

- 靠近星球时预加载。
- 离开星球时卸载。
- 中间用星球 LOD 遮盖细节变化。

## 11. 高空看到圆形地面与 LOD 设计

### 11.1 实现条件

要让玩家飞高后看到地面是圆的，需要三个条件：

```text
星球半径不能过大
玩家视距足够远
远处地形使用球形 LOD
```

如果星球太大，飞几百米也几乎看不到曲率。

游戏中建议使用小星球尺寸：

```text
小行星：半径 128 ~ 512 方块
小星球：半径 1024 ~ 4096 方块
大星球：半径 8192+ 方块
```

如果希望玩家飞高一点就明显看到圆，可以从半径 512 ~ 2048 方块开始测试。

### 11.2 推荐 LOD 层级

```text
LOD 0：真实方块 chunk
LOD 1：简化 chunk mesh
LOD 2：星球地表低精度 mesh
LOD 3：完整星球低模球体
LOD 4：远处贴图 / billboard
```

示例：

```text
距离 0 ~ 200m：
  真实方块 chunk

距离 200 ~ 1000m：
  简化 chunk mesh

距离 1000m 以上：
  球形星球代理 mesh

太空视角：
  星球 LOD + 大气 + 云层
```

### 11.3 近处和远处的区别

近处：

```text
真实方块、真实碰撞、真实挖掘、真实建造、真实机器
```

远处：

```text
只显示外观、不参与精细碰撞、不参与机器逻辑、不保存单个方块细节
```

## 12. 坐标精度问题

外太空尺度通常很大，不能只使用 float。

否则玩家远离宇宙原点后，容易出现：模型抖动、相机闪烁、碰撞不稳定、方块错位、远处物体跳动。

推荐使用 Floating Origin / 浮动原点：

```text
宇宙逻辑坐标：double
局部渲染坐标：float
```

逻辑坐标：

```cpp
Vector3d universe_position;
```

渲染坐标：

```cpp
Vector3 render_position = object.universe_position - camera_origin;
```

玩家附近永远保持在原点附近：

```text
玩家局部渲染位置 ≈ (0, 0, 0)
```

这样可以减少精度问题。

## 13. 3D 渲染与交互

### 13.1 当前 ChunkRendererBridge3D

`scripts/ChunkRendererBridge.gd` 当前职责：

- 创建或接收 `GDWorldData`。
- 构建默认 `GDWorldGenConfig`。
- 按玩家所在 chunk 请求异步生成。
- 把可视 chunk 中非空气 cell 按 material 分组。
- 每种 material 创建一个 `MultiMeshInstance3D`，每个方块一个 instance。
- 为每个实体方块创建 `StaticBody3D + CollisionShape3D`，供玩家碰撞和射线检测。
- 按预算限制每帧请求数量和 chunk view 构建数量。
- 提供 `world_position_to_cell()`、`cell_to_world_position()`、`get_cell_info()`、`refresh_cell()` 等接口。

当前方案简单但正确，适合验证 3D 交互；后续需要优化碰撞和网格生成，否则大地图会产生过多碰撞节点。

### 13.2 渲染优化路线

| 阶段 | 目标 | 说明 |
|---|---|---|
| R0 | MultiMesh 方块渲染 | 早期实现，便于快速迭代 |
| R1 | 只渲染可见面 | ✅ 已完成（`build_greedy_mesh` 中 face culling） |
| R2 | greedy meshing | ✅ 已完成（`GDChunkHelper::build_greedy_mesh` 合并同材质相邻面） |
| R3 | chunk 级碰撞合并 | ✅ 已完成（`ConcavePolygonShape3D`，每 chunk 一个 StaticBody3D） |
| R4 | 方块 atlas 材质 | ✅ 已完成（`BlockAtlasBuilder` 按维度运行时构建共享 atlas，`terrain_block.gdshader` 改为 atlas 采样，见 V30） |
| R5 | 区块 LOD/远景 | 🔲 待实现（远处 chunk 降面数或用 impostor，见 V34） |

当前材质 ID 仍使用内置原型表：

| ID | key | 含义 |
|---:|---|---|
| 0 | `snt:air` | 空气 |
| 1 | `snt:stone` | 石头 |
| 2 | `snt:dirt` | 泥土 |
| 3 | `snt:sand` | 沙子 |
| 4 | `snt:water` | 水 |
| 5 | `snt:lava` | 熔岩 |
| 6 | `snt:ore_iron` | 铁矿 |
| 7 | `snt:ore_copper` | 铜矿 |
| 8 | `snt:ore_coal` | 煤矿 |
| 9 | `snt:wood` | 木头 |

### 13.4 世界对象渲染

`WorldObjectRenderer.gd` 当前用 `BoxMesh` 组合临时模型：

- 工作台：主体、桌面、桌腿。
- 熔炉：石质方块、炉口、发热点。
- 梯子：薄面板 mesh，根据贴墙方向旋转（由 `ChunkRendererBridge` 渲染）。

这是类 MC 风格的正确起点。后续应把这类对象统一为 block model 资源：同一个 block id 对应一个碰撞盒、渲染模型、交互面、UI 入口和存档状态。

## 14. 玩家控制器设计

玩家控制器需要支持动态重力方向。

普通角色控制器通常假设：

```text
up = 世界 Y+
down = 世界 Y-
```

但球形星球需要：

```cpp
down = normalize(planet_center - player_position);
up = -down;
```

移动方向需要投影到当前切平面：

```cpp
Vector3 move_dir_on_surface = project_on_plane(input_dir, up);
```

角色姿态需要逐渐对齐：

```text
当前 up 方向 -> 目标 up 方向
```

不要瞬间旋转，否则容易眩晕。

推荐：

```text
角色身体快速对齐
相机稍微平滑对齐
飞船模式和步行模式分开处理
```

当前 `scripts/PlayerController.gd` 为 `CharacterBody3D`：

- 鼠标控制 yaw/pitch，`Camera3D` 位于 `Head/Camera3D`。
- WASD 移动，Shift 冲刺，Space 跳跃。
- 每帧从相机向前发射射线，命中方块后计算目标 cell 和可放置 cell。
- 左键提交 `mine_block` 命令。
- 右键优先打开/交互对象，再尝试放置工作台、熔炉、梯子等方块对象。
- 命令由 `GDGameCommandServer` 校验并写入世界/背包，再通过同步信号刷新渲染和 UI。

## 15. 命令与同步

3D 交互仍遵守 server-authoritative 边界：

```text
PlayerController3D / UI
    └── submit_command(command)
            ↓
GDGameCommandServer
    ├── 校验距离、工具、目标方块、背包、放置冲突
    ├── 写入 GDWorldData / Inventory / world managers
    └── emit sync signals
            ↓
ChunkRendererBridge3D / UI
    ├── on_terrain_cell_synced()
    └── inventory_synced / world_object_synced / furnace_synced
```

当前命令：

| Command | 当前来源 | 服务端写入 | 同步 |
|---|---|---|---|
| `mine_block` | `PlayerController` 左键射线 | `GDWorldData.set_terrain_cell`、掉落入背包 | `terrain_cell_synced`、`inventory_synced` |
| `place_object` | `PlayerController` 右键 | 工作台/熔炉 manager、梯子地形方块、背包消耗 | `world_object_synced` / `terrain_cell_synced`、`inventory_synced` |
| `craft_recipe` | `CraftingUI` | 背包输入/输出 | `inventory_synced` |
| `furnace_take_output` | `FurnaceUI` | 熔炉输出、玩家背包 | `furnace_synced`、`inventory_synced` |
| `furnace_insert_input` | `FurnaceUI` | 背包消耗、熔炉输入槽 | `furnace_synced`、`inventory_synced` |
| `furnace_insert_fuel` | `FurnaceUI` | 背包消耗、熔炉燃料槽 | `furnace_synced`、`inventory_synced` |

命令拒绝必须一次性输出有用 warning，并发 `command_rejected`。遇到无法判断的问题，优先加低频日志，让用户运行后提供日志。

## 16. 存档设计

### 16.1 目标存档结构

每个星球单独存档：

```text
save/
├─ universe.json
├─ planets/
│  ├─ planet_001/
│  │  ├─ meta.json
│  │  ├─ chunks/
│  │  └─ edits/
│  ├─ planet_002/
│  │  ├─ meta.json
│  │  ├─ chunks/
│  │  └─ edits/
│  └─ planet_003/
│     ├─ meta.json
│     ├─ chunks/
│     └─ edits/
```

`universe.json` 保存：

- 星球 ID。
- 星球位置。
- 星球半径。
- 星球种子。
- 星球类型。
- 轨道信息。
- 空间站信息。

`planet_xxx/meta.json` 保存：

- 星球生成参数。
- 星球重力参数。
- 星球资源参数。
- 星球生态参数。

`chunks/` 保存已生成或被修改的 chunk。

`edits/` 可以只保存玩家修改过的方块，未修改区域由种子重新生成。

### 16.2 当前格式（过渡）

当前 `SaveManager` / `ChunkSerializer` 保存的是 legacy chunk：

```text
regions/{layer}~{region_x}~{region_y}.region
```

chunk 内保存三维 `TerrainData`、实体、机器和 connector id 等字段。

### 16.3 目标格式

目标区域文件应按 3D chunk 分组：

```text
regions/{planet_id}~{region_x}~{region_y}~{region_z}.region
```

header 需要包含：

```text
world_seed
worldgen_schema_version
worldgen_content_hash
block_material_table: [runtime_id, key]
planet_id
chunk_size_x/y/z
```

加载旧存档时：

1. 读取旧 `layer`。
2. 将 `surface` 映射到 `planet_id=0, chunk_y=0`。
3. 如果遇到旧 `underground`，仅作为迁移输入映射到较低 `chunk_y`，不再恢复为双层玩法。
4. 按 material key 重映射 ID，缺失内容时使用 `snt:unknown_solid` 或拒绝加载并输出一次性错误。

## 17. 美术与资产规格

3D 方块风格降低了早期素材压力，但仍要保持规格：

- 方块基础贴图可以从 `16x16` 或 `32x32` 开始，后续可升到 `64x64`。
- 物品 icon 仍按项目规范导出到 `64x64` 透明画布，UI 按槽位缩放显示。
- 方块贴图与物品 icon 不强行同尺寸；世界方块、机器模型、背包图标是不同资产规格。
- 机器/工作台/祭坛等对象优先用块状模型组合，复杂机器再升级为专用低模。
- 同一种 material 应有稳定 key，世界渲染、掉落、百科和 UI 共用同一内容定义。

当前纯色材质只是占位。进入 R4 后，应新增：

```text
resource/blocks/
├── terrain/
├── ores/
├── machine/
└── magic/
```

## 20. GDScript 用法示例

```gdscript
var world := GDWorldData.new()
world.seed = randi()
world.worldgen_config = BuiltinTerrainContent.create_default_config()
```

```gdscript
world.request_chunk_async("overworld", 0, 0, 0)

func _process(_delta: float) -> void:
    world.process_async_results()
```

### 查询和修改方块

```gdscript
var cell := Vector3i(12, 4, -4)
var chunk := chunk_bridge.cell_to_chunk(cell)
var local := chunk_bridge.cell_to_local(cell)
var data := world.get_terrain_cell(
    "overworld", chunk.x, chunk.y, chunk.z, local.x, local.y, local.z)

command_server.submit_command({
    "type": GameCommandServer.COMMAND_MINE_BLOCK,
    "dimension": &"overworld",
    "chunk": chunk,
    "local": local,
    "cell": cell,
    "expected_material": int(data.get("material", 0)),
})
```

## 22. 最终总结

当前目标最推荐的方案是：

```text
全局固定方块
+
球形体素星球
+
星球局部坐标
+
外太空非体素空间
+
多星球独立体素世界
+
远景 LOD 球体
+
球心重力
```

这套方案的优点是：

- 符合方块体素视觉。
- 保留普通体素世界的简单规则。
- 高空能看到圆形星球。
- 可以支持绕星球行走。
- 可以支持外太空。
- 可以支持多个星球。
- 可以支持飞船和星际旅行。
- 不需要复杂的 Cube-Sphere 面边界映射。
- 不需要让方块随球面旋转。

最终可以理解为：

```text
每个星球都是一个从普通 3D 方块网格中"挖出来"的球。
宇宙只是摆放这些球体和飞船的普通大尺度空间。
```

## 23. 岩层与方块重力系统设计

### 23.1 岩层系统（RockLayerRule）

岩层系统参考群峦传说（TerraFirmaCraft），将地下岩石按区域划分为不同岩层。不同岩层有不同的硬度、坍塌概率和伴生矿石。

**核心数据结构：**

```cpp
struct RockLayerRule {
    std::string key;
    std::string dimension_id;
    TerrainMaterialId rock_material;   // 岩层方块材质
    float noise_scale;                 // 噪声区域选择
    int noise_octaves;
    float noise_min, noise_max;        // 噪声范围
    float depth_min, depth_max;        // 深度范围
    float hardness_multiplier;         // 硬度倍率
    float collapse_chance;             // 坍塌概率
    std::vector<TerrainMaterialId> associated_ores;  // 伴生矿石
};
```

**生成流程：**

`pass_rock_layer` 在 `pass_base_terrain` 之后运行，根据球面方向噪声和深度选择岩层材质。星球使用球面方向作为噪声输入，确保同一地表区域的岩层延伸到地下。

**矿脉关联：**

`pass_ore` 在有岩层规则时，检查当前方块所属岩层的 `associated_ores` 列表。只有岩层允许的矿石才能在该区域生成。例如花岗岩区可以生成铁矿+铜矿，玄武岩区只能生成铁矿。

### 23.2 方块重力标志

| 标志 | 值 | 含义 | 示例 |
|---|---|---|---|
| `TF_GRAVITY_FALL` | 1 << 6 | 下方为空时立即下落 | 沙子、砂砾 |
| `TF_COLLAPSE_RISK` | 1 << 7 | 失去支撑时概率坍塌 | 石头、深层岩 |
| `TF_SUPPORT_BEAM` | 1 << 8 | 阻止附近坍塌 | 木支撑、钢支撑 |

### 23.3 运行时玩法配置（GameplayConfig）

`GameplayConfig` 独立于 `WorldGenConfigSnapshot`，可以在运行时修改：

```gdscript
# 全局配置
world_data.set_gameplay_config({
    "enable_collapse": true,          # 坍塌总开关
    "collapse_chance_multiplier": 1.0, # 坍塌概率倍率
    "max_collapse_chain": 64,         # 最大连锁坍塌数
    "support_beam_radius": 5,         # 支撑梁范围
    "enable_gravity_fall": true,      # 落沙总开关
    "max_gravity_fall_chain": 64,     # 最大连锁落沙数
})

# 每星球覆盖
var config = world_data.get_gameplay_config()
config["planet_overrides"] = {
    "overworld": {
        "has_enable_collapse": true,
        "enable_collapse": false,  # 关闭 overworld 坍塌
    }
}
world_data.set_gameplay_config(config)
```

### 23.4 方块物理系统（BlockPhysicsSystem）

`BlockPhysicsSystem` 作为 `SimulationSystem` 子系统，优先级 0（在机器系统之前执行）。

**数据流：**

```text
玩家挖掘/放置方块
    → GDGameCommandServer 推入 BlockPhysicsEvent 到 WorldData
    → BlockPhysicsSystem.tick_active() 消费事件
    → 调度 PendingCheck（延迟 tick 执行）
    → process_gravity_fall() / process_collapse()
    → 连锁反应：schedule_gravity_fall_after_mine() / schedule_collapse_after_mine()
```

**落沙逻辑：**

1. 检查方块是否为 `TF_GRAVITY_FALL`
2. 计算重力方向（球心重力或默认 Y-）
3. 检查重力方向下一格是否为空/液体
4. 移动方块到下一格

**坍塌逻辑：**

1. 检查方块是否为 `TF_COLLAPSE_RISK`
2. 检查附近是否有 `TF_SUPPORT_BEAM`（曼哈顿距离 ≤ `support_beam_radius`）
3. 检查重力方向下一格是否有实心支撑
4. 按概率坍塌：`collapse_chance × collapse_chance_multiplier`

**球心重力适配：**

`compute_gravity_step()` 根据方块到球心的向量，选择绝对值最大的轴作为重力方向，返回体素步进向量。平坦世界返回 `(0, -1, 0)`。

### 23.5 GDScript 用法

```gdscript
# 注册方块物理子系统
tick_system.register_block_physics_system()

# 查询玩法配置
var config = world_data.get_gameplay_config_for_dimension("overworld")
print("collapse enabled: ", config["enable_collapse"])
print("gravity fall enabled: ", config["enable_gravity_fall"])

# 运行时关闭坍塌
world_data.set_gameplay_config({"enable_collapse": false})
```
