# 世界生成与地图系统设计

## 1. 概述

"科学与神学"的世界采用**双层互通**设计：Surface（地表）和 Underground（地下）通过 Connector 相连。世界数据在 C++ 核心层管理（`WorldData` / `ChunkData` / `TerrainData`），Godot GDScript 层负责渲染和交互。

核心架构原则：**Godot 不是世界本身**，TileMapLayer 只是 `ChunkData` 的可视化代理。

## 2. 图层系统

### 2.1 当前图层（2 层）

| 图层 ID | 名称 | 描述 |
|---------|------|------|
| `surface` | 地表 | 开放世界，有草地、树木、水域、建筑 |
| `underground` | 地下 | 洞穴系统，石壁、矿脉、遗迹 |

### 2.2 坐标系统

统一全局坐标系：`layer_id(StringName) + cell_position(Vector2i)`。

Chunk 坐标 = `{layer_id, chunk_x, chunk_y}`，Chunk 大小 = 32×32 cells。

在 GDScript 中通过 `MapLayerController` 管理图层切换。

### 2.3 图层设计决策

> 当前为双层结构：`surface`（地表）和 `underground`（地下）。地下层内部通过垂直深度区分不同区域（浅层洞穴、深层矿井、废墟层等），不拆分为独立图层。所有内容（遗迹、工厂、矿井）位于同一 `underground` 层中，通过 ChunkData 的附加数据（depth_band、zone_type 等）区分。
>
> C++ 端 `TerrainGenerator::generate_chunk()` 对未知层有 `else { fill with stone }` 回退保护。

## 4. 地形生成流水线（TerrainGenerator）

### 4.1 六阶段生成

`TerrainGenerator` 调用 6 个 Pass，每个 Pass 使用独立种子的 `NoiseGenerator`，天然线程安全。

```
generate_chunk(layer_id, cx, cy)
├── 1. pass_base_terrain()    ✅ 已实现
├── 2. pass_biome()           ✅ 已实现
├── 3. pass_ore()             ✅ 已实现
├── 4. pass_structure()       ✅ 已实现
├── 5. pass_object()          ✅ 已实现
└── 6. pass_gameplay()        ✅ 已实现
```

### 4.2 Pass 1: 基础地形（已实现）

**Surface 层**：
- 使用 elevation noise（`scale=0.02, octaves=4`）区分地表高度
- 顶部行：DIRT（可通行），进一步用 detail noise 添加 SAND 斑块
- 下部行：STONE

**Underground 层**：
- 使用 cave noise（`scale=0.04, octaves=4`）生成洞穴
- 洞穴内：AIR（可通行），边缘区域保持 STONE

### 4.3 Pass 2: 生物群系（已实现）

使用两个大尺度噪声 **temperature**（温度）和 **humidity**（湿度）驱动生物群系覆盖 (`scale=0.015~0.02`)：

| 条件 | Surface 行为 | Underground 行为 |
|------|-------------|-----------------|
| 热 + 干 | DIRT → **SAND**（沙漠） | — |
| 近水（2 格内） | DIRT → **SAND**（沙滩） | — |
| 冷 + 干 | DIRT → **STONE**（石质高地） | — |
| 温 + 湿 | 保持 DIRT（沃土） | AIR → **DIRT**（洞穴泥地） |
| 干 + 冷洞穴 | — | AIR → **SAND**（干燥洞穴沙地） |

温度/湿度各用独立 sub-seed 的 NoiseGenerator，确保世界种子下的确定性分布。

### 4.4 Pass 3: 矿石生成（已实现）

使用两组 noise：
- `ore_noise`（`scale=0.08, octaves=3`）：矿脉密度
- `vein_noise`（`scale=0.15, octaves=2`）：矿脉形状

在 STONE 格中放置：
- ORE_IRON（铁矿石）：密度 > 0.6 且形状 > 0.4
- ORE_COPPER（铜矿石）：密度 > 0.4 且形状 > 0.6
- ORE_COAL（煤矿石）：密度 > 0.7 且形状 > 0.3

### 4.5 Pass 4: 大型结构（已实现）

使用 `density` + `shape` 双噪声驱动结构生成：

| 特征 | Surface | Underground |
|------|---------|-------------|
| **石山** | 圆形 STONE 隆起（半径 2-6） | — |
| **悬崖** | 水边 DIRT→STONE 垂直面 | — |
| **大厅** | — | 扩大 AIR 洞穴（半径 2-7） |
| **熔岩池** | — | 洞穴底部 AIR→**LAVA** |
| **地下湖** | — | 洞穴底部 AIR→**WATER** |

每个特征使用独立的 sub-seed，`density` 阈值 >0.55，`shape` 控制半径/位置偏移。

### 4.6 Pass 5: 小型物体（已实现）

使用 `density` 噪声 + 随机种子偏移确定物体位置：

| 物体 | Surface | Underground |
|------|---------|-------------|
| **碎石堆** | DIRT/SAND→STONE 散落 1-4 格 | — |
| **石环遗迹** | 水边 8 格 STONE 环（中心可通行，半径 4） | — |
| **钟乳石柱** | — | 顶到底 STONE 填充（2-4 格高） |
| **额外矿簇** | — | 已有矿脉旁追加邻近 ORE 格 |

所有物体使用 `WorldSeed.chunk_seed(PASS_OBJECT, cx, cy)` 派生的确定性位置。Pass 5 直接修改地形材质（STONE 碎石/石柱），而非添加实体数据。

### 4.7 Pass 6: Gameplay 元素（已实现）

使用 placement noise（`scale=0.15, octaves=1`）确定 Connector 位置：

| Connector 类型 | 触发层 | 激活方式 | 方向 |
|---|---|---|---|
| Cave Entrance | Surface | 交互（E 键） | 双向 |
| Rift | Surface | 自动（经过即触发） | 单向（Surface→Underground） |
| Ruin Gate | Underground | 交互（E 键） | 双向，初始锁定 |

每个 Connector 有确定性的 `connector_id`（由 `WorldSeed` 的 seed 推导），放置位置记录在 `ChunkData.connectors` 中。

### 4.8 跨层联动生成 [已实现]

Surface ↔ Underground 使用确定性噪声对齐，无需实际交换 ChunkData：

| 方向 | 效果 | 实现位置 |
|------|------|----------|
| Surface → Underground | 地表洞穴入口下方 → 地下入口大厅（清除石头、铺设泥地） | `pass_structure` underground 分支 |
| Underground → Surface | 大型地下洞穴上方 → 地表塌陷坑（石环 + 空气凹陷） | `pass_structure` surface 分支 |

**原理**：在 `pass_structure` 中加入 `cross_noise`，使用 `STRUCTURE + 10` sub-seed 生成。地表和地下在相同 chunk 坐标下计算相同的噪声值，自动对齐特征位置，无需跨层数据传递。两个效果使用不同的噪声偏移（+11000 / +13000 global cell offset），避免重叠。

## 5. 噪声生成器（NoiseGenerator）

### 5.1 当前实现

自定义 Perlin 分形噪声（~140 行 C++）：
- 256 字节排列表（Fisher-Yates 洗牌、确定性种子）
- Perlin 改善衰减曲线：`6t⁵ − 15t⁴ + 10t³`
- 分形八度叠加（可配置 octaves / persistence）
- 返回值范围 [-1, 1]

### 5.2 保留原因

- 纯 C++ 栈分配，**零堆内存分配**
- **天然线程安全**：每个 `generate_chunk()` 调用在栈上构造独立实例
- 虚函数开销为零（inline 方法）
- 与 Godot `FastNoiseLite`（Object/RefCounted 子类）相比，在 `WorkerThreadPool` 工作线程中使用 Godot Object 存在线程安全风险

> 将来若需要 Cellular / DomainWarp 等高级噪声，可引入独立 C++ 头文件 [FastNoiseLite](https://github.com/Auburn/FastNoiseLite)（非 Godot 绑定版本），直接嵌入核心层，避免 Godot Object 依赖。

## 6. 异步 Chunk 生成

### 6.1 架构

使用 Godot 内置 `WorkerThreadPool::get_singleton()` 单例：

```
GDScript __process__ → GDWorldData.process_async_results()
GDScript → GDWorldData.request_chunk_async()
  └── WorkerThreadPool::add_native_task()
      └── 工作线程 → _async_chunk_callback()
          ├── TerrainGenerator::generate_chunk()  # 栈上 NoiseGenerator
          └── mutex → async_results_.push(result)
```

### 6.2 GDWorldData 异步接口

```cpp
class GDWorldData : public godot::Resource {
    // 非阻塞：提交 chunk 生成任务
    void request_chunk_async(layer_id, chunk_x, chunk_y);

    // 每帧调用：消费已完成的结果，emit chunk_ready 信号
    int64_t process_async_results();

    int64_t get_async_pending_count() const;  // 排队中 + 运行中
    int64_t get_worker_thread_count() const;  // CPU 核心数
    int64_t get_async_completed_count() const;
    int64_t get_max_async_queue_size() const; // 默认 256
};
```

### 6.3 容量控制

`max_async_queue_size`（默认 256）：当 `get_async_pending_count() >= max` 时，`request_chunk_async()` 静默丢弃新请求，防止无界队列增长。

### 6.4 线程安全保证

- `TerrainGenerator::generate_chunk()` 只读 `world_seed_`（const），其余栈上分配
- `NoiseGenerator` 在 `generate_chunk()` 内构造独立 `perm_` 数组
- 结果通过 `std::mutex` 保护的 `std::queue` 回传主线程
- `emit_signal("chunk_ready")` 仅在主线程的 `process_async_results()` 中调用

### 10.4 区域文件格式 [已实现]

> 类似 Minecraft Region File 的 `"chunk_x.chunk_y.region"` 格式（32×32 chunk 分组）。每个区域文件最多包含 1024 个 chunk，大幅减少文件系统开销，提升批量加载 I/O 局部性。
>
> **格式 (v1):**
> - 目录：`regions/{layer}~{region_x}~{region_y}.region`

> - 文件结构：Header（version, region_x, region_y, layer_id）+ Index（stored_count, 每个 chunk 的 local_x, local_y, data_size）+ Data（序列化 ChunkData 数据块）

## 11. Chunk 渲染桥接层 [已实现]

三层架构：**C++ GDExtension 绑定层**（GDChunkManager / GDChunkView） + **GDScript 集成桥**（ChunkRendererBridge） + **材质映射**（TerrainMaterialPalette）。

### 11.1 三层 Chunk 模型

| 层级 | 半径 | 行为 |
|------|------|------|
| **Loaded**（模拟层） | `loaded_radius`（默认 10） | ChunkData 保留在 WorldData 内存中，机器/网络继续模拟 |
| **Visible**（渲染层） | `view_radius`（默认 6） | 创建 GDChunkView（TileMapLayer 子节点）用于视觉渲染 |
| **Force-loaded**（强制加载） | 指定 chunk 列表 | 无视玩家距离，始终保留在内存和模拟中（工厂机器 chunk） |

要求：`loaded_radius ≥ view_radius`。`unload_distant_chunks` 默认为 false（所有生成的 chunk 保留内存）。

### 11.2 GDChunkManager（C++ GDExtension 绑定）

`Node` 子类，管理所有 chunk 的三层生命周期：
- **Player 追踪**：`set_player_chunk(layer, cx, cy)` 每帧更新玩家位置
- **刷新**：`refresh_chunks()` 对比三层半径，卸载/隐藏/显示/加载 chunk
- **强制加载**：`force_load_chunk()` / `force_unload_chunk()` 管理工厂 chunk
- **信号连接**：`on_chunk_ready(layer, cx, cy)` 连接到 `GDWorldData.chunk_ready`
- **层级容器**：通过 `surface_container_path` / `underground_container_path` NodePath 定位 TileMapLayer 父节点

### 11.3 GDChunkView（C++ GDExtension 绑定）

`TileMapLayer` 子类，负责单个 chunk 的视觉渲染：
- `update_from_terrain(terrain_dict)`：将 TerrainData（PackedByteArray materials + PackedInt32Array flags）绘制到 TileMapLayer
- `clear_chunk()`：清除所有 tile
- 内部使用 `TerrainMaterialPalette` 将 `TerrainMaterial` 枚举映射为 tileset 坐标

### 11.4 TerrainMaterialPalette（C++ 静态库）

### 11.5 ChunkRendererBridge（GDScript 集成桥）

`scripts/ChunkRendererBridge.gd`，`class_name ChunkRendererBridge`，`extends Node`：

```
ChunkRendererBridge (Node)
├── @export world_data: GDWorldData
├── @export chunk_manager: GDChunkManager
├── @export surface_container_path: NodePath → "Layers/SurfaceLayer"
├── @export underground_container_path: NodePath → "Layers/UndergroundLayer"
├── @export seed / loaded_radius / view_radius / tile_size
│
├── initialize()        — 创建/连接 GDWorldData + GDChunkManager
├── process_frame()     — 每帧调用：process_async_results + set_player_chunk + refresh_chunks
├── force_load_factory_chunk() — 强制加载工厂 chunk
├── release_factory_chunk()    — 释放强制加载
└── unload_all()        — 卸载所有 chunk
```

`_ready()` 中 `call_deferred("initialize")`，`_generate_initial_chunks()` 生成 `start_chunk_radius`（默认 2）范围内所有 surface + underground chunk。

## 18. GDScript 用法示例

### 异步 Chunk 加载

```gdscript
var world = GDWorldData.new()
world.seed = randi()
world.max_async_queue_size = 256

func _process(delta):
    # 消费已完成的异步生成结果（emit chunk_ready 信号）
    var completed = world.process_async_results()

# 请求异步生成
world.request_chunk_async("surface", 0, 0)
world.request_chunk_async("underground", 0, 0)

# 同步生成（立即返回）
var terrain = world.get_or_generate_chunk("surface", 0, 0)
# terrain = {size_x: 32, size_y: 32, materials: PackedByteArray, flags: PackedInt32Array}
```

### 存档保存/加载

```gdscript
var saved_count = world.save_world("user://saves/world_001")
var loaded_seed = world.load_world("user://saves/world_001")
var saves = GDWorldData.list_saves("user://saves")
```
