# 世界生成与地图系统设计

## 1. 概述

"科学与神学"的世界采用**双层互通**设计：Surface（地表）和 Underground（地下）通过 Connector 相连。世界数据在 C++ 核心层管理（`WorldData` / `ChunkData` / `TerrainData`），Godot GDScript 层负责渲染和交互。

核心架构原则：**Godot 不是世界本身**，TileMapLayer 只是 `ChunkData` 的可视化代理。

## 2. 图层系统

### 2.1 当前图层（2 层）

| 图层 ID | 名称 | 描述 |
|---------|------|------|
| `surface` | 地表 | 开放世界，有草地、树木、水域、建筑 |
| `underground` | 地下 | 洞穴系统，石壁、矿脉、遗迹 |

### 2.2 坐标系统

统一全局坐标系：`layer_id(StringName) + cell_position(Vector2i)`。

Chunk 坐标 = `{layer_id, chunk_x, chunk_y}`，Chunk 大小 = 32×32 cells。

在 GDScript 中通过 `MapLayerController` 管理图层切换。

### 2.3 图层设计决策

> 当前为双层结构：`surface`（地表）和 `underground`（地下）。地下层内部通过垂直深度区分不同区域（浅层洞穴、深层矿井、废墟层等），不拆分为独立图层。所有内容（遗迹、工厂、矿井）位于同一 `underground` 层中，通过 ChunkData 的附加数据（depth_band、zone_type 等）区分。
>
> C++ 端 `TerrainGenerator::generate_chunk()` 对未知层有 `else { fill with stone }` 回退保护。

## 3. 世界数据模型（C++ 核心）

### 3.1 WorldData

```cpp
class WorldData {
    std::unordered_map<ChunkKey, ChunkData> chunks_;

    const ChunkData* get_chunk(layer_id, cx, cy) const;
    ChunkData* get_chunk(layer_id, cx, cy);
    bool has_chunk(layer_id, cx, cy) const;
    void set_chunk(layer_id, cx, cy, ChunkData);
    void remove_chunk(layer_id, cx, cy);
    size_t chunk_count() const;
};
```

### 3.2 ChunkData（32×32 格区块）

```cpp
struct ChunkKey {
    std::string layer_id;
    int chunk_x, chunk_y;
    // 支持哈希和比较，作为 unordered_map 键
};

struct ChunkData {
    int chunk_x, chunk_y;
    ChunkState state;       // UNLOADED/GENERATING/GENERATED/ACTIVE/SLEEPING
    TerrainData terrain;     // 32×32 地形格
    std::vector<ConnectorPlacement> connectors;
    std::vector<EntityId> entities;         // 实体引用
    std::vector<EntityId> machines;         // 机器引用
    std::vector<EntityId> connector_ids;    // Connector 运行时 ID
};
```

### 3.3 TerrainData

```cpp
enum class TerrainMaterial : uint8_t {
    AIR, STONE, DIRT, SAND, WATER, LAVA,
    ORE_IRON, ORE_COPPER, ORE_COAL,
    // 可扩展
};

enum TerrainFlags : uint32_t {
    WALKABLE  = 0x01,
    SOLID     = 0x02,
    LIQUID    = 0x04,
    MINEABLE  = 0x08,
};

struct TerrainCell {
    TerrainMaterial material;
    uint32_t flags;
};

struct TerrainData {
    int size_x, size_y;      // 32×32
    std::vector<TerrainCell> cells;
};
```

> 当前代码中的 `TerrainMaterial` enum 是早期原型内置材质 ID。目标架构中，材质 ID 不再通过不断扩展 C++ enum 管理，而由 GDScript/数据层注册稳定内容 key，再冻结为 C++ 可在线程中安全读取的生成快照。详见第 4 节。

> **未实现（文档中存在但代码中未定义）**：
> - `TerrainFlags::BLOCKS_LIGHT` — 原先文档描述但 C++ 代码中只有 WALKABLE/SOLID/LIQUID/MINEABLE 四种标志

## 4. 地形内容注册与世界生成架构（目标）

### 4.1 核心原则

世界生成采用“**内容定义在 GD/数据层，确定性执行在 C++ 核心层**”的边界：

- GDScript/数据文件负责注册地形材质、渲染映射、生物群系、矿脉、结构、掉落与工具需求等内容定义。
- C++ 核心层负责 `WorldData` / `ChunkData` / `TerrainData` 存储、确定性噪声、异步 chunk 生成、存档序列化和服务端权威写入。
- WorkerThreadPool 工作线程只读取不可变快照，不调用 GDScript，不访问 Godot Object，不读取可变 Resource。
- 内容 ID 不通过不断追加 C++ enum 扩展；C++ 只理解数值 ID、flags 和生成规则快照。
- Godot TileMapLayer 仍只是 `ChunkData` 的可视化代理，材质到 atlas tile 的映射也由注册表提供。

这保证了三件事：

1. 内容迭代不用频繁改 C++ enum 和重编译核心。
2. 异步生成保持线程安全和可复现。
3. 单人、LAN、专用服务器都能共享同一套权威生成配置。

### 4.2 分层结构

```
GDScript / 数据层
├── TerrainContentRegistry.gd
│   ├── register_material()
│   ├── register_tile_mapping()
│   ├── register_biome()
│   ├── register_generation_rule()
│   └── freeze_to_worldgen_config()
├── data/worldgen/**/*.tres 或 .json
└── ItemDatabase / TerrainDropTable / WikiDatabase 等内容表

GDExtension 绑定层
├── GDTerrainContentRegistry      # 接收 GD 注册，构建不可变 C++ 快照
├── GDWorldGenConfig              # 可序列化/可校验的生成配置资源
├── GDWorldData                   # 持有 seed + config snapshot，提交异步 chunk 生成
└── GDChunkView / GDChunkManager  # 使用注册后的 palette 渲染 chunk

C++ 核心层
├── WorldGenConfigSnapshot        # 纯 C++ POD/值类型快照
├── TerrainMaterialDef            # id/key/flags/硬度等
├── TerrainRenderDef              # layer/material -> source/atlas/variant
├── BiomeDef / OreVeinDef / StructureDef / GenerationRule
├── TerrainGenerator              # 只读 snapshot + seed，确定性生成
└── WorldData / SaveManager       # 存储、查询、存档
```

### 4.3 地形材质 ID 策略

材质使用两层身份：

| 身份 | 类型 | 作用 |
|------|------|------|
| 内容 key | `StringName` / `std::string` | 稳定语义 ID，例如 `snt:grass`, `snt:sand`, `snt:ore_copper` |
| 运行时 ID | `uint16_t` 或 `uint32_t` | `TerrainCell` 内存储和网络/存档压缩使用 |

规则：

- 内置材质保留固定 ID，兼容当前原型：`air=0`, `stone=1`, `dirt=2`, `sand=3`, `water=4`, `lava=5`, `ore_iron=6`, `ore_copper=7`, `ore_coal=8`, `wood=9`, `leaves=10`。
- 新材质必须先注册 key，再分配或声明运行时 ID；禁止在玩法代码里直接散落魔法数字。
- 存档不只保存数字 ID，还保存 `id -> key` 的材质表和 content pack/config hash。加载时按 key 重映射，避免后续注册顺序变化导致旧世界错材质。
- C++ enum 只作为 legacy/builtin 兼容层保留，不再作为内容扩展入口。
- ID 冲突、缺失 key、flags 缺失、渲染映射缺失都应在 freeze 阶段报错或一次性 warning，而不是生成时高频刷日志。

目标数据结构：

```cpp
using TerrainMaterialId = uint16_t;

struct TerrainMaterialDef {
    TerrainMaterialId id;
    std::string key;
    std::string display_name;
    uint32_t flags;
    float hardness;
    std::string required_tool_tag;
    int required_mining_level;
    std::vector<DropDef> drops;
};
```

### 4.4 GD 侧注册入口

GD 侧是内容注册的唯一入口。典型启动流程：

```gdscript
var terrain_registry := GDTerrainContentRegistry.new()

terrain_registry.register_material({
    "key": "snt:grass",
    "id": 11,
    "display_name": "Grass",
    "flags": GDTerrainContentRegistry.FLAG_WALKABLE | GDTerrainContentRegistry.FLAG_MINEABLE,
    "hardness": 0.35,
    "required_tool_tag": "shovel",
    "required_mining_level": 0,
    "drops": [
        {"item_key": "snt:stone_tiny_dust", "count": 1, "chance": 0.20},
    ],
})

terrain_registry.register_tile_mapping({
    "material_key": "snt:grass",
    "layer": "surface",
    "source_id": 0,
    "atlas": Vector2i(0, 2),
    "variant_count": 12,
})

terrain_registry.register_biome({
    "key": "snt:temperate_forest",
    "layer": "surface",
    "temperature": Vector2(-0.15, 0.45),
    "humidity": Vector2(0.10, 1.00),
    "base_material": "snt:grass",
    "tree_density": 0.32,
})

var frozen_config := terrain_registry.freeze()
world_data.set_worldgen_config(frozen_config)
```

注册表需要支持：

- `register_material(def: Dictionary) -> bool`
- `register_tile_mapping(def: Dictionary) -> bool`
- `register_biome(def: Dictionary) -> bool`
- `register_ore_vein(def: Dictionary) -> bool`
- `register_structure_rule(def: Dictionary) -> bool`
- `register_connector_rule(def: Dictionary) -> bool`
- `freeze() -> GDWorldGenConfig`
- `validate() -> Array[String]`
- `get_material_id(key) -> int`
- `get_material_key(id) -> StringName`

`freeze()` 后的配置不可变；如果需要改内容，必须创建新 registry/config，并等待旧异步任务排空后切换。

### 4.5 C++ 快照与线程安全

`GDWorldGenConfig` 只在主线程构建。传给 C++ 后转换为纯值类型：

```cpp
struct WorldGenConfigSnapshot {
    uint32_t schema_version;
    uint64_t content_hash;
    std::vector<TerrainMaterialDef> materials;
    std::vector<BiomeDef> biomes;
    std::vector<OreVeinDef> ore_veins;
    std::vector<StructureRule> structures;
    std::vector<ConnectorRule> connectors;
    std::unordered_map<std::string, TerrainMaterialId> material_ids_by_key;
};
```

`GDWorldData` 持有：

```cpp
std::shared_ptr<const WorldGenConfigSnapshot> worldgen_config_;
std::unique_ptr<TerrainGenerator> generator_;
```

异步任务只复制 `shared_ptr<const WorldGenConfigSnapshot>`，生成期间不允许修改内部内容。这样保留当前 WorkerThreadPool 设计：

- 不在工作线程调用 GDScript。
- 不在工作线程读写 Godot Resource。
- 不在工作线程访问可变全局注册表。
- 同一 seed + 同一 config hash + 同一 chunk 坐标得到同一结果。

### 4.6 数据驱动生成 Pass

目标生成器分为“固定执行器 + 数据规则”：

```
TerrainGenerator::generate_chunk(layer, cx, cy)
├── pass_base_terrain(config.base_rules)
├── pass_biome(config.biomes)
├── pass_ore(config.ore_veins)
├── pass_structure(config.structures)
├── pass_object(config.object_rules)
└── pass_gameplay(config.connector_rules)
```

Pass 类型由 C++ 内置执行器实现，参数由 GD/数据层注册：

| Pass 类型 | C++ 执行器 | 数据层可配置内容 |
|-----------|------------|------------------|
| Base Terrain | elevation/cave/fill 规则 | 高度噪声、默认材质、水位、洞穴阈值 |
| Biome | temperature/humidity/depth 匹配 | 生物群系条件、主材质、过渡材质、权重 |
| Ore | vein/noise cluster | 矿石材质、宿主材质、密度、形状、深度区间 |
| Structure | blob/ring/pool/carve | 石丘、塌陷坑、洞穴大厅、湖、熔岩池 |
| Object | scatter/patch | 树、碎石、遗迹碎片、装饰占位 |
| Gameplay | connector placement | 洞口、裂隙、遗迹门、锁定状态 |

GD 不直接注册任意脚本回调进 worker。若将来要支持真正自定义算法，有两条路径：

- **数据驱动优先**：扩展 C++ 内置 pass 类型和参数 schema。
- **Native 插件 pass**：通过 C++/GDExtension 注册线程安全 pass，要求不访问 Godot Object。

### 4.7 渲染映射注册

`GDChunkView` 不应硬编码 `TerrainMaterial enum -> atlas coords`。目标是使用冻结后的 palette：

```cpp
struct TerrainTileMapping {
    TerrainMaterialId material_id;
    std::string layer_id;
    int source_id;
    int atlas_x;
    int atlas_y;
    int variant_count;
    bool enabled;
};
```

渲染规则：

- `AIR` 或 `enabled=false` 不绘制 tile。
- `variant_count > 1` 时使用 chunk 坐标 + local 坐标 + world seed 派生稳定 variant。
- 缺失映射只打一条去重 warning，并可使用 debug tile；不要逐格打印。
- TileSet source、atlas 行列、变体数量由 GD/数据层注册。
- 世界小地图、百科、掉落图标也复用同一材质 key，不再各自维护散乱 ID。

### 4.8 掉落、工具和交互规则

地形掉落与工具需求属于内容定义，应从 `TerrainDropTable.gd` 迁移到材质注册表或由注册表驱动：

```gdscript
register_material({
    "key": "snt:stone",
    "drops": [{"item_key": "snt:stone_dust", "count": 1}],
    "required_tool_tag": "pickaxe",
    "required_mining_level": 1,
})
```

服务端命令仍在 `GDGameCommandServer` 中权威校验：

1. 根据 cell material id 查询冻结内容表。
2. 校验 `MINEABLE` flag。
3. 校验工具 tag 和 mining level。
4. 写入 `AIR`。
5. 根据 drop rule 发物品。
6. emit `terrain_cell_synced` / `inventory_synced`。

客户端 UI 可以预检查，但不能绕过服务端校验。

### 4.9 存档与兼容

存档 header 增加世界生成配置元数据：

```text
world_seed
worldgen_schema_version
worldgen_content_hash
material_table_count
material_table: [runtime_id, key]
```

Chunk 内仍保存压缩后的 material id。加载流程：

1. 读取 save 内 material table。
2. 用当前 registry 按 key 查找 material id。
3. 如果 id 相同，直接加载。
4. 如果 id 不同，构建 remap table 后加载。
5. 如果 key 缺失，使用 `snt:unknown_solid` 或拒绝加载，并给出一次性错误日志。

当前 v1/v2 存档中只有旧 enum 数字，按内置 legacy 表映射到 key。

### 4.10 多人和服务端边界

世界生成配置必须由服务端决定：

- Integrated Server：本机服务端从本地数据构建 registry/config。
- Open to LAN：房主服务端广播 config hash 和必要内容表。
- Dedicated Server：服务端加载自己的 content pack，客户端必须匹配 hash 或下载/同步表现层配置。

客户端不能单独注册影响玩法的地形材质、掉落、硬度、矿脉或结构规则。客户端可注册本地化文本、图标、可视化替代资源，但不能改变服务端权威 material id 和 generation rules。

### 4.12 迁移步骤

| 阶段 | 目标 | 说明 |
|------|------|------|
| P0 | 保留当前 enum，补齐文档边界 | 明确 enum 是 legacy/builtin，不再作为扩展入口 |
| P1 | 新增 GD 注册表和 `GDWorldGenConfig` | GD 注册材质、palette、biome，freeze 成 C++ 快照 |
| P2 | `TerrainGenerator` 改读 config snapshot | 先复刻当前硬编码行为，保证生成结果可对照 |

| P6 | 添加更多地形内容 | Grass、dry grass、mud、clay、gravel、basalt、ruins floor 等通过 GD 数据注册 |

### 4.13 对当前“地形只有沙子”的处理方向

短期修复不应继续追加 C++ enum。推荐处理路径：

1. 检查当前 `TerrainMaterialPalette` 是否把 `SAND` 错映射到 DIRT 图块，先修正渲染映射。
2. 暴露 palette 注册接口，让 GD 配置 `SAND` 使用 sand atlas 行。
3. 在 GD 注册 `grass` 等新材质前，先用现有 `DIRT/SAND/STONE/WATER/WOOD/LEAVES` 调整生成权重，确认视觉丰富度。
4. 再通过注册表新增 `grass/mud/gravel/clay` 等内容材质，避免把每个内容材质都固化到 C++ enum。

## 5. 地形生成流水线（当前 TerrainGenerator 实现）

### 5.1 六阶段生成

`TerrainGenerator` 调用 6 个 Pass，每个 Pass 使用独立种子的 `NoiseGenerator`，天然线程安全。

```
generate_chunk(layer_id, cx, cy)
├── 1. pass_base_terrain()    ✅ 已实现
├── 2. pass_biome()           ✅ 已实现
├── 3. pass_ore()             ✅ 已实现
├── 4. pass_structure()       ✅ 已实现
├── 5. pass_object()          ✅ 已实现
└── 6. pass_gameplay()        ✅ 已实现
```

### 5.2 Pass 1: 基础地形（已实现）

**Surface 层**：
- 使用 elevation noise（`scale=0.02, octaves=4`）区分地表高度
- 顶部行：DIRT（可通行），进一步用 detail noise 添加 SAND 斑块
- 下部行：STONE

**Underground 层**：
- 使用 cave noise（`scale=0.04, octaves=4`）生成洞穴
- 洞穴内：AIR（可通行），边缘区域保持 STONE

### 5.3 Pass 2: 生物群系（已实现）

使用两个大尺度噪声 **temperature**（温度）和 **humidity**（湿度）驱动生物群系覆盖 (`scale=0.015~0.02`)：

| 条件 | Surface 行为 | Underground 行为 |
|------|-------------|-----------------|
| 热 + 干 | DIRT → **SAND**（沙漠） | — |
| 近水（2 格内） | DIRT → **SAND**（沙滩） | — |
| 冷 + 干 | DIRT → **STONE**（石质高地） | — |
| 温 + 湿 | 保持 DIRT（沃土） | AIR → **DIRT**（洞穴泥地） |
| 干 + 冷洞穴 | — | AIR → **SAND**（干燥洞穴沙地） |

温度/湿度各用独立 sub-seed 的 NoiseGenerator，确保世界种子下的确定性分布。

### 5.4 Pass 3: 矿石生成（已实现）

使用两组 noise：
- `ore_noise`（`scale=0.08, octaves=3`）：矿脉密度
- `vein_noise`（`scale=0.15, octaves=2`）：矿脉形状

在 STONE 格中放置：
- ORE_IRON（铁矿石）：密度 > 0.6 且形状 > 0.4
- ORE_COPPER（铜矿石）：密度 > 0.4 且形状 > 0.6
- ORE_COAL（煤矿石）：密度 > 0.7 且形状 > 0.3

### 5.5 Pass 4: 大型结构（已实现）

使用 `density` + `shape` 双噪声驱动结构生成：

| 特征 | Surface | Underground |
|------|---------|-------------|
| **石山** | 圆形 STONE 隆起（半径 2-6） | — |
| **悬崖** | 水边 DIRT→STONE 垂直面 | — |
| **大厅** | — | 扩大 AIR 洞穴（半径 2-7） |
| **熔岩池** | — | 洞穴底部 AIR→**LAVA** |
| **地下湖** | — | 洞穴底部 AIR→**WATER** |

每个特征使用独立的 sub-seed，`density` 阈值 >0.55，`shape` 控制半径/位置偏移。

### 5.6 Pass 5: 小型物体（已实现）

使用 `density` 噪声 + 随机种子偏移确定物体位置：

| 物体 | Surface | Underground |
|------|---------|-------------|
| **碎石堆** | DIRT/SAND→STONE 散落 1-4 格 | — |
| **石环遗迹** | 水边 8 格 STONE 环（中心可通行，半径 4） | — |
| **钟乳石柱** | — | 顶到底 STONE 填充（2-4 格高） |
| **额外矿簇** | — | 已有矿脉旁追加邻近 ORE 格 |

所有物体使用 `WorldSeed.chunk_seed(PASS_OBJECT, cx, cy)` 派生的确定性位置。Pass 5 直接修改地形材质（STONE 碎石/石柱），而非添加实体数据。

### 5.7 Pass 6: Gameplay 元素（已实现）

使用 placement noise（`scale=0.15, octaves=1`）确定 Connector 位置：

| Connector 类型 | 触发层 | 激活方式 | 方向 |
|---|---|---|---|
| Cave Entrance | Surface | 交互（E 键） | 双向 |
| Rift | Surface | 自动（经过即触发） | 单向（Surface→Underground） |
| Ruin Gate | Underground | 交互（E 键） | 双向，初始锁定 |

每个 Connector 有确定性的 `connector_id`（由 `WorldSeed` 的 seed 推导），放置位置记录在 `ChunkData.connectors` 中。

### 5.8 跨层联动生成 [已实现]

Surface ↔ Underground 使用确定性噪声对齐，无需实际交换 ChunkData：

| 方向 | 效果 | 实现位置 |
|------|------|----------|
| Surface → Underground | 地表洞穴入口下方 → 地下入口大厅（清除石头、铺设泥地） | `pass_structure` underground 分支 |
| Underground → Surface | 大型地下洞穴上方 → 地表塌陷坑（石环 + 空气凹陷） | `pass_structure` surface 分支 |

**原理**：在 `pass_structure` 中加入 `cross_noise`，使用 `STRUCTURE + 10` sub-seed 生成。地表和地下在相同 chunk 坐标下计算相同的噪声值，自动对齐特征位置，无需跨层数据传递。两个效果使用不同的噪声偏移（+11000 / +13000 global cell offset），避免重叠。

## 6. 噪声生成器（NoiseGenerator）

### 6.1 当前实现

自定义 Perlin 分形噪声（~140 行 C++）：
- 256 字节排列表（Fisher-Yates 洗牌、确定性种子）
- Perlin 改善衰减曲线：`6t⁵ − 15t⁴ + 10t³`
- 分形八度叠加（可配置 octaves / persistence）
- 返回值范围 [-1, 1]

### 6.2 保留原因

- 纯 C++ 栈分配，**零堆内存分配**
- **天然线程安全**：每个 `generate_chunk()` 调用在栈上构造独立实例
- 虚函数开销为零（inline 方法）
- 与 Godot `FastNoiseLite`（Object/RefCounted 子类）相比，在 `WorkerThreadPool` 工作线程中使用 Godot Object 存在线程安全风险

> 将来若需要 Cellular / DomainWarp 等高级噪声，可引入独立 C++ 头文件 [FastNoiseLite](https://github.com/Auburn/FastNoiseLite)（非 Godot 绑定版本），直接嵌入核心层，避免 Godot Object 依赖。

## 7. 异步 Chunk 生成

### 7.1 架构

使用 Godot 内置 `WorkerThreadPool::get_singleton()` 单例：

```
GDScript __process__ → GDWorldData.process_async_results()
GDScript → GDWorldData.request_chunk_async()
  └── WorkerThreadPool::add_native_task()
      └── 工作线程 → _async_chunk_callback()
          ├── TerrainGenerator::generate_chunk()  # 栈上 NoiseGenerator
          └── mutex → async_results_.push(result)
```

### 7.2 GDWorldData 异步接口

```cpp
class GDWorldData : public godot::Resource {
    // 非阻塞：提交 chunk 生成任务
    void request_chunk_async(layer_id, chunk_x, chunk_y);

    // 每帧调用：消费已完成的结果，emit chunk_ready 信号
    int64_t process_async_results();

    int64_t get_async_pending_count() const;  // 排队中 + 运行中
    int64_t get_worker_thread_count() const;  // CPU 核心数
    int64_t get_async_completed_count() const;
    int64_t get_max_async_queue_size() const; // 默认 256
};
```

### 7.3 容量控制

`max_async_queue_size`（默认 256）：当 `get_async_pending_count() >= max` 时，`request_chunk_async()` 静默丢弃新请求，防止无界队列增长。

### 7.4 线程安全保证

- `TerrainGenerator::generate_chunk()` 只读 `world_seed_`（const），其余栈上分配
- `NoiseGenerator` 在 `generate_chunk()` 内构造独立 `perm_` 数组
- 结果通过 `std::mutex` 保护的 `std::queue` 回传主线程
- `emit_signal("chunk_ready")` 仅在主线程的 `process_async_results()` 中调用

## 12. Chunk 渲染桥接层 [已实现]

三层架构：**C++ GDExtension 绑定层**（GDChunkManager / GDChunkView） + **GDScript 集成桥**（ChunkRendererBridge） + **材质映射**（TerrainMaterialPalette）。

### 12.1 三层 Chunk 模型

| 层级 | 半径 | 行为 |
|------|------|------|
| **Loaded**（模拟层） | `loaded_radius`（默认 10） | ChunkData 保留在 WorldData 内存中，机器/网络继续模拟 |
| **Visible**（渲染层） | `view_radius`（默认 6） | 创建 GDChunkView（TileMapLayer 子节点）用于视觉渲染 |
| **Force-loaded**（强制加载） | 指定 chunk 列表 | 无视玩家距离，始终保留在内存和模拟中（工厂机器 chunk） |

要求：`loaded_radius ≥ view_radius`。`unload_distant_chunks` 默认为 false（所有生成的 chunk 保留内存）。

### 12.2 GDChunkManager（C++ GDExtension 绑定）

`Node` 子类，管理所有 chunk 的三层生命周期：
- **Player 追踪**：`set_player_chunk(layer, cx, cy)` 每帧更新玩家位置
- **刷新**：`refresh_chunks()` 对比三层半径，卸载/隐藏/显示/加载 chunk
- **强制加载**：`force_load_chunk()` / `force_unload_chunk()` 管理工厂 chunk
- **信号连接**：`on_chunk_ready(layer, cx, cy)` 连接到 `GDWorldData.chunk_ready`
- **层级容器**：通过 `surface_container_path` / `underground_container_path` NodePath 定位 TileMapLayer 父节点

### 12.3 GDChunkView（C++ GDExtension 绑定）

`TileMapLayer` 子类，负责单个 chunk 的视觉渲染：
- `update_from_terrain(terrain_dict)`：将 TerrainData（PackedByteArray materials + PackedInt32Array flags）绘制到 TileMapLayer
- `clear_chunk()`：清除所有 tile
- 内部使用 `TerrainMaterialPalette` 将 `TerrainMaterial` 枚举映射为 tileset 坐标

### 12.4 TerrainMaterialPalette（C++ 静态库）

`src/godot_adapters/terrain_material_palette.{hpp,cpp}`，材质→tile 映射：

| TerrainMaterial | Surface 瓦片 | Underground 瓦片 |
|----------------|-------------|-----------------|
| AIR | — | — |
| STONE | (4,0) | (3,3) |
| DIRT | (0-3,0) 4 变体 | (0-2,3) 3 变体 |
| SAND | (0-3,0) 4 变体 | (0-2,3) 3 变体 |
| WATER | (8,0) | (8,0) |
| LAVA | (9,0) | (9,0) |
| ORE_IRON | (5,0) | (4,3) |
| ORE_COPPER | (6,0) | (5,3) |
| ORE_COAL | (7,0) | (6,3) |

支持每层独立 `source_id`，默认均为 0。variant_count > 1 时使用 variant_seed 选择变体。

### 12.5 ChunkRendererBridge（GDScript 集成桥）

`scripts/ChunkRendererBridge.gd`，`class_name ChunkRendererBridge`，`extends Node`：

```
ChunkRendererBridge (Node)
├── @export world_data: GDWorldData
├── @export chunk_manager: GDChunkManager
├── @export surface_container_path: NodePath → "Layers/SurfaceLayer"
├── @export underground_container_path: NodePath → "Layers/UndergroundLayer"
├── @export seed / loaded_radius / view_radius / tile_size
│
├── initialize()        — 创建/连接 GDWorldData + GDChunkManager
├── process_frame()     — 每帧调用：process_async_results + set_player_chunk + refresh_chunks
├── force_load_factory_chunk() — 强制加载工厂 chunk
├── release_factory_chunk()    — 释放强制加载
└── unload_all()        — 卸载所有 chunk
```

`_ready()` 中 `call_deferred("initialize")`，`_generate_initial_chunks()` 生成 `start_chunk_radius`（默认 2）范围内所有 surface + underground chunk。

```gdscript
var world = GDWorldData.new()
world.seed = randi()
world.max_async_queue_size = 256

func _process(delta):
    # 消费已完成的异步生成结果（emit chunk_ready 信号）
    var completed = world.process_async_results()

# 请求异步生成
world.request_chunk_async("surface", 0, 0)
world.request_chunk_async("underground", 0, 0)

```
