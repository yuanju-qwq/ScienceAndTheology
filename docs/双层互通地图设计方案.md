# 双层互通地图设计方案

本文档用于规划当前 Godot 项目中的“地表和地下双层互通地图”系统。目标是实现类似《奈斯启示录》那种上下层互相影响、可以通过洞口、楼梯、裂缝、井道等地点切换的立体地图结构。

## 设计目标

双层地图不应该只是两张彼此独立的地图，而应该表现为同一个世界在不同高度上的两个层面。

核心目标如下：

- 地表和地下共用同一套二维坐标逻辑。
- 玩家可以通过明确的连接点在地表和地下之间切换。
- 地表和地下的地形、机关、路径和探索目标可以互相影响。
- 系统结构要方便后续扩展到多层地图，例如地表、地下 1 层、地下 2 层、建筑内部等。
- 初期实现要简单稳定，后续再逐步增加视觉表现和复杂交互。

## 核心概念

推荐把地图位置表示为三部分：

```gdscript
var layer_id: StringName
var cell_position: Vector2i
var world_position: Vector2
```

其中：

- `layer_id` 表示玩家当前所在层级，例如 `surface` 或 `underground`。
- `cell_position` 表示玩家在 TileMap 中的格子坐标。
- `world_position` 表示玩家在场景中的实际坐标。

地表和地下使用相同的 TileMap 尺寸、原点和坐标系。这样同一个 `Vector2i(12, 8)` 可以同时代表地表上的某个位置，也可以代表地下对应位置。

## 推荐场景结构

建议先创建一个主地图场景，例如 `WorldMap.tscn`：

```text
WorldMap
├─ SurfaceLayer
├─ UndergroundLayer
├─ ConnectorManager
├─ Player
└─ Camera2D
```

各节点职责如下：

- `SurfaceLayer`：负责地表地图，可使用 `TileMapLayer` 或多个场景块。
- `UndergroundLayer`：负责地下地图，可使用 `TileMapLayer` 或多个场景块。
- `ConnectorManager`：负责管理所有上下层连接点。
- `Player`：玩家角色。
- `Camera2D`：跟随玩家，并根据当前层级调整边界或视觉效果。

如果地图较大，可以把每层拆成多个地图块：

```text
WorldMap
├─ Layers
│  ├─ Surface
│  │  ├─ Chunk_0_0
│  │  ├─ Chunk_0_1
│  │  └─ Chunk_1_0
│  └─ Underground
│     ├─ Chunk_0_0
│     ├─ Chunk_0_1
│     └─ Chunk_1_0
├─ ConnectorManager
├─ Player
└─ Camera2D
```

初期不建议一开始就做分块加载。当前项目还处在初始阶段，先用完整的双层 TileMap 验证玩法更合适。

## 图层切换方案

第一版建议采用“切层式双层地图”。

玩家在地表时：

- 显示地表层。
- 启用地表碰撞。
- 隐藏或弱化地下层。
- 地下敌人、机关、事件可以暂停或进入低频更新。

玩家在地下时：

- 显示地下层。
- 启用地下碰撞。
- 隐藏或弱化地表层。
- 相机边界切换到地下地图范围。

切层流程如下：

1. 玩家进入连接点触发区域。
2. 锁定玩家输入。
3. 播放进入洞口、下楼、落下或升起动画。
4. 切换当前层级。
5. 把玩家移动到目标位置。
6. 更新显示、碰撞、相机边界和小地图状态。
7. 解锁玩家输入。

## 连接点设计

连接点是地表和地下互通的核心。每个连接点应至少包含起点、终点、方向和状态。

推荐创建一个资源类：

```gdscript
class_name MapConnector
extends Resource

@export var from_layer: StringName
@export var from_cell: Vector2i
@export var to_layer: StringName
@export var to_cell: Vector2i
@export var one_way := false
@export var locked := false
@export var connector_type: StringName
```

示例数据：

```gdscript
{
	"from_layer": "surface",
	"from_cell": Vector2i(12, 8),
	"to_layer": "underground",
	"to_cell": Vector2i(12, 9),
	"one_way": false,
	"locked": false,
	"connector_type": "cave_entrance"
}
```

常见连接点类型：

- 洞口：地表进入地下洞穴。
- 楼梯：建筑、遗迹或地下城常用。
- 井：可以下到地下水道或洞穴。
- 裂缝：通常是单向下落或需要特殊能力返回。
- 电梯：适合科技、矿井、神秘设施等主题。
- 传送阵：适合神学、仪式、遗迹类玩法。

## 层级控制脚本

建议创建一个 `MapLayerController.gd`，专门负责当前层级切换。

职责包括：

- 保存当前层级。
- 控制不同层级的显示和隐藏。
- 控制不同层级的碰撞启用状态。
- 通知玩家、相机、敌人、事件系统当前层级变化。
- 提供统一的 `change_layer()` 方法。

伪代码如下：

```gdscript
extends Node

signal layer_changed(old_layer: StringName, new_layer: StringName)

@export var surface_layer: Node
@export var underground_layer: Node

var current_layer: StringName = "surface"

func change_layer(target_layer: StringName) -> void:
	if current_layer == target_layer:
		return

	var old_layer := current_layer
	current_layer = target_layer

	_apply_layer_visibility()
	_apply_layer_collision()

	layer_changed.emit(old_layer, current_layer)

func _apply_layer_visibility() -> void:
	surface_layer.visible = current_layer == "surface"
	underground_layer.visible = current_layer == "underground"

func _apply_layer_collision() -> void:
	# 可根据项目实际节点类型实现。
	# 第一版也可以先只切 visible，碰撞后续再精细化。
	pass
```

## 玩家传送流程

建议由连接点节点检测玩家进入，再调用统一的地图切换方法。

伪代码如下：

```gdscript
func use_connector(connector: MapConnector) -> void:
	if connector.locked:
		return

	player.set_input_enabled(false)

	await _play_transition_animation(connector)

	layer_controller.change_layer(connector.to_layer)

	var target_world_position := tile_map.map_to_local(connector.to_cell)
	player.global_position = target_world_position

	await _finish_transition_animation(connector)

	player.set_input_enabled(true)
```

这一段逻辑不要分散写在每个洞口或楼梯里，否则后期很难维护。所有连接点都应该走统一入口。

## 地表和地下的玩法关联

双层地图最重要的不是“能切换”，而是两层之间要产生设计关系。

可以采用以下设计：

- 地表河流影响地下暗河路线。
- 地表神庙正下方是地下遗迹。
- 地表炸开岩壁后，地下出现新的入口。
- 地下启动机关后，地表桥梁升起或大门打开。
- 地表能看到目标，但必须从地下绕路到达。
- 地下通道可以绕过地表的封锁区域。
- 地表的祭坛、树根、管线或机械结构延伸到地下。
- 地下怪物、污染、神秘力量会逐步影响地表环境。

这样玩家会感觉自己探索的是一个立体世界，而不是两个关卡列表。

## 视觉表现阶段

建议分三个阶段实现。

### 第一阶段：基础切换

这是最重要的版本。

实现内容：

- 地表和地下各一层地图。
- 玩家通过连接点切换。
- 每次只显示当前层。
- 碰撞和相机随层级变化。

优点：

- 实现简单。
- 易于调试。
- 不容易出现渲染和碰撞混乱。

### 第二阶段：局部透视

在基础切换稳定后，可以加入局部提示。

示例：

- 地表洞口附近显示地下空间的暗色轮廓。
- 地下天井附近显示来自地表的光束。
- 某些薄地板区域可以隐约看到下方空间。
- 小地图可以显示另一层的已探索轮廓。

这一阶段能显著增强双层地图的真实感。

### 第三阶段：双层同屏

某些特殊区域可以同时看到地表和地下。

示例：

- 断崖。
- 竖井。
- 地下桥。
- 塌陷区域。
- 透明地板或神秘观测装置。

这一阶段视觉效果最好，但复杂度也最高。建议等系统稳定后再做。

## 存档数据设计

存档至少应记录：

```gdscript
{
	"current_layer": "surface",
	"player_cell": Vector2i(10, 6),
	"visited_layers": ["surface", "underground"],
	"unlocked_connectors": ["cave_entrance_001", "well_003"],
	"world_flags": {
		"opened_temple_gate": true,
		"activated_underground_machine": false
	}
}
```

重点是连接点状态和世界事件状态。因为地下机关影响地表时，必须能在读档后恢复正确状态。

## 开发顺序建议

推荐按以下顺序开发：

1. 创建 `WorldMap.tscn`。
2. 添加 `SurfaceLayer` 和 `UndergroundLayer`。
3. 让两层使用相同 tile size 和坐标原点。
4. 创建 `MapLayerController.gd`。
5. 实现地表和地下的显示切换。
6. 创建 `MapConnector` 资源或节点。
7. 实现玩家踩到连接点后切换层级。
8. 加入切换动画和输入锁定。
9. 加入相机边界切换。
10. 加入小地图或地图探索记录。
11. 加入地表和地下互相影响的机关。
12. 再考虑局部透视和双层同屏效果。

## 当前项目建议

当前项目还是比较干净的 Godot 4.5 项目，适合先从最小可玩原型开始。

建议第一版只实现以下内容：

- 一个 `WorldMap.tscn`。
- 一个地表 `TileMapLayer`。
- 一个地下 `TileMapLayer`。
- 一个玩家角色。
- 两个连接点：地表洞口和地下出口。
- 一个 `MapLayerController.gd`。

第一版目标不是好看，而是验证：

- 玩家能从地表进入地下。
- 玩家能从地下回到地表。
- 切换后碰撞正确。
- 切换后相机跟随正确。
- 连接点数据可以继续扩展。

等这个闭环跑通后，再开始做更复杂的地图内容和视觉表现。

## 总结

最稳妥的设计是：地表和地下共享二维坐标系，但通过 `layer_id` 区分当前所在层。地图显示、碰撞、相机、事件和小地图都跟随 `layer_id` 切换。

先实现“切层式双层地图”，再逐步加入局部透视和双层同屏效果。这样既能快速做出可玩的原型，又不会在前期被复杂渲染和多层碰撞拖住进度。


地图素材先于角色素材
因为你现在的核心玩法是双层地图、洞口、裂缝、机关、局部透视。地图素材会直接影响 tile size、碰撞、层级表现和视觉读图。

先找一套风格统一的环境包
最好同时覆盖：

地表：草地、道路、岩壁、洞口、断桥、祭坛
地下：洞穴、遗迹、暗河、石门、机关
过渡元素：裂缝、井口、楼梯、光束、阴影轮廓
角色素材先用简单可动的
角色不用一开始就特别精致，但至少要有：

idle
walk 四方向，或至少左右/上下移动能看得过去
统一像素比例，比如 16x16、24x24、32x32 任选一种
机关素材也很重要
你现在已经有祭坛、泵机、桥、封门这些逻辑了，接下来需要对应视觉状态：

未激活/已激活祭坛
封闭/开启石门
断桥/升起桥
地表裂缝/地下裂缝对应状态
