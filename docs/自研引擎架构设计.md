## 1. 背景与决策

### 1.1 为什么放弃 Godot

Voxel 游戏的性能瓶颈与 Godot 的设计哲学存在结构性冲突：

| Voxel 游戏需要 | Godot 提供 | 冲突 |
|---|---|---|
| 海量小数据（百万级 cell） | Node 树、Object 体系 | Node 是重量级对象，每个都带 _get_property、信号、notif |
| 1 draw call / chunk | MeshInstance3D + 多材质拆分 | 默认多 draw call |
| 自定义碰撞（chunk 级） | Bullet 物理 broadphase | 25 个 StaticBody 仍会被遍历 |
| 主线程零阻塞 | 主线程同步 _process | GDScript 主循环跑不掉 |
| GPU 直接上传 | ArrayMesh → RenderingServer 中转 | 多一层拷贝 |

C++ GDExtension 只能下沉"算 mesh 数据"，无法下沉"提交 mesh 给 GPU 和维护场景树"。
而 Java 版 MC 的快恰恰快在渲染提交层 — 它根本没有场景图，直接 glDraw。
继续在 Godot 体系内优化天花板有限，永远会比 Java MC 卡。

### 1.2 决策记录

经过 6 种架构方案对比（完全自研 / Godot 当壳 / Fork Godot / 移植 Godot 模块 / 换引擎 / 进程分离），
选定方案 A：**完全自研引擎**。

| 决策项 | 选择 | 理由 |
|---|---|---|
| 主架构 | 完全自研引擎 | 从根本上消除 Godot 场景图开销 |
| 渲染 API | 仅 Vulkan | 一套代码覆盖 Win/Linux/Android；macOS 走 MoltenVK |
| 窗口/输入 | SDL3 | 跨平台，输入/音频/窗口一体 |
| UI 系统 | 完整复刻 MUI（Modern UI） | MVVM + 保留模式 + SDF 文本 + HarfBuzz + emoji |
| 数据架构 | 纯 ECS（Entity-Component-System） | 海量机器/生物/掉落物的数据连续布局 + 可并行 tick |
| ECS 内核 | EnTT (header-only, MIT) | C++ ECS 事实标准，sparse set 存储 |
| 脚本系统 | C++NativeScript（参考 GroovyScript 设计） | 单一语言栈、类型安全、DLL 热重载 |
| 物理引擎 | 自研 voxel 碰撞查询 | 针对 voxel 优化，无 broadphase 负担 |
| 音频 | miniaudio | 单文件头，跨平台，0 依赖 |
| 资产格式 | JSON + 二进制 raw | JSON 可读，二进制高效 |
| 构建系统 | CMake | 现有项目已用，DLL 模块化构建 |
| 编程语言 | C++20（单一语言，脚本也是 C++） | 类型安全 + 性能极致 + 0 学习成本 |
| 仓库策略 | 同仓库新增 `engine/` 目录 | 保留 git 历史，渐进迁移 |
| Godot 移除时机 | P1 开始立即移除 | 避免双轨维护 |

### 1.2.1 脚本系统选型理由

经过 5 种脚本方案对比（硬嵌 JVM+Groovy / Lua+sol2 / AngelScript / ChaiScript / 自研 DSL / C++NativeScript），
选定 **C++NativeScript**：

- **单一语言栈**：C++ 引擎 + C++ 脚本，不引入第二门语言
- **类型安全**：编译期检查，运行时不会因类型错误崩溃
- **性能极致**：原生 C++ 函数调用，无解释器开销
- **0 学习成本**：开发者已会 C++
- **热重载**：DLL/so/dylib 动态库重载，1-3 秒生效
- **参考 GroovyScript 设计**：API 风格、reload 机制、双列表状态管理、packmode、命令系统
- **代价**：整合包作者需要 C++ 编译环境（项目不开放给非程序员写脚本）

### 1.2.2 ECS 选型理由

经过 4 种数据架构方案对比（纯 OOP / 纯 ECS / 混合架构 / 轻量 ECS），选定 **纯 ECS**：

- **工厂自动化场景**：GregTech 风格机器数量轻松过 1000，必须数据连续布局
- **并行 tick**：不同 System 可并行跑（机器/生物/网络独立）
- **扩展性**：加新能力 = 加新 Component，不改现有代码
- **与 NativeScript 兼容**：脚本通过 API 访问 Component，不直接碰内存
- **EnTT 内核**：header-only，sparse set 存储，C++ ECS 事实标准

### 1.3 代价承认

| 维度 | 代价 |
|---|---|
| 现有 GDScript 脚本 | 30+ 个全部作废重写为 C++NativeScript |
| 现有 .tscn 场景 | 全部作废 |
| 现有 .tres 资源 | 改为 JSON 格式重新组织 |
| 现有 C++ 代码 | ~80% 算法层可复用，需剥离 godot-cpp 依赖；Manager 类要重构为 ECS System |
| Godot 编辑器 | 失去，调坐标靠日志/ImGui |
| 回到现状功能 | 3-6 个月（全职） |

## 2. 设计原则

1. **数据驱动**：所有配置走 JSON，玩法逻辑走 C++NativeScript，引擎核心是 C++ ECS。
2. **零主线程阻塞**：meshing、IO、布局全在 worker 线程，主线程只做提交。
3. **零场景图**：不维护 Node 树，渲染对象用 RID 表直接管理。
4. **数据导向（DOD）**：POD Component + sparse set 存储，cache 友好。
5. **MVVM 分层**：UI 严格分 View / ViewModel / Model，数据绑定驱动。
6. **算法优先于抽象**：hot path 保持 plain struct + 索引数组。
7. **可演进为无头服务端**：核心系统不依赖窗口、相机、UI。
8. **跨平台预留**：Vulkan + SDL3 + miniaudio 已覆盖主流平台；macOS 走 MoltenVK。
9. **单一语言栈**：C++ 通吃引擎 + 脚本，避免跨语言 binding 维护成本。

依赖规则：
- 上层可依赖下层，反之禁止。
- 同层模块之间通过事件总线（EventBus）通信，禁止直接引用。
- `ecs_world` 是核心调度枢纽，所有 System 通过它访问 Component。
- `data_core` 独立于渲染后端，可单独编译为无头服务端。
- `script` 通过 ECS API 访问 Component，不直接碰内存。

## 5. 核心子系统设计

### 5.1 Voxel 渲染系统

**目标**：1 draw call / chunk，主线程零阻塞，对齐 Java MC（Sodium 风格）。

```text
Main Thread                          Worker Thread Pool
─────────────                        ──────────────────
1. 排序 chunk by 距离                1. 取 chunk terrain 数据
2. frustum cull (native 数组)        2. 跑 greedy mesh (复用现有算法)
3. for each visible chunk:           3. 写入 staging VertexBuffer
      vkCmdBindVertexBuffers         4. atomic 标记 ready
      vkCmdDrawIndexed               5. callback 通知主线程 upload
```

**关键设计**：
- chunk → RID 表：`unordered_map<ChunkCoord, ChunkRenderHandle>`
- 每个 ChunkRenderHandle 持有 vertex buffer / index buffer / instance transform
- 材质 atlas 化：所有方块纹理合到一张图集，UV2 区分
- LOD：near = full mesh + collision；mid = simplified mesh；far = billboard
- meshing 完全在 worker 线程，主线程只 `vkCmdBind` + `vkCmdDraw`

**复用现有代码**：
- `src/bindings/world/gd_chunk_helper.cpp::build_greedy_mesh` 算法逻辑保留
- `src/bindings/world/gd_chunk_helper.cpp::build_collision_faces` 算法逻辑保留
- `src/core/save/*` region_file / chunk_serializer 文件格式保留
- `src/core/simulation/*` 地形生成保留

### 5.2 Voxel 物理系统

**目标**：玩家走 voxel 碰撞，无 broadphase 负担。

- 玩家 AABB vs voxel grid：直接采样 `WorldData.get_cell(cell)`，O(N) N=玩家附近 cell 数
- 射线检测：DDA 算法遍历 voxel grid，O(射线路径长度)
- 刚体（掉落物、生物）：Jolt Physics（远期）；当前只有玩家和静态机器，自研够用
- 远距离 chunk 无碰撞（已有 `horizontal_lod0_radius` 概念）

#### 5.5.4 Reload 机制（参考 GS 双列表状态管理）

5 个 DLL reload 坑及对应解决方案：

| 坑 | 方案 |
|---|---|
| 指针/引用失效 | 事件存 `(script_id, callback_id)`，不存函数指针 |
| 全局/静态变量丢失 | 脚本不持有跨 reload 状态，走 `snt::state` API |
| 内存所有权混乱 | 脚本传值（struct copy），不传指针 |
| 符号冲突 | `SNT_SCRIPT_ENTRY()` 宏生成 file-unique 符号 |
| 文件锁 (Windows) | 卸载后 sleep 50ms，编译到 .tmp 再 rename |

```cpp
// ScriptManager::reload_script_internal
void reload_script_internal(ScriptEntry& entry) {
    // 1. Call on_reload (script-defined cleanup)
    if (entry.reload_fn) entry.reload_fn();
    // 2. Registry cleanup: scripted list cleared, backup restored
    registry_hub.on_script_unload(entry.id);
    // 3. Unload DLL (Windows: wait 50ms after FreeLibrary)
    entry.lib.unload();
    std::this_thread::sleep_for(std::chrono::milliseconds(50));
    // 4. Load new DLL
    entry.lib.load(entry.dll_path);
    // 5. Resolve entry symbol (file-unique)
    entry.entry_fn = entry.lib.get_symbol("snt_entry_" + entry.id);
    // 6. Re-call entry to re-register
    entry.entry_fn();
}
```

### 5.8 ECS 架构

采用纯 ECS（Entity-Component-System）作为引擎核心数据架构。基于 EnTT（header-only，MIT 协议）。

#### 5.8.1 核心类型

```cpp

constexpr EntityId INVALID_ENTITY = 0;

struct Entity {
    EntityId id;
    uint32_t version;  // 每次复用 id 时 +1
```

#### 5.8.2 Component 存储策略：Sparse Set

选 Sparse Set 而非 Archetype（Unity DOTS 那种），理由：
- 加/删 Component O(1)
- 支持运行时动态 Component（脚本注册需要）
- 现成成熟库：EnTT

```cpp
// engine/engine/ecs/world.h
namespace snt::ecs {

// World wraps entt::registry, adds engine-specific helpers.
class World {
public:
    // Entity lifecycle
    Entity create_entity();
    void destroy_entity(Entity e);
    bool is_alive(Entity e) const;

    // Component ops (compile-time known types, fast path)
    template<typename T>
    void add_component(Entity e, T&& comp);
    template<typename T>
    T* get_component(Entity e);
    template<typename T>
    bool has_component(Entity e) const;
    template<typename T>
    void remove_component(Entity e);

    // Component ops (runtime-registered types, script path)
    void add_component(Entity e, ComponentTypeId tid, const void* data);
    void* get_component(Entity e, ComponentTypeId tid);
    void remove_component(Entity e, ComponentTypeId tid);

    // Query: returns all entities matching a component signature
    template<typename... Components>
    auto view();

    // System registration
    void register_system(std::unique_ptr<ISystem> sys);
    void update_systems(float delta);

private:
    entt::registry _registry;
    std::vector<std::unique_ptr<ISystem>> _systems;
    std::unordered_map<ComponentTypeId, std::unique_ptr<ComponentStorageBase>> _runtime_storages;
};

} // namespace snt::ecs
```

#### 5.8.7 ECS + NativeScript 集成

脚本通过以下方式访问 ECS，**不直接碰 Component 内存**：

1. **注册运行时 Component**（脚本自定义状态）：
   ```cpp
   snt::components::register_type({
       .name = "BronzeFurnaceState",
       .fields = {{"temperature", FieldType::FLOAT, "20.0"}}
   });
   ```

2. **注册自定义 System**（声明 Component 需求）：
   ```cpp
   snt::systems::register_system({
       .name = "bronze_furnace_tick",
       .required_components = {"MachineComponent", "BronzeFurnaceState"},
       .tick_rate = 20.0f,
       .callback_id = "on_bronze_furnace_tick"
   });
   ```

3. **事件派发**：引擎查询匹配 Entity，通过 `SystemTickEvent` 派发给脚本 callback：
   ```cpp
   struct SystemTickEvent {
       std::string system_name;
       float delta;
       uint64_t entity_count;
       EntityId* entities;        // array of matching entities
   };
   ```

4. **Component 访问**（值类型，安全）：
   ```cpp
   // 标准 Component（typed，快路径）
   PositionComponent pos = snt::entity::get_position(e);
   snt::entity::set_position(e, pos);

   // 运行时 Component（by name，慢路径）
   double temp = snt::components::get_float(e, "BronzeFurnaceState", "temperature");
   snt::components::set_float(e, "BronzeFurnaceState", "temperature", temp + 2.5);
   ```

#### 5.8.8 性能特性

| 特性 | 实现 |
|---|---|
| 数据连续布局 | EnTT sparse set，同类型 Component 内存连续 |
| Cache 友好 | System 遍历 view 时按 Component 类型遍历，命中率高 |
| 并行 tick | Worker System 可多线程跑（EnTT view 可分片迭代） |
| 批量处理 | 一次 tick 处理所有 MachineComponent，无虚函数调用 |

## 8. 阶段计划

每个阶段结束后游戏必须能跑（哪怕只是空窗口）。

### P0 — 架构基线（本文档）

**目标**：共识基线
**产出**：本文档
**验证**：文档评审通过

### P1 — 骨架搭建

**目标**：SDL3 窗口 + Vulkan 三角形 + ImGui 调试层
**产出**：
- `engine/CMakeLists.txt` 骨架
- `engine/platform/window.cpp` SDL3 窗口
- `engine/render_backend/vulkan_device.cpp` Vulkan 初始化
- `engine/engine_app/main.cpp` 主循环
- 一个三角形能画出来
**验证**：运行可执行文件，看到 Vulkan 三角形

### P2 — 数据核心移植 + ECS 骨架

**目标**：把 `GDWorldData` + `TerrainGenerator` 剥离 godot-cpp，移到 `engine/engine/data/`；同时搭建 ECS 骨架
**产出**：
- `world_data.cpp` 用 std::vector 替换 PackedByteArray
- `terrain_generator.cpp` 算法保留，去 Variant
- `save_manager.cpp` region_file 格式保留
- `engine/engine/ecs/` 骨架：World / Entity / Component / System 接口（基于 EnTT）
- `SystemScheduler` 主线程 + worker pool 调度框架
- 标准 Component 定义（Position/Velocity/Health 等）
- 单元测试：能生成一个 chunk 的 terrain 数据；能创建 Entity + 加 Component + 跑一个空 System
**验证**：
- `./engine_test data_test` 输出 chunk (0,0,0) 的 non-air cell 数
- `./engine_test ecs_test` 创建 1000 个 Entity + Component，跑 System，输出耗时

### P3 — Voxel 渲染

**目标**：自研 `ChunkRenderer`，能渲染一个 chunk
**产出**：
- `greedy_mesh.cpp` 从 `gd_chunk_helper` 移植
- `collision_faces.cpp` 同上
- `chunk_renderer.cpp` Vulkan vertex buffer + draw call
- `vertex_buffer_pool.cpp` 内存池
- 主循环能加载 chunk 数据并渲染
**验证**：运行看到地面 chunk（哪怕只是纯色）

### P4 — 玩家控制 + 物理

**目标**：SDL 输入 + 自研碰撞 + 第一人称相机
**产出**：
- `voxel_collision.cpp` AABB vs voxel grid
- `player_controller.cpp` 移动 / 跳跃 / 重力
- `ray_cast.cpp` DDA 算法
- 鼠标锁定 + 视角控制
**验证**：能走动看世界，能挖方块

### P5 — 资产管线 + 贴图

**目标**：贴图加载 + 材质 atlas + 完整方块渲染
**产出**：
- `loader.cpp` JSON + 二进制
- `texture_cache.cpp` stb_image
- `font_atlas.cpp` FreeType
- `shader_cache.cpp` shaderc
- 方块贴图能正确显示
**验证**：能看到不同材质的方块

### P6 — UI 系统（MUI 复刻）

**目标**：完整复刻 MUI 全部能力（见 5.3.2）
**产出**：
- View 系统：ViewGroup + 基础控件 + 布局三阶段
- ViewModel + 数据绑定
- 文本引擎：HarfBuzz + ICU + SDF + Emoji
- Arc 2D 渲染器
- 动画系统
- 至少实现 hotbar / inventory / crafting 三个 UI
**验证**：能打开背包、合成、看到中文 + Emoji 正确渲染

### P7 — 玩法迁移（C++NativeScript + ECS System）

**目标**：机器、quest、save、网络全部迁移为 C++NativeScript + ECS System
**产出**：
- `ScriptManager` + `DynamicLibrary` + `FileWatcher` + `RegistryHub`
- 5 个机器系统重构为 `MachineTickSystem`（机器类型区分 furnace/bloomery/pit_kiln/charcoal_pit/anvil）
- `QuestRegistry` + quest 注册 API
- `SaveManager` C++ 实现（region_file 复用）
- SteamP2P 网络层剥离 Godot
- 至少 3 个示例脚本 DLL：`furnace_recipes.dll` / `bronze_furnace.dll` / `basic_quests.dll`
- `/snt reload` 命令可用
**验证**：能烧矿、能接任务、能存档、能联机；改 .cpp 重编译后 `/snt reload` 立即生效

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Vulkan 学习曲线陡 | P1/P3 延期 | 参考 vkguide.dev；先用 vulkan-tutorial 跑通三角形 |
| MUI 复刻工作量巨大 | P6 延期 | 分子阶段：先 View+ViewModel，再文本引擎，再动画 |
| ECS 学习曲线 + 重构成本 | P2/P7 延期 | EnTT 文档完善；先骨架后迁移；Manager→System 是机械映射 |
| DLL 热重载跨平台坑 | P7 延期 | 5 个坑已有对应方案（见 5.5.4）；先用 Windows 验证 |
| 跨平台坑（macOS MoltenVK） | macOS 不工作 | 优先 Win/Linux，macOS 后期处理 |
| 现有 C++ 剥离 godot-cpp 工作量大 | P2 延期 | 渐进式：先保留 godot-cpp 编译，逐步替换 |
| 资产格式迁移 | 旧存档失效 | 写 .tres → JSON 转换器；旧存档一次性迁移 |
| 网络协议变更 | 旧客户端不兼容 | 协议版本号；不向后兼容，强制升级 |
| 脚本 DLL 编译等待 | 开发体验 | CMake 增量编译 1-3 秒（仅改 1 个文件）；file watcher 自动 reload |

## 10. 不做什么（边界）

- **不做编辑器**：调试用 ImGui，不做 Godot 那样的可视化编辑器
- **不做跨平台移动端**：Windows/Linux/macOS 优先，Android/iOS 后期再说
- **不做 VR/AR**：当前不考虑
- **不做向后兼容**：旧存档一次性迁移，之后不再保证
- **不做 mod API**：先把核心玩法跑通，mod API 后期再开
- **不引入重型第三方引擎**：不依赖 Unreal/Bevy/O3DE
- **不做光线追踪**：光栅化优先，RT 后期可选
